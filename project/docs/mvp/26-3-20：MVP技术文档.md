## 1. 文档目标

本文档用于把原先分散的 3 份材料合并为一份统一基线，同时回答以下问题：

1. 这个 MVP 到底要做什么，不做什么
2. 技术路线、仓库结构、接口契约按什么口径冻结
3. 前后端、数据处理、会话与溯源功能应如何落地
4. 实现阶段该按什么顺序推进、验收、测试与发布

使用原则：

- 方案设计、实现拆解、仓库骨架以本文件为准
- 如后续出现冲突，以本文件中的冻结决策为准
- 当前不冻结开发排期；对外展示版排期另行单独编制

## 2. 背景

当前学术写作场景中，高校学生、科研工作者普遍存在以下痛点：

1. 文献管理与复用效率低，写作时无法快速定位和复用已有参考文献
2. 引用溯源难度大，观点、数据难以快速回溯到原文位置
3. 通用 AI 写作工具无法基于私有文献库稳定生成可信内容，容易出现编造文献、内容失真

因此，需要打造一款以用户自定义知识库为核心的论文写作辅助工具，深度嵌入写作流程，解决效率与严谨性问题。

## 3. 目标、非目标、范围与边界

### 3.1 目标

1. 以 WPS 插件形式嵌入写作场景
2. 支持基于私有文献知识库的问答与续写
3. 支持来源溯源，其中页码级定位为 MVP 必做能力，bbox 高亮作为增强项
4. 在插件中输入 PDF 路径后，完成自动清洗、切片、建索引与检索问答闭环

### 3.2 非目标

1. 不做全流程 Agentic 系统
2. 不做知识图谱
3. 不做 Word 插件适配

### 3.3 范围

前端范围：

1. 插件宿主集成与布局
2. WPS 文档上下文读取与同步
3. 与后端的 HTTP + SSE 通信
4. 会话、导入、对话、来源展示与来源跳转

后端范围：

1. MinerU API 清洗 PDF -> Markdown + JSON
2. 父子块语义切片与映射
3. ChromaDB + BM25 混合检索
4. Query 改写、RRF 融合、Reranker 重排
5. FastAPI + Pydantic 接口层
6. SQLite 会话管理
7. 基于 LangChain / 轻量 LangGraph 的检索生成编排

### 3.4 边界

1. MVP 阶段使用 MinerU API，后续再研究源码本地化
2. 仅支持 PDF 格式文献导入
3. 单机部署，无多用户并发需求
4. 不支持实时协作编辑
5. 溯源能力以页级定位为交付底线，bbox 高亮不能作为 MVP 阻塞项

## 4. 冻结决策

### 4.1 技术路线冻结

| 层级 | 冻结选型 | 说明 |
|------|----------|------|
| 前端插件 | WPSJS + Vue 3 + TypeScript | 按调研结论执行 |
| 前端通信 | HTTP + SSE | 先验证兼容性；失败时降级为轮询或分块响应 |
| 本地服务 | Python + FastAPI + Pydantic | 与 LangChain / LangGraph 生态一致 |
| 解析 | MinerU API | MVP 直接调用，暂不做本地化 |
| 向量库 | ChromaDB | 与当前设计保持一致 |
| 关键词检索 | BM25 | 作为混合检索的一支 |
| 会话存储 | SQLite | 负责会话、消息、任务状态 |
| 编排 | LangChain / 轻量 LangGraph | 只用于检索生成编排，不扩成全流程 Agent |
| 溯源 | 页码 + bbox 元数据 | 页级定位为 MVP 必做，bbox 高亮为增强项 |

### 4.2 工程决策冻结

1. MVP 阶段采用单仓 `monorepo`
2. 后端导出的 OpenAPI 为接口唯一事实来源
3. 前端类型由 OpenAPI 生成，不手写双份 DTO
4. 运行时数据与源码目录分离，并使用目录化布局
5. Windows 为当前主开发环境，脚本默认采用 PowerShell

### 4.3 本期不纳入实现主线的内容

- FAISS 替换 ChromaDB
- 知识图谱
- 全流程 Agentic
- Word 插件适配
- 多用户并发与远程部署
- MinerU 本地化重构
- Docker 化

#### 4.4.3 LLM 供应商

**用户自填配置**：支持用户输入任意 OpenAI 兼容的 API 端点和 Key。

```json
{
  "llm_providers": [
    {
      "name": "智谱 GLM",
      "base_url": "https://open.bigmodel.cn/api/paas/v4",
      "api_key": "user-provided",
      "models": ["glm-4", "glm-4-flash"]
    },
    {
      "name": "DeepSeek",
      "base_url": "https://api.deepseek.com/v1",
      "api_key": "user-provided",
      "models": ["deepseek-chat", "deepseek-reasoner"]
    }
  ]
}
```

**模型发现机制**：
1. 用户输入 base_url + api_key
2. 调用 `/v1/models` 端点获取可用模型列表
3. 用户从列表选择或手动输入模型名称

#### 4.4.4 MinerU API

| 配置项 | 值 |
|--------|-----|
| API 端点 | `https://mineru.net/api/v4/extract/task` |
| 文档地址 | https://mineru.net/apiManage/docs |
| 用途 | PDF 解析 → Markdown + JSON |

**MinerU 输出文件结构**：

```text
<doc_id>/
├── original.pdf              # 原始 PDF
├── full.md                   # 完整 Markdown 文本
├── content_list_v2.json      # 结构化内容（含 bbox）
├── layout.json               # 布局信息
├── block_list.json           # 块级元素列表
└── images/                   # 提取的图片
```

**bbox 格式**：`[x1, y1, x2, y2]`（左上角 + 右下角，像素坐标）

**页码索引**：从 0 开始

### 4.5 检索策略细节

#### 4.5.1 四种输入场景的改写策略

| 场景 | 输入组合 | 权重策略 | Query 改写逻辑 |
|------|----------|----------|----------------|
| 场景 1 | 仅 Prompt | Prompt 100% | 直接使用 Prompt 作为检索 Query |
| 场景 2 | 已写内容 | 已写 100% | 基于已写内容推断续写意图，生成检索 Query |
| 场景 3 | 已写 + 圈选 | 已写 30% + 圈选 70% | 组合两段文本，生成检索 Query |
| 场景 4 | 已写 + 圈选 + Prompt | 已写 20% + 圈选 30% + Prompt 50% | 三段文本组合，生成检索 Query |

**实现方式**：三种 Query 分别进行向量召回，结果合并后过滤无关内容，打包返回给 Agent。

#### 4.5.2 RRF 融合参数

| 参数 | 值 | 说明 |
|------|-----|------|
| k | 60 | RRF 默认参数 |
| 向量召回 Top-K | 30 | ChromaDB 召回数量 |
| 关键词召回 Top-K | 30 | BM25 召回数量 |
| Rerank 后 Top-K | 10 | 最终返回给 LLM 的文档数 |

**调优策略**（未来）：
- 头部太弱、长尾噪音多 → 调小 k
- 头部过强、漏掉中排名有价值文档 → 调大 k
- 计划：自适应 k 区间

#### 4.5.3 切片策略

| 策略 | 值 | 说明 |
|------|-----|------|
| 切片方式 | 语义切分 | 优先保证语义完整 |
| 大小 | 不固定 | 根据语义边界自然分割 |
| 重叠 | ~10% | 首尾重合缓冲区 |

**父子块定义**（待数据结构分析后确定）：
- 当前倾向：父块 = 段落级，子块 = 句子级
- 需结合 MinerU 输出的 `content_list_v2.json` 结构细化

### 4.6 SSE 降级策略

| 配置项 | 值 |
|--------|-----|
| 触发条件 | 连续 3 次无响应 |
| 降级策略 | 指数退避重试 |
| 最大重试次数 | 5 次 |
| 最终降级 | HTTP 轮询 或 WebSocket |

### 4.7 溯源元数据格式

每个检索到的 Chunk 需携带以下元数据：

```json
{
  "chunk_id": "chunk_001",
  "source": "D:/papers/example.pdf",
  "page": 0,
  "bbox": [100, 200, 400, 350],
  "type": "paragraph",
  "parent_id": "para_003",
  "content": "检索到的文本内容..."
}
```

## 5. 总体架构

### 5.1 最小闭环

本系统不是先把算法做满，而是先打通下面这条最小闭环：

```text
WPS 插件
  -> 输入 PDF 路径
  -> 后端导入、解析、切片、建索引
  -> 插件发起问答 / 续写
  -> 后端检索、生成、SSE 返回
  -> 插件展示回答与来源
  -> 用户继续多轮写作
```

### 5.2 核心交互流程

```
┌─────────────────────────────────────────────────────────────────┐
│                        WPS 插件                                  │
├─────────────────────────────────────────────────────────────────┤
│  常驻循环：每 5 秒读取文档上下文并同步                           │
│  对话循环：点击发送后组合输入、触发检索、流式展示                 │
└─────────────────────────────────────────────────────────────────┘
                              │
                        HTTP + SSE
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                        后端服务                                  │
├─────────────────────────────────────────────────────────────────┤
│  导入链路：PDF -> MinerU -> 切片 -> 索引                         │
│  查询链路：上下文归一化 -> Query 改写 -> 混合检索 -> 生成         │
│  会话链路：Session / Message / Sync 持久化                     │
└─────────────────────────────────────────────────────────────────┘
```

### 5.3 四种输入场景与权重

| 场景 | 输入 | 建议权重 |
|------|------|----------|
| 场景 1 | 仅 Prompt | Prompt 100% |
| 场景 2 | 已写内容 | 已写内容 100% |
| 场景 3 | 已写内容 + 圈选内容 | 已写 30% + 圈选 70% |
| 场景 4 | 已写内容 + 圈选内容 + Prompt | 已写 20% + 圈选 30% + Prompt 50% |

### 5.4 检索与生成流程

```
┌─────────────────────────────────────────────────────────────────┐
│              written_content + selected_content + prompt       │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  QueryContextBuilder -> Query 改写 -> ChromaDB/BM25 并行召回   │
│  -> RRF 融合 -> Reranker -> 带来源答案生成 -> SSE 分片返回      │
└───────────────────────┼─────────────────────────────────────────┘
                        ▼
              返回 content / sources / done / error
```

### 5.5 数据导入流程

```
┌─────────────────────────────────────────────────────────────────┐
│                    用户输入：PDF 路径                             │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      数据清洗管道                                │
│  路径校验 -> MinerU 解析 -> 结果持久化 -> 语义切片 -> 元数据映射   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      切片与索引                                  │
│  父子块映射 -> ChromaDB 索引 -> BM25 索引 -> 状态可查询 / 可恢复  │
└─────────────────────────────────────────────────────────────────┘
```

### 5.6 模块职责

| 模块 | 主要职责 |
|------|----------|
| WPS 插件 | UI、宿主集成、文档上下文读取、会话与来源交互 |
| API 层 | 路由、参数校验、响应格式化、SSE 输出 |
| 工作流层 | 导入流程、查询流程的跨步骤编排 |
| 服务层 | Query 改写、检索、重排、生成、会话操作、同步 |
| 仓储层 | SQLite、ChromaDB、BM25、任务状态持久化 |

## 7. 契约、接口与数据

### 7.1 契约共享规则

以 FastAPI 导出的 OpenAPI 为后端接口唯一事实来源。

```text
contracts/
├── openapi/
│   └── openapi.snapshot.json
└── README.md
```

工作方式：

1. 后端定义 Pydantic 请求/响应模型
2. 通过 FastAPI 导出 OpenAPI schema
3. 将 schema 快照保存到 `contracts/openapi/openapi.snapshot.json`
4. 前端基于快照生成 `apps/wps-plugin/src/generated/api-types.ts`

### 7.2 API 设计

| 接口 | 方法 | 功能 | 请求 | 响应 |
|------|------|------|------|------|
| `/health` | GET | 健康检查 | - | `{ "status": "ok" }` |
| `/ingest` | POST | 导入 PDF，执行清洗 + 切片 + 索引 | `{ "pdf_path": "D:/papers/xxx.pdf" }` | `{ "task_id": "abc123", "status": "processing" }` |
| `/status/{task_id}` | GET | 查询导入与索引构建状态 | - | `{ "status": "completed", "chunks": 156 }` |
| `/session/create` | POST | 创建新会话 | `{ "title": "论文写作" }` | `{ "session_id": "sess_001" }` |
| `/session/list` | GET | 获取会话列表 | - | `{ "sessions": [...] }` |
| `/session/{id}/history` | GET | 获取会话历史 | - | `{ "messages": [...] }` |
| `/session/{id}` | DELETE | 删除会话 | - | `{ "status": "deleted" }` |
| `/query` | POST | RAG 检索 + 生成，SSE 流式返回 | 见下方 | Server-Sent Events |
| `/sync` | POST | 常驻循环同步写作上下文 | `{ "session_id": "...", "written_content": "..." }` | `{ "status": "synced" }` |

### 7.3 `/query` 请求结构

```json
{
  "session_id": "sess_001",
  "written_content": "深度学习在图像识别...",
  "selected_content": "卷积神经网络的池化层",
  "prompt": "解释池化层的作用",
  "mode": "qa"
}
```

### 7.6 SQLite Schema

```sql
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,
    title TEXT,
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    metadata JSON
);

CREATE TABLE messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT,
    role TEXT,
    content TEXT,
    timestamp TIMESTAMP,
    metadata JSON,
    FOREIGN KEY (session_id) REFERENCES sessions(id)
);

CREATE TABLE tool_calls (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    message_id INTEGER,
    tool_name TEXT,
    tool_input JSON,
    tool_output JSON,
    timestamp TIMESTAMP,
    FOREIGN KEY (message_id) REFERENCES messages(id)
);

CREATE INDEX idx_messages_session ON messages(session_id);
CREATE INDEX idx_messages_timestamp ON messages(timestamp);
```

## 8. 实施原则

### 8.1 总体原则

- 先冻结接口契约，再并行推进前后端
- 先打通最小闭环，再叠加高级能力
- 先做可验证的页级溯源，再追求 bbox 级精细交互
- 先确保单机稳定可演示，再做性能优化和扩展

### 8.2 开发顺序原则

本项目必须按以下顺序推进：

1. 关键技术验证
2. 前后端骨架与接口契约
3. PDF 导入与索引闭环
4. 检索与问答闭环
5. 会话管理与常驻同步
6. 溯源交互、打磨、验收与发布

### 8.3 Definition of Done

任一任务完成，至少满足：

- 有明确代码产物或文档产物
- 有可操作的验收方式
- 不依赖口头约定
- 能被后续任务直接消费

### 8.4 排期说明

本文档暂不冻结开发排期。对外展示版排期、对外演示节奏与资源缓冲安排，后续单独编制。

## 12. 风险与异常处理

### 12.1 风险清单

| 风险 | 影响 | 优先级 | 应对策略 |
|------|------|--------|----------|
| WPS 对 SSE 支持不稳定 | 流式输出受阻 | 高 | Phase 0 提前验证；失败则降级 |
| MinerU API 限流或不稳定 | 导入链路卡住 | 高 | 做任务状态机、重试、失败提示 |
| bbox 高亮能力不足 | 溯源体验弱于理想目标 | 高 | MVP 先保页级定位，bbox 作为增强 |
| 长文档轮询过重 | 影响编辑体验 | 中 | 只提取窗口文本，去重同步，频率可配 |
| ChromaDB 本地索引损坏 | 检索不可用 | 中 | 提供从解析产物重建索引能力 |
| SQLite 损坏 | 会话丢失 | 中 | WAL 模式、自动备份、损坏恢复 |
| 模型 / 外部服务配置复杂 | 调试成本高 | 中 | 前期固定默认配置，设置项放到 Phase 5 |

### 12.2 异常处理要求

- PDF 路径不存在：提示用户检查路径
- PDF 加密或损坏：提示文件无法解析
- 文献库为空：引导用户先导入 PDF
- 会话历史过长：按策略摘要旧对话
- 用户输入为空：忽略，不触发检索
- SSE 断线：自动重连或按降级路径处理

## 14. 结论

1. MVP 以 WPS 插件 + 本地 FastAPI 服务的单仓结构推进
2. OpenAPI 是前后端接口契约的唯一事实来源
3. 页级溯源是交付底线，bbox 高亮作为增强项处理
4. 工程推进顺序必须是：风险验证 -> 契约冻结 -> 骨架联调 -> 导入闭环 -> 问答闭环 -> 会话同步 -> 溯源与发布

因此，本项目不建议直接从 RAG 算法细节开始写代码，而应优先完成：

`风险验证 -> 契约冻结 -> 骨架联调 -> 导入闭环 -> 问答闭环`
