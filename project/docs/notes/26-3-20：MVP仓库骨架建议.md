# MVP仓库骨架建议

> 文档状态：Draft v1  
> 更新时间：2026-03-20  
> 建议用途：作为前后端项目初始化的第一版仓库骨架说明

## 1. 目标

这份文档回答 4 个问题：

1. 这个 MVP 应该用单仓还是双仓
2. 前端 WPS 插件和后端 Python 服务目录该怎么拆
3. API 契约如何共享，避免前后端各写一套类型
4. 第一天应该按什么顺序把项目脚手架搭起来

本文档默认你还没有正式创建项目仓库，因此以“推荐结构”和“初始化顺序”为主。

## 2. 先定结论

### 2.1 仓库形态

MVP 阶段建议使用 **单仓（monorepo）**，不要一开始就拆成前后端两个仓库。

原因很直接：

- 当前是单机本地服务 + WPS 插件，边界虽然清楚，但变更频繁
- 前后端 API 契约会在早期持续变化，单仓更容易同步修改
- 安装说明、调试脚本、技术文档、测试样例都需要共存
- 目前是 MVP，不值得先引入跨仓协作成本

### 2.2 什么时候再拆仓

以下情况同时出现时，再考虑拆仓：

- 插件前端和本地服务开始由不同人长期维护
- API 契约基本稳定
- 发布节奏明显不同
- 后端开始从“本地服务”演化成“独立服务端”

在此之前，单仓收益明显更高。

## 3. 推荐仓库根目录

建议项目根目录先定为一个独立文件夹，例如：

```text
D:\Projects\wps-thesis-mvp
```

名字不是重点，重点是目录职责固定。

推荐第一版根目录结构：

```text
wps-thesis-mvp/
├── apps/
│   ├── wps-plugin/
│   └── local-service/
├── contracts/
├── docs/
├── scripts/
├── .editorconfig
├── .gitattributes
├── .gitignore
├── README.md
└── pnpm-workspace.yaml
```

### 3.1 为什么这样拆

| 目录 | 作用 | 为什么需要 |
|------|------|------------|
| `apps/wps-plugin` | WPS 插件前端 | 宿主集成、UI、WPS API 封装 |
| `apps/local-service` | 本地 Python 服务 | 导入、索引、检索、会话、SSE |
| `contracts` | API 契约产物 | 避免前后端重复维护接口定义 |
| `docs` | 方案、计划、决策 | 与实现同仓，方便同步更新 |
| `scripts` | PowerShell 启动/构建脚本 | 你的环境是 Windows，脚本应该直接适配 |

## 4. 前端骨架建议

前端目录建议如下：

```text
apps/wps-plugin/
├── manifest.xml
├── ribbon.xml
├── package.json
├── tsconfig.json
├── vite.config.ts
├── .env.example
├── public/
│   └── icons/
├── src/
│   ├── main.ts
│   ├── App.vue
│   ├── components/
│   │   ├── layout/
│   │   ├── ingest/
│   │   ├── chat/
│   │   ├── session/
│   │   └── source/
│   ├── composables/
│   │   ├── useWps.ts
│   │   ├── useSelection.ts
│   │   ├── useDocumentContextSync.ts
│   │   ├── useChatStream.ts
│   │   └── useSessions.ts
│   ├── services/
│   │   ├── apiClient.ts
│   │   ├── streamClient.ts
│   │   ├── pluginStorage.ts
│   │   └── wpsClient.ts
│   ├── stores/
│   │   ├── chatStore.ts
│   │   ├── sessionStore.ts
│   │   └── settingsStore.ts
│   ├── generated/
│   │   └── api-types.ts
│   ├── styles/
│   └── types/
└── tests/
```

### 4.1 前端目录设计原则

- `components` 只放 UI 组件，不直接调用 WPS 原生 API
- `composables` 放页面行为和业务交互逻辑
- `services` 放通信、存储、宿主适配
- `stores` 只保存跨组件状态，不要把所有逻辑都塞进 store
- `generated` 放接口生成文件，禁止手改

### 4.2 前端推荐最小依赖

MVP 第一版建议只引入必要依赖：

| 依赖 | 用途 | 是否建议 |
|------|------|----------|
| `vue` | UI 框架 | 必须 |
| `typescript` | 类型系统 | 必须 |
| `vite` | 构建与开发 | 必须 |
| `pinia` | 会话/设置/消息状态 | 建议 |
| `zod` | 前端接口结果兜底校验 | 建议 |
| `marked` 或 `markdown-it` | 回答内容渲染 | 建议 |

MVP 阶段不建议先引入很重的 UI 组件库。  
原因：WPS 插件宿主环境本身就有限制，先把结构跑通更重要。

```text
    ├── unit/
    ├── integration/

```

### 5.1 后端目录设计原则

- `api/routers` 只做路由、参数校验、响应格式化
- `services` 做单一职责的业务动作，不负责跨模块编排
- `workflows` 负责“导入流程”和“查询流程”这种跨步骤 orchestration
- `repositories` 只面向存储，不直接耦合 HTTP 或 UI 语义
- `clients` 只面向外部服务调用，例如 MinerU、Embedding、LLM

### 5.2 为什么不把所有东西都放在 `services`

因为这个项目天然有三类逻辑：

- 单步动作，例如切片、检索、消息持久化
- 跨步骤流程，例如导入工作流、问答工作流
- 外部依赖适配，例如 MinerU API、LLM API

如果不从一开始拆开，后面 FastAPI 工程会很快变成一个不可维护的大文件集合。

## 6. 契约共享建议

这是这个项目最容易被忽略，但最该提前定型的部分。

### 6.1 推荐方案

以 **FastAPI 导出的 OpenAPI** 作为后端接口的唯一事实来源，前端类型由它生成，不手写双份 DTO。

推荐目录：

```text
contracts/
├── openapi/
│   └── openapi.snapshot.json
└── README.md
```

### 6.2 推荐工作方式

1. 后端先定义 Pydantic 请求/响应模型
2. 通过 FastAPI 导出 OpenAPI schema
3. 把 schema 快照保存到 `contracts/openapi/openapi.snapshot.json`
4. 前端基于快照生成 `apps/wps-plugin/src/generated/api-types.ts`

### 6.3 这样做的好处

- 后端模型变更有明确来源
- 前端不会靠猜字段名写接口
- 联调时可以快速发现 breaking change
- 文档、测试、类型三者能保持一致

### 6.4 禁止事项

- 禁止前端手写一套和后端重复的接口类型
- 禁止在多个地方各存一份“差不多一样”的 schema
- 禁止把 `generated` 目录里的文件当业务文件修改

## 7. 运行时数据与仓库代码分离

这个项目有大量本地运行数据，必须从一开始和代码分开。

### 7.1 建议的数据目录

运行时数据统一放在：

```text
apps/local-service/data/
```

其下分层如下：

```text
data/
├── documents/     # 原始 PDF 与解析产物
├── chroma_db/     # ChromaDB 数据
├── bm25/          # BM25 索引
├── cache/         # 临时缓存
├── tmp/           # 临时中间文件
└── sessions.db    # SQLite 会话库
```

### 7.2 为什么一定要这样

- 便于排障和备份
- 便于后续做清理和重建
- 不会把索引文件混进源码目录
- 后面如果换部署方式，也容易整体迁移

## 8. 文档和决策记录建议

既然已经开始写技术文档，就不要让文档留在仓库外长期漂移。

推荐把当前这些文档后续同步进仓库：

```text
docs/
├── architecture/
├── plans/
├── research/
└── decisions/
```

### 8.1 建议迁移方式

| 目录 | 放什么 |
|------|--------|
| `docs/architecture` | MVP 技术文档、架构图 |
| `docs/plans` | MVP 实现计划、任务拆分 |
| `docs/research` | WPS 调研、MinerU 调研 |
| `docs/decisions` | 关键技术决策，例如 SSE 降级、溯源策略 |

### 8.2 ADR 建议

从这个项目开始，建议至少记录 3 条简短 ADR：

1. 为什么 MVP 选单仓
2. 为什么接口以 OpenAPI 为唯一事实来源
3. 为什么溯源先做页级定位，再逐步增强到 bbox

## 9. Windows 友好的脚本组织方式

因为你的目标环境是 Windows + WPS，所以仓库脚本不要默认站在 Linux/macOS 视角。

推荐目录：

```text
scripts/
├── dev/
│   ├── start-backend.ps1
│   ├── start-plugin.ps1
│   ├── export-openapi.ps1
│   └── generate-frontend-types.ps1
├── check/
│   ├── test-backend.ps1
│   └── test-frontend.ps1
└── release/
    └── build-plugin.ps1
```

### 9.1 为什么推荐 PowerShell

- 你的当前工作环境就是 PowerShell
- WPS 插件开发主要是 Windows 场景
- 后续启动本地服务、导出契约、构建插件都更容易落地

## 10. 初始化顺序建议

不要一上来就开始写业务代码。先把骨架定型，后面的效率会高很多。

### Step 1：建根目录

先创建以下内容：

- `apps/`
- `contracts/`
- `docs/`
- `scripts/`
- `.editorconfig`
- `.gitattributes`
- `.gitignore`
- `README.md`

### Step 2：先起后端骨架

建议先起后端，而不是先起前端。

原因：

- API 契约以后端为源头更稳定
- 前端早期最依赖的是接口字段和状态约定
- FastAPI 的 OpenAPI 导出能尽快进入工作流

后端骨架最小目标：

- `/health` 跑通
- 配置加载可用
- `app/main.py` 可启动
- `tests/` 可执行

### Step 3：导出第一版 OpenAPI

哪怕此时只有 `/health` 和一个占位 `/query`，也先把契约导出。

目标是尽快建立下面这个流程：

`后端模型 -> OpenAPI -> 前端类型`

### Step 4：再起前端骨架

前端第一版只做：

- 插件能在 WPS 中加载
- 能显示“后端已连接/未连接”
- 能请求 `/health`
- 有最小页面布局

### Step 5：补充脚本和说明

只有当下面三件事都存在，仓库才算真正可用：

- 启动脚本
- 环境变量示例
- README 启动说明

## 11. 第一版 README 应该写什么

根目录 `README.md` 建议只保留最关键内容：

1. 项目一句话说明
2. 仓库结构
3. 环境要求
4. 前端启动方式
5. 后端启动方式
6. 契约生成方式
7. 已知限制

不要在第一版 README 里写大段产品愿景，先让别人能跑起来。

至少忽略以下内容：

```gitignore
# frontend
node_modules/
dist/
.vite/

# python
.venv/
```

可以在 `data/` 里保留 `.gitkeep` 或 README，用于说明目录用途，但不要把真实运行数据提交进去。

如果你下一步真的开始搭仓库，我建议第一批只创建这些文件：

```text

├── pnpm-workspace.yaml
├── apps/wps-plugin/package.json
├── apps/wps-plugin/manifest.xml
├── apps/wps-plugin/ribbon.xml
├── apps/wps-plugin/src/main.ts
```

这批文件的目标很明确：

- 后端先能启动
- 前端先能挂载
- 仓库先有运行入口

## 14. 第一阶段不要过度设计的地方

以下内容先不要在骨架阶段做重：

- 不要先引入复杂微前端或多包工程
- 不要先做“插件前端共享组件库”
- 不要先做多环境配置矩阵
- 不要先做 Docker 化
- 不要先做可插拔模型抽象工厂

MVP 真正需要的是一个稳定的最小闭环，而不是一套看起来完整但没人能用起来的工程模板。

## 15. 推荐的第一天落地顺序

如果你明天就开始搭项目，我建议按这个顺序：

1. 创建根目录和基础文件
2. 初始化 `apps/local-service`
3. 跑通 `/health`
4. 导出第一版 OpenAPI
5. 初始化 `apps/wps-plugin`
6. 在 WPS 中加载插件空壳
7. 插件请求 `/health`
8. 补 PowerShell 启动脚本
9. 写最小 README

这一天结束时，不要求有 RAG，不要求有索引，不要求有会话。  
只要求一件事：

`WPS 插件 <-> 本地 FastAPI 服务` 的最小链路已经成立。

## 16. 结论

第一版仓库骨架建议核心只有 3 条：

- 用单仓，不要过早拆仓
- 以后端 OpenAPI 作为接口唯一事实来源
- 先搭最小链路，再逐步填业务模块

如果后续要正式开始搭项目，下一份最值得写的不是“大而全任务计划”，而是：

- `仓库初始化 checklist`
- `首批目录与文件脚手架`
- `health -> query -> ingest` 的最小接口占位实现

这样你就不是在“继续写文档”，而是在把文档转成真正能跑的工程起点。
