### 2.2 技术冻结

为避免实现阶段继续分叉，MVP 先冻结为以下技术路线：

| 层级 | 冻结选型 | 说明 |
|------|----------|------|
| 前端插件 | WPSJS + Vue 3 + TypeScript | 按调研结论执行 |
| 前端通信 | HTTP + SSE | 需要做兼容性验证，失败时降级为轮询或分块响应 |
| 本地服务 | Python + FastAPI + Pydantic | 与 LangChain/LangGraph 生态一致 |
| 解析 | MinerU API | MVP 直接调用，暂不做本地化 |
| 向量库 | ChromaDB | 与当前设计文档保持一致 |
| 关键词检索 | BM25 | 作为混合检索的一支 |
| 会话存储 | SQLite | 负责会话、消息、任务状态 |
| 编排 | LangChain / 轻量 LangGraph | 仅用于检索生成编排，不扩成全流程 Agent |
| 溯源 | 页码 + bbox 元数据 | 前端先做可点击来源与页级定位，bbox 高亮作为增强 |

## 3. 实施原则

### 3.1 总体原则

- 先冻结接口契约，再并行推进前后端
- 先打通最小闭环，再叠加高级能力
- 先做可验证的页级溯源，再追求 bbox 级精细交互
- 先确保单机稳定可演示，再做性能优化和扩展

### 3.2 开发顺序原则

本项目必须按以下顺序推进：

1. 关键技术验证
2. 前后端骨架与接口契约
3. PDF 导入与索引闭环
4. 检索与问答闭环
5. 会话管理与常驻同步
6. 溯源交互、打磨、验收与发布

### 3.3 Definition of Done

任一任务完成，至少满足：

- 有明确代码产物或文档产物
- 有可操作的验收方式
- 不依赖口头约定
- 能被后续任务直接消费

## 5. 总体工作流拆分

整个实现分成 6 个 Phase 和 3 条主工作流。

### 5.1 三条主工作流

| 工作流 | 目标 | 关键输出 |
|--------|------|----------|
| 前端工作流 | WPS 插件 UI、文档内容监听、来源交互、流式展示 | 可装载的插件包 |
| 后端工作流 | 解析、切片、索引、检索、会话、SSE | 本地 FastAPI 服务 |
| 联调验收工作流 | 契约对齐、真实 PDF 验证、性能与异常回归 | 可演示的 MVP |

### 5.2 Phase 划分

| Phase | 名称 | 核心目标 |
|-------|------|----------|
| Phase 0 | 风险前置验证 | 先消灭可能卡死项目的未知点 |
| Phase 1 | 骨架与契约 | 先跑通前后端最小闭环 |
| Phase 2 | 数据导入链路 | 先让知识库真正可用 |
| Phase 3 | 检索与问答链路 | 跑通核心价值链 |
| Phase 4 | 会话与常驻同步 | 让写作上下文持续可用 |
| Phase 5 | 溯源、打磨与发布 | 形成可演示版本 |

## 6. Phase 0：风险前置验证

### 6.1 阶段目标

在正式编码前，先验证 3 个最高风险点：

- WPS 插件侧的流式通信可行性
- WPS 插件侧的内容监听与上下文提取可行性
- 溯源跳转的最小 MVP 交互路径可行性

### 6.2 任务拆分

| ID | 任务 | 输出物 | 依赖 | 验收标准 |
|----|------|--------|------|----------|
| P0-1 | 验证 `Selection` 读取与 `WindowSelectionChange` 监听 | 验证脚本 + 结论记录 | 无 | 能获取选中文本、光标位置、段落范围 |
| P0-2 | 验证 WPS WebView 对 `EventSource` / SSE 的兼容性 | 通信验证页 + 结论记录 | P0-1 | 能稳定接收连续分片；如失败，给出降级方案 |
| P0-3 | 验证 5 秒轮询读取当前段落/上下文的成本 | 性能记录 | P0-1 | 轮询不明显卡顿，不破坏编辑体验 |
| P0-4 | 验证来源跳转 MVP 路径 | 决策记录 | 无 | 明确“页级定位如何做”；bbox 高亮是否能本期落地有结论 |
| P0-5 | 冻结最终本地目录布局 | 目录规范文档 | 无 | 前后端、数据目录、缓存目录都有固定位置 |

### 6.3 本阶段决策门槛

Phase 0 结束前必须明确：

- 若 `SSE` 不稳定，前端改用什么降级方案
- 若 `bbox` 无法直接高亮，MVP 如何退化为“页码定位 + 来源卡片”
- 轮询内容缓存是保存在前端内存、PluginStorage 还是后端缓存

## 9. Phase 3：检索、重排与流式问答

### 9.1 阶段目标

实现核心价值链：

- 输入 `written_content + selected_content + prompt`
- 组合权重与 Query 改写
- 启动混合检索
- RRF 融合
- Reranker 重排
- 带来源生成回答
- SSE 流式返回前端

### 9.2 任务拆分

| ID | 任务 | 输出物 | 依赖 | 验收标准 |
|----|------|--------|------|----------|
| P3-1 | 设计四种场景的输入归一化与权重策略 | `QueryContextBuilder` | P1-1,P2-13 | 四种场景都有固定规则，不靠分散判断 |
| P3-2 | 实现 Query 改写器 | `QueryRewriteService` | P3-1 | 可并行生成多 Query，失败时回退原 Query |
| P3-3 | 实现 Chroma 检索器 | `VectorRetriever` | P2-8,P3-2 | 能按父块/子块检索 Top-K |
| P3-4 | 实现 BM25 检索器 | `KeywordRetriever` | P2-9,P3-2 | 能按关键词检索 Top-K |
| P3-5 | 实现 RRF 融合与去重 | `HybridRanker` | P3-3,P3-4 | 输出稳定的融合结果 |
| P3-6 | 实现 Reranker 重排序 | `RerankService` | P3-5 | 能从 Top-30 收敛到 Top-10 |
| P3-7 | 实现带来源的答案生成器 | `AnswerGenerationService` | P3-6 | 生成结果包含引用来源，不允许裸回答 |

| P3-9 | 前端会话对话区 UI | 输入框、消息区、模式状态 | P1-6,P3-8 | 能发起问答/续写并展示消息 |
| P3-10 | 前端流式消费器 | SSE 客户端 / 降级客户端 | P0-2,P3-8 | 可连续显示 token 或文本分片 |
| P3-11 | 前端来源卡片展示 | 来源列表组件 | P3-8,P3-9 | 能展示文献路径、页码、片段摘要 |

### 9.3 本阶段前端必须补齐的点

这部分正是前序计划里最容易漏掉的前端工作，必须明确写入：

- 输入区必须支持 `prompt` 与模式切换
- 消息区必须支持流式增量渲染
- 来源区必须支持点击查看来源
- 生成中状态必须支持取消或失败提示
- 对话状态必须与 session 绑定，不能只是临时展示

### 10.1 阶段目标

| P4-10 | 会话链路端到端验证 | 真实写作流程脚本 | P4-1~P4-9 | 可连续写作多轮，不丢上下文 |

### 10.3 常驻同步的实施约束

为了避免插件持续轮询对编辑体验造成影响，Phase 4 需要同时满足：

- 轮询频率默认 5 秒
- 当文档未变化时不重复上送
- 长文本只上送截断窗口，不全量传整篇
- 同步失败不打断当前编辑

## 12. 前端专项任务清单

前端并不是“做个壳子”，本项目至少有 8 个独立模块要落实。

| 模块 | 任务内容 | 阶段 |
|------|----------|------|
| 插件宿主层 | `manifest.xml`、入口页、WPS 生命周期管理 | Phase 1 |
| WPS 交互层 | Selection 读取、段落提取、轮询同步、命令封装 | Phase 0 / 1 / 4 |
| API 通信层 | HTTP 客户端、SSE 客户端、降级通信方案 | Phase 1 / 3 |

## 15. 验收标准

### 15.1 P0 验收

- 状态查询可看到成功或失败

## 16. 风险清单与应对策略

| 风险 | 影响 | 优先级 | 应对策略 |
|------|------|--------|----------|
| WPS 对 SSE 支持不稳定 | 流式输出受阻 | 高 | Phase 0 提前验证；失败则降级 |
| MinerU API 限流或不稳定 | 导入链路卡住 | 高 | 做任务状态机、重试、失败提示 |
| bbox 高亮能力不足 | 溯源体验弱于设计目标 | 高 | MVP 先保页级定位，bbox 作为增强 |
| 长文档轮询过重 | 影响编辑体验 | 中 | 只提取窗口文本，去重同步，频率可配 |
| ChromaDB 本地索引损坏 | 检索不可用 | 中 | 提供从解析产物重建索引能力 |
| 多模型/多服务配置复杂 | 调试成本高 | 中 | Phase 5 再开放设置项，前期固定默认配置 |

## 17. 待确认项

以下问题不阻止当前实现计划落地，但应在 Phase 0 或 Phase 1 尽快确认：

1. 来源跳转是否要求“直接在插件内展示 PDF”，还是允许打开外部查看器
2. SSE 在目标 WPS 版本中是否稳定，若不稳定优先降级到哪一种方案
3. 模型供应商是否在 MVP 阶段固定为一套，还是需要保留切换能力
4. 导入入口是纯路径输入，还是要在插件里额外做文件选择器

## 18. 推荐的实际执行顺序

如果按“最快拿到可演示版本”的目标，建议实际推进顺序如下：

1. 先做 `P0-1 ~ P0-4`
2. 然后做 `P1-1 ~ P1-8`
3. 接着做 `P2-2、P2-3、P2-5、P2-6、P2-8、P2-10、P2-11`
4. 在导入闭环可用后，再做 `P3-1 ~ P3-13`
5. 然后补 `P4-1 ~ P4-10`
6. 最后集中处理 `P5-1 ~ P5-10`

这个顺序的目标不是“功能平均推进”，而是尽快形成一个真实可演示闭环：

`导入 PDF -> 建索引 -> 输入问题 -> 检索回答 -> 展示来源 -> 多轮继续写`

## 19. 最终交付清单

MVP 结束时，至少应交付以下内容：

- 一套可加载的 WPS 插件

- 一份安装与运行说明
- 一份已知问题与边界说明
- 一套最小自动化测试
- 一组真实论文验证样例与演示脚本

## 20. 结论

这份实现计划的核心调整点有两个：

- 明确把前端工作拆成独立工作流，而不是默认“后端完成后顺手接 UI”
- 先把高风险交互点前置验证，避免在后期才发现 WPS 插件能力边界

因此，本项目不建议直接从 RAG 算法细节开始写代码，而应先完成：

`风险验证 -> 契约冻结 -> 骨架联调 -> 导入闭环 -> 问答闭环`

只要这个顺序不乱，后面的溯源、会话、打包都会顺很多。
