### 1.3 状态迁移时序（v4完整版）

**【代码事实】** `app/modules/ingestion/service.py:50-122`

```python
# 完整流程：MinerU 解析 → 清洗 → VLM 图片描述 → 混合切分 → Embedding → Qdrant 存储

def ingest_document(self, record: DocumentRecord) -> DocumentRecord:
    current_record = self.repository.save_document(record.transition("parsing"))
    
    # 阶段1: MinerU 解析 (pending → parsing)
    mineru_payload = self.mineru_client.run_pdf_task(current_record.file_path)
    
    # 阶段2: 清洗 (parsing → cleaning)
    current_record = self.repository.save_document(current_record.transition("cleaning"))
    cleaned_document = self._clean_document(current_record, mineru_payload)
    
    # 阶段3: VLM 图片描述 (cleaning，无状态变更)
    if image_count > 0:
        chunks_with_images = asyncio.run(self._describe_images(...))
    else:
        chunks_with_images = self._blocks_to_chunks(...)
    
    # 阶段4: 混合切分 (cleaning，无状态变更)
    text_chunks = self._chunk_chunks(current_record, chunks_with_images)
    
    # 阶段5: Embedding (cleaning → indexing)
    current_record = self.repository.save_document(current_record.transition("indexing"))
    embeddings = asyncio.run(self._embed_chunks(text_chunks))
    
    # 阶段6: Qdrant 存储 (indexing → completed)
    self.qdrant_store.add_chunks(current_record.document_id, final_chunks, embeddings)
    completed_record = current_record.transition("completed")
    return self.repository.save_document(completed_record)
```

**关键变化**：
- ✅ VLM描述阶段不引入新状态，在cleaning阶段内完成
- ✅ 状态机保持稳定，未因VLM集成而变更

## 二、VLM描述失败对状态机的影响

### 2.1 失败处理逻辑

**【代码事实】** `app/processing/describer.py:81-88`

```python
async def _describe_one(index: int, chunk: Chunk) -> tuple[int, str, str]:
    abs_path = paper_dir / chunk.image_path
    try:
        description = await vlm_client.describe_image(abs_path)
        return index, str(chunk.image_path), description
    except Exception as e:
        print(f"[WARN] VLM failed for {chunk.image_path}: {e}")
        return index, str(chunk.image_path), "description unavailable"
```

**【模型推断】**：
- VLM描述失败**不会**中断导入流程
- 失败的图片使用占位符 `"description unavailable"`
- 状态机继续推进到completed

**【代码事实】** `app/processing/describer.py:123-124`

### 3.2 失败标记

**【代码事实】** `app/modules/ingestion/service.py:334-342`

```python
def _mark_failed(self, record: DocumentRecord, error: IngestionError) -> DocumentRecord:
    """把当前记录推进到 failed，并写入统一错误信息."""
    failed_record = record.transition("failed").model_copy(
        update={
            "error_stage": error.stage,
            "error_message": error.message,
        }
    )
    return self.repository.save_document(failed_record)
```

**【模型推断】**：
- 任何阶段失败都会记录`error_stage`和`error_message`
- 失败后可通过`failed → pending`重新导入
- 失败不影响其他文档的导入

### 5.2 ￥问题￥：缺乏并发控制

**【代码事实】** `app/modules/ingestion/service.py:78`

```python
chunks_with_images = asyncio.run(self._describe_images(...))
```

**问题**：
- 同步函数内调用`asyncio.run()`，可能阻塞事件循环
- VLM并发由内部Semaphore控制，但外部无法感知

**影响**：
- 高并发导入时可能导致资源竞争
- 无法优雅取消正在进行的VLM请求

**建议**：
- 导入服务改为全异步
- 添加CancellationToken支持

### 5.3 ￥问题￥：无状态回滚机制

**【代码事实】** `app/modules/ingestion/service.py:102-103`

```python
self.qdrant_store.add_chunks(current_record.document_id, final_chunks, embeddings)
```

**问题**：
- 如果Qdrant存储失败，前面的Embedding结果无法回滚
- 部分成功的情况没有处理

**建议**：
- 实现事务性写入（先写临时表，成功后移动）
- 添加断点续传机制
