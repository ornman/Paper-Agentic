### 前端 API 客户端

#### API 客户端接口

**位置**：`src/services/api-client.ts`【243行】

**安全增强**：本地回环地址白名单

**证据**：`api-client.ts:44-109`
```typescript
const LOCAL_API_HOSTNAME_ALLOWLIST = new Set([
  '127.0.0.1', 'localhost', '[::1]', '::1'
])

function normalizeConfiguredApiBaseUrl(configuredBaseUrl: string): string {
  // 拒绝协议相对地址
  if (configuredBaseUrl.startsWith('//')) {
    return DEFAULT_API_BASE_URL
  }

  let parsedUrl: URL
  try {
    parsedUrl = new URL(configuredBaseUrl)
  } catch {
    return DEFAULT_API_BASE_URL
  }

  // 校验协议、主机名、路径
  const hasAllowedProtocol = parsedUrl.protocol === 'http:' ||
                             parsedUrl.protocol === 'https:'
  const hasAllowedHostname = LOCAL_API_HOSTNAME_ALLOWLIST.has(
    parsedUrl.hostname
  )
  const isPureOrigin = (parsedUrl.pathname === '/' || !parsedUrl.pathname) &&
                       !parsedUrl.search && !parsedUrl.hash

  if (!hasAllowedProtocol || !hasAllowedHostname || !isPureOrigin) {
    return DEFAULT_API_BASE_URL
  }

  return parsedUrl.origin
}
```

**理由**：
- 防止本地 PDF 路径泄露到外部服务器
- 导入接口会携带 `file_path` 参数
- 误配可能导致路径信息外传

### SSE 流式协议

#### 服务端发送事件格式

**位置**：`src/services/sse-client.ts`

**事件类型**：

1. **chunk** - 文本片段
   ```
   event: chunk
   data: {"text": "..."}
   ```

2. **sources** - 来源列表
   ```
   event: sources
   data: {"sources": [...]}
   ```

3. **done** - 完成
   ```
   event: done
   data: {}
   ```

4. **error** - 错误
   ```
   event: error
   data: {"message": "..."}
   ```

**证据**：`conversation.ts:336-361`
```typescript
await postAskInspirationStream(payload, {
  onChunk(chunkText) {
    if (status.value !== 'streaming') {
      startStreaming()

  },

    if (status.value !== 'streaming') {
```

## ￥问题￥ - 接口设计缺陷

### 1. ￥API 版本管理缺失￥

**事实**：所有端点都在 `/api/v1/`，但无版本切换机制

**问题**：
- 未来升级到 v2 时需要同时维护两套路由
- 无法平滑迁移客户端
- 破坏性变更无法感知

**建议**：
```python
# 添加版本协商
@api_version_header("X-API-Version")
def get_version():
    return {"current": "v1", "deprecated": []}
```

### 2. ￥SSE 连接中断无重连￥

**事实**：`sse-client.ts` 无自动重连逻辑

**问题**：
- 网络抖动导致请求失败
- 用户需要手动刷新
- 用户体验差

**证据**：`src/services/sse-client.ts` 无 `retry` 逻辑

### 3. ￥错误码不统一￥

**事实**：后端返回 `code` 字段，但前端只检查 `response.ok`

**问题**：
- 业务错误和 HTTP 错误混淆
- 前端无法区分错误类型
- 错误处理不精确

**证据**：
- 后端：`ApiResponseEnvelope<TData>` 有 `code` 字段
- 前端：`api-client.ts:204` 只检查 `response.ok`

### 4. ￥API 限流缺失￥

**事实**：所有端点无速率限制

**问题**：
- 恶意请求可能导致资源耗尽
- VLM 并发无限制（虽然配置了 `kimi_vlm_concurrency=10`）
- 无 API 配额管理

**风险**：
- Kimi API 账单爆炸
- 服务器资源被占满

## 接口调用矩阵

| 客户端 | 服务端 | 协议 | 频率 | 数据量 |
|--------|--------|------|------|--------|
| 前端 | FastAPI | HTTP/SSE | 中 | 小 |
| KimiVLMClient | Kimi API | HTTPS | 低 | 中（图片Base64） |
| KimiLLMClient | Kimi API | HTTPS | 高 | 小 |
| SiliconFlowEmbeddingClient | SiliconFlow | HTTPS | 高 | 大（批量文本） |
| SiliconFlowRerankClient | SiliconFlow | HTTPS | 中 | 中 |
| MineruClient | MinerU API | HTTPS | 低 | 大（PDF） |

**性能影响**：
- Embedding 是最频繁的大数据量调用（需要批量优化）
- VLM 调用最少但最慢（需要缓存）
- SSE 流式返回用户体验好但服务器压力大
