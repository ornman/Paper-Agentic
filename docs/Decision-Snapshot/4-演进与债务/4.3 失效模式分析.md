# 4.3 失效模式分析

**版本**: v4 (2025-04-10)  
**与v3版本的变化**: 新增VLM相关失效模式，前端简化后风险降低

## 一、VLM相关失效模式

### 1.1 VLM API限流

**失效场景**：
- 并发超过10个请求
- 短时间内大量图片需要描述

**【代码事实】** `app/processing/describer.py:79`

```python
semaphore = asyncio.Semaphore(CONCURRENCY)  # CONCURRENCY=10
```

**失效表现**：
- HTTP 429 Too Many Requests
- 描述请求被拒绝

**降级策略**：

**【代码事实】** `app/processing/describer.py:86-88`

```python
except Exception as e:
    print(f"[WARN] VLM failed for {chunk.image_path}: {e}")
    return index, str(chunk.image_path), "description unavailable"
```

**影响评估**：
- ✅ 不中断导入流程
- ⚠️ 图片描述降级为占位符
- 🟡 检索质量可能下降

**￥问题￥：无重试机制**

**问题**：
- 限流后立即失败
- 无退避重试
- 无全局限流

**建议**：
```python
@retry_async(
    max_retries=3,
    backoff_factor=2,  # 指数退避
    retry_on_exceptions=[RateLimitError],
)
async def _describe_one(...):
    ...
```

### 1.2 VLM超时

**失效场景**：
- 图片过大（>10MB）
- 网络延迟
- VLM服务响应慢

**失效表现**：
- `asyncio.TimeoutError`
- 连接超时

**降级策略**：
- 同上，使用占位符

**影响评估**：
- ✅ 不中断导入流程
- ⚠️ 部分图片无描述

**￥问题￥：无超时配置**

**问题**：
- 使用默认超时
- 无法配置超时时间

**建议**：
```python
async with asyncio.timeout(30):  # 30秒超时
    description = await vlm_client.describe_image(abs_path)
```

### 1.3 VLM返回格式错误

**失效场景**：
- API返回非文本
- JSON解析失败
- 空响应

**失效表现**：
- `ValueError`
- `JSONDecodeError`

**降级策略**：
- 使用占位符

**影响评估**：
- ✅ 不中断导入流程
- ⚠️ 单个图片失效

## 二、MinerU相关失效模式

**【代码事实】** `app/core/config.py:141`

```python
mineru_timeout: int = Field(default=300, ge=1, description="MinerU 最大等待时间（秒）")
```

**失效表现**：
- 5分钟后超时
- `IngestionError`

**降级策略**：

**【代码事实】** `app/modules/ingestion/service.py:334-342`

```python
def _mark_failed(self, record: DocumentRecord, error: IngestionError) -> DocumentRecord:
    failed_record = record.transition("failed").model_copy(
        update={
            "error_stage": error.stage,
            "error_message": error.message,

    return self.repository.save_document(failed_record)
```

**影响评估**：
- ❌ 导入失败
- ✅ 错误信息明确
- ✅ 可重新导入

**￥问题￥：无断点续传**

**问题**：
- 超时后需要重新开始

**建议**：

### 2.3 MinerU返回格式错误

**失效场景**：

**校验机制**：

**【代码事实】** `app/modules/ingestion/service.py:303-333`

```python
def _validate_success_payload_shape(self, mineru_payload: dict) -> None:

    if not isinstance(mineru_payload, dict):
        raise IngestionError(...)
    
    if not _MINERU_MINIMAL_SUCCESS_KEYS.issubset(mineru_payload.keys()):

    pages = mineru_payload.get("pages")
```

## 三、Embedding/Rerank失效模式

### 3.1 SiliconFlow限流

**失效场景**：
- 批量Embedding超过限流
- 短时间大量请求

**失效表现**：
- HTTP 429
- `RateLimitError`

**重试机制**：

**【代码事实】** `app/core/retry.py`

```python
@retry_async(max_retries=3)
async def _embed_chunks(self, chunks: list[dict]) -> list[list[float]]:
    """为 chunks 生成 Embedding（带重试）."""
    ...
```

**降级策略**：
- 重试3次
- 失败后导入失败

**影响评估**：
- ❌ 导入失败
- ✅ 有重试机制
- ⚠️ 无退避策略

**￥问题￥：无退避策略**

**问题**：
- 立即重试
- 可能再次触发限流

**建议**：
```python
@retry_async(
    max_retries=3,
    backoff_factor=2,  # 指数退避
    retry_on_exceptions=[RateLimitError],
)
```

### 3.2 Embedding维度不匹配

**失效场景**：
- API返回维度与配置不一致
- 模型版本变更

**校验机制**：

**【代码事实】** `app/core/config.py:286-298`

```python
@model_validator(mode="after")
def validate_pinned_model_contract(self) -> "Settings":
    """校验模型契约，防止向量维度和模型漂移污染索引。"""
    if self.embedding_model != PINNED_EMBEDDING_MODEL:
        raise ValueError(f"EMBEDDING_MODEL 必须固定为 {PINNED_EMBEDDING_MODEL}")
    
    if self.embedding_dimensions != PINNED_EMBEDDING_DIMENSIONS:
        raise ValueError(f"EMBEDDING_DIMENSIONS 必须固定为 {PINNED_EMBEDDING_DIMENSIONS}")
    
    return self
```

**影响评估**：
- ✅ 启动时校验
- ✅ 防止维度漂移
- ✅ 保护索引一致性

## 四、Qdrant失效模式

### 4.1 磁盘空间不足

**失效场景**：
- 数据目录磁盘满
- Qdrant无法写入

**失效表现**：
- `OSError`
- `DiskFull`

**降级策略**：
- 导入失败
- 错误记录

**￥问题￥：无磁盘检查**

**问题**：
- 事后才发现磁盘满
- 可能导致数据损坏

**建议**：
```python
def check_disk_space(path: str, min_free_gb: int = 10):
    """检查磁盘剩余空间."""
    stat = os.statvfs(path)
    free_gb = stat.f_bavail * stat.f_frsize / (1024**3)
    if free_gb < min_free_gb:
        raise IOError(f"磁盘空间不足: {free_gb:.1f}GB")
```

### 4.2 Collection已存在

**失效场景**：
- 重复导入同一文档
- Collection维度冲突

**处理机制**：

**【代码事实】** `app/stores/qdrant_store.py:80-92`

```python
if collection_name in existing:
    # 已存在，检查维度是否匹配
    info = self.client.get_collection(collection_name)
    if info.config.params.vectors.size != vector_size:
        raise ValueError(
            f"Collection {collection_name} exists with "
            f"dimension {info.config.params.vectors.size}, "
            f"but got {vector_size}"
        )
    return
```

**影响评估**：
- ✅ 维度不匹配时立即失败
- ✅ 维度匹配时复用Collection
- ⚠️ 无提示Collection已存在

### 4.3 Qdrant服务不可用

**失效场景**：
- 本地Qdrant进程未启动
- 端口被占用
- 配置错误

**失效表现**：
- 连接拒绝
- `ConnectionError`

**降级策略**：
- 导入失败
- 问答失败

**￥问题￥：无健康检查**

**问题**：
- 请求时才发现不可用
- 无主动监控

**建议**：
```python
@app.on_event("startup")
async def check_qdrant_health():
    """启动时检查Qdrant健康状态."""
    try:
        qdrant_store.get_collections()
    except Exception as e:
        raise RuntimeError(f"Qdrant不可用: {e}")
```

## 五、前端失效模式（v4简化后）

### 5.1 SSE连接中断

**失效场景**：
- 网络中断
- 后端崩溃
- 超时

**失效表现**：
- 流式输出停止
- 无错误提示

**处理机制**：

**【代码事实】** `src/services/sse-client.ts`

```typescript
eventSource.onerror = (error) => {
  console.error('SSE error:', error);
  eventSource.close();
};
```

**影响评估**：
- ✅ 连接中断后自动关闭
- ⚠️ 无自动重连
- ⚠️ 用户无感知

**￥问题￥：无自动重连**

**问题**：
- 连接中断后需手动刷新
- 可能丢失部分内容

**建议**：
```typescript
let retryCount = 0;
const maxRetries = 3;

eventSource.onerror = (error) => {
  if (retryCount < maxRetries) {
    setTimeout(() => {
      retryCount++;
      reconnect();
    }, 1000 * retryCount);
  } else {
    eventSource.close();
    showError('连接已断开');
  }
};
```

### 5.2 WPS API不可用

**失效场景**：
- WPS版本不兼容
- API变更
- 权限不足

**失效表现**：
- 功能不可用
- 报错

**降级策略**：
- 禁用相关功能
- 提示用户

**影响评估**：
- ✅ 前端简化后依赖减少
- ✅ 风险降低

## 七、￥问题￥汇总

### 7.1 高优先级

1. **无断点续传**（￥问题￥）
   - 位置：导入流程
   - 影响：超时后需重新开始
   - 建议：实现阶段状态持久化

2. **无磁盘检查**（￥问题￥）
   - 位置：Qdrant存储
   - 影响：磁盘满导致数据损坏
   - 建议：添加磁盘空间检查

### 7.2 中优先级

3. **VLM无重试**（￥问题￥）
   - 位置：VLM描述
   - 影响：限流后立即失败
   - 建议：添加退避重试

4. **无退避策略**（￥问题￥）
   - 位置：Embedding/Rerank
   - 影响：重试可能再次触发限流
   - 建议：实现指数退避

5. **SSE无重连**（￥问题￥）
   - 位置：前端SSE
   - 影响：连接中断需手动刷新
   - 建议：实现自动重连

### 7.3 低优先级

6. **无健康检查**（￥问题￥）
   - 位置：Qdrant
   - 影响：启动时才发现不可用
   - 建议：添加启动健康检查
