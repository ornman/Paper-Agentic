**【代码事实】** `app/core/config.py:189-201`

```python

    default=12,
    ge=4,
    le=64,
    description="HNSW m 参数（每个向量连接的最大节点数）",
)
qdrant_hnsw_ef_construct: int = Field(
    default=64,
    ge=10,
    le=200,
    description="HNSW ef_construct 参数（构建索引时的搜索范围）",
)

    default=5000,
    ge=1000,
    le=50000,
    description="全扫描阈值（小于此值直接全扫描）",

```

### 1.2 参数说明

| 参数 | 默认值 | 范围 | 作用 | 内存影响 |
|------|--------|------|------|----------|
| m | 12 | 4-64 | 每个向量连接数 | ↑m → ↑内存 |
| ef_construct | 64 | 10-200 | 构建时搜索范围 | ↑ef → ↑构建时间 |
| full_scan_threshold | 5000 | 1k-50k | 全扫描阈值 | ↓threshold → ↓内存 |

### 1.3 与v3版本对比

| 参数 | v3 | v4 | 变化原因 |
|------|----|----|----------|
| m | 16 (Qdrant默认) | 12 | 降低内存占用 |
| ef_construct | 100 (Qdrant默认) | 64 | 加快构建速度 |
| full_scan_threshold | 10000 (Qdrant默认) | 5000 | 小collection更高效 |

## 二、磁盘占用估算

### 2.1 单篇论文

**【代码事实】** `app/stores/qdrant_store.py:136-152`

```python
points = [
    PointStruct(
        id=uuid5(NAMESPACE_DNS, chunk.id),
        vector=embedding,  # 1536维 * 4字节 = 6KB
        payload={
            "content": chunk.content,        # 平均1KB
            "section": chunk.section,        # 100B
            "page": chunk.page,              # 8B
            "image_path": chunk.image_path,  # 200B
            "chunk_type": chunk.chunk_type,  # 20B
            "paper": chunk.paper,            # 50B
            "original_id": chunk.id,         # 50B
        },
    )
]
```

**估算模型**：
- **向量数据**: 1536维 × 4字节 × (1 + m/2) ≈ 6KB × 7 ≈ 42KB/点
- **元数据**: ~1.6KB/点
- **HNSW图**: m × 4字节 × 2 ≈ 12 × 8 ≈ 96B/点
- **索引**: ~10%额外开销

**单篇论文（100个chunk）**：
- 向量: 100 × 42KB ≈ 4.2MB
- 元数据: 100 × 1.6KB ≈ 160KB
- HNSW: 100 × 96B ≈ 9.6KB
- 索引: ~440KB
- **总计**: ~4.8MB/篇

### 2.2 86篇论文（当前规模）

**【模型推断】**：
- 86篇 × 4.8MB ≈ **412MB**

**包含Qdrant开销**：
- 索引文件: ~20%
- WAL日志: ~5%
- **实际**: ~520MB

### 2.3 1000篇论文（目标规模）

**【模型推断】**：
- 1000篇 × 4.8MB ≈ **4.8GB**

## 八、￥问题￥：资源配置风险

### 8.1 Qdrant内存优化代价

**问题**：
- m从16降到12，可能降低召回率
- ef_construct从100降到64，可能降低索引质量

**影响**：
- 检索准确性可能下降
- 需要实际测试验证

**建议**：
- A/B测试不同参数组合
- 根据实际检索质量调整

### 8.2 磁盘增长预测

**问题**：
- 1000篇论文需要6GB
- 未考虑日志、备份增长

**影响**：
- C盘空间有限（Windows）
- 可能影响系统稳定性

**建议**：
- 配置数据目录到D盘
- 实现定期清理机制
- 添加磁盘监控告警

### 8.3 VLM内存峰值

**问题**：
- 并发10个VLM请求，可能触发API限流
- 本地缓存50MB，可能持续增长

**影响**：
- 高并发导入时可能失败
- 内存持续增长可能OOM

**建议**：
- 添加LRU缓存淘汰
- 实现请求队列限流

3. **磁盘监控**
   ```python
   def check_disk_space(path: str, min_free_gb: int = 10):
       """检查磁盘剩余空间."""
       stat = os.statvfs(path)
       free_gb = stat.f_bavail * stat.f_frsize / (1024**3)
       if free_gb < min_free_gb:
           raise IOError(f"磁盘空间不足: {free_gb:.1f}GB")
   ```
