```
内存 = 向量数 × (m × 维度 × 4 bytes + 元数据)
```

| 论文数量 | chunks 数量 | Qdrant 内存 | 峰值内存（构建索引时） |
|---------|-----------|-----------|-------------------|
| 1 | 150 | 9 MB | 18 MB |
| 10 | 1500 | 90 MB | 180 MB |
| 100 | 15000 | 900 MB | 1.8 GB |
| 500 | 75000 | 4.5 GB | 9 GB |
| 2000 | 300000 | 18 GB | 36 GB |

**注意**：
- 这是 Qdrant 进程的内存占用，不是 Python 进程
- Qdrant 会将索引加载到内存以加速检索
- 如果内存不足，会触发 OOM

CACHE_FILENAME = "image_descriptions.json"

def _save_cache(paper_dir: Path, cache: dict[str, str]) -> None:
    """保存图片描述缓存."""
    cache_path = paper_dir / CACHE_FILENAME

### 【模型推断】临时文件残留问题

| 场景 | 临时文件 | 是否清理 | 后果 |
|------|---------|---------|------|
| 导入成功 | `image_descriptions.json` | ❌ 保留 | ✅ 加速下次导入 |

### 建议：清理策略

```python
# 伪代码（未来扩展）

class CleanupStrategy:
    """临时文件清理策略."""

    @staticmethod
    def cleanup_on_import_success(paper_dir: Path):
        """导入成功后保留缓存（加速下次导入）。"""
        pass  # 不清理

    @staticmethod
    def cleanup_on_import_failure(paper_dir: Path):
        """导入失败后删除损坏的缓存。"""
        cache_path = paper_dir / "image_descriptions.json"
        cache_path.unlink(missing_ok=True)

    @staticmethod
    def cleanup_on_delete_paper(paper_dir: Path):
        """删除论文时清理所有缓存。"""
        cache_path = paper_dir / "image_descriptions.json"
        cache_path.unlink(missing_ok=True)

    @staticmethod
    def cleanup_orphaned_cache(base_dir: Path):
        """定期清理孤立缓存文件."""
        for cache_path in base_dir.rglob("image_descriptions.json"):
            paper_dir = cache_path.parent
            # 检查对应的 PDF 是否还存在
            pdf_files = list(paper_dir.glob("*.pdf"))
            if not pdf_files:
                cache_path.unlink()
```

## CPU 资源消耗

### 【模型推断】CPU 密集型操作

| 操作 | CPU 占用类型 | 耗时（单篇） | 并发影响 |
|------|-------------|-------------|---------|
| MinerU API 调用 | 网络 I/O | 5-10 秒 | ✅ 可并发 |
| 清洗（字符串处理） | CPU 密集 | 1-2 秒 | ⚠️ Python GIL |
| VLM 图片描述 | 网络 I/O | 5 秒 × 10 张 | ✅ 可并发 |
| 混合切分（jieba 分词） | CPU 密集 | 2-3 秒 | 🔴 单线程 |
| Embedding | 网络 I/O | 2-5 秒 | ✅ 可并发 |
| Qdrant 存储 | 磁盘 I/O | 1-2 秒 | ⚠️ 本地模式单线程 |

### CPU 瓶颈分析

**【代码事实】同步阻塞点**

**文件位置**：`app/modules/ingestion/service.py:78, 96`

```python
chunks_with_images = asyncio.run(self._describe_images(...))  # 🔴 阻塞
embeddings = asyncio.run(self._embed_chunks(text_chunks))  # 🔴 阻塞
```

**【模型推断】影响**：

- 虽然 VLM 和 Embedding 本身是网络 I/O，可并发
- 但 `asyncio.run()` 会阻塞当前线程
- 如果 10 个用户同时导入，会占用 10 个 FastAPI 工作线程
- CPU 密集型操作（清洗、切分）会受 GIL 限制，无法真正并行

### 建议：CPU 优化策略

```python
# 伪代码（未来扩展）

# 1. 使用 ProcessPoolExecutor 处理 CPU 密集型任务
from concurrent.futures import ProcessPoolExecutor

executor = ProcessPoolExecutor(max_workers=4)

async def clean_document_async(document: dict) -> dict:
    loop = asyncio.get_event_loop()
    return await loop.run_in_executor(
        executor,
        clean_document_sync,  # CPU 密集型函数
        document,
    )

# 2. 全异步化，避免阻塞工作线程
async def ingest_document_async(record: DocumentRecord):
    chunks = await self._describe_images(...)  # ✅ 非阻塞
    embeddings = await self._embed_chunks(...)  # ✅ 非阻塞
```

## 总结

### 资源配置的优点

1. ✅ Qdrant 使用内存优化配置（m=12, ef_construct=64）
2. ✅ 分布式 Collection 设计，删除效率高
3. ✅ VLM 缓存加速重复导入

### 资源配置的局限

1. 🔴 无临时文件清理机制
2. 🔴 无资源监控
3. 🔴 同步阻塞点浪费线程资源
4. 🔴 CPU 密集型操作无并行化
