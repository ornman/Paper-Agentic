## 架构决策

### 为什么采用分布式多 Collection 架构？
￥这个改了，我们不要这个，资源紧张，不能这样￥
**证据**:
- `app/stores/qdrant_store.py:29-37` — 每篇论文独立 collection
- `app/modules/library/models.py:33-39` — `distributed` 模式定义，`parent_child` 已废弃
- `app/services_archive/retrieval_service.py:1-9` — 注释明确说明"分布式架构优势"

**原因分析**:
1. **原子性删除**: 删除论文时直接删除整个 collection，无需逐点清理
2. **可扩展性**: 预留支持多种资源类型（paper_xxx, video_xxx, note_xxx）
3. **知识图谱准备**: Collection 作为图节点，为未来图检索铺路
4. **隔离性**: 不同文献的向量物理隔离，避免互相干扰

**权衡**:
- Collection 数量随文献量增长，但 Qdrant 对此支持良好
- 需要维护 collection 命名规范（paper_{uuid}）

### 为什么废弃 parent_child 索引模式？

**证据**:
- `app/modules/library/models.py:34` — `parent_child` 标记为"已废弃，保留用于兼容性"
- `app/modules/library/models.py:38` — 仍在 `DOCUMENT_INDEX_MODES` 中保留

**原因分析**（需人工补充）:
- 推测：parent_child 模式复杂度高于收益
- 维持兼容性是为了不破坏已有数据的 index_mode 字段
￥这个索引指的是？具体逻辑是什么？再回去详细看看

【AI 回答】parent_child 是**父子块索引模式**（从测试代码推导）：
- 父块：600-700 token 的大块，保留完整上下文
- 子块：从父块切分的小块，用于精细检索
- 回挂机制：子块命中后回挂父块获取完整上下文

废弃原因：复杂度高，distributed 模式更简单。

从简，先跑通，之后再优化。

### 为什么使用 RRF（Reciprocal Rank Fusion）融合？

**证据**:
- `app/services_archive/retrieval_service.py:24-61` — RRF 算法实现
- 注释说明核心思想："不同 Collection 的分数可能不可比，RRF 只看排名"

**原因分析**:
1. **分数归一化难题**: 不同 Collection 的相似度分数分布可能不一致
2. **排名公平性**: RRF 只看排名位置（1/(k+rank+1)），不依赖绝对分数
3. **多 Query 融合**: 支持 Query 改写后的多路检索结果融合
4. **参数可调**: k 值（默认 60）可调节高位排名权重

**代码位置**: `app/services_archive/retrieval_service.py:24-61`

### 为什么切分策略如此复杂？

**证据**:
- `app/processing/chunker.py:1-8` — 注释说明混合策略
- `app/processing/chunker.py:84-136` — `pack_chunks_if_fit()` 打包逻辑
- `app/processing/chunker.py:138-202` — `split_long_chunk()` 切分逻辑
- `app/processing/chunker.py:205-310` — `chunk_by_semantic_units()` 主策略

**3 层策略**:
1. **打包**: 两个语义块合计 ≤32k → 打包一起
2. **分开**: 超过 32k 但单个都不超 → 分开，不用重叠
3. **切分 + 重叠**: 单个超过 32k → 平均切分到 24k，首尾重叠接近 32k
￥这个切分和重叠策略之后再说，这个也有问题的，不能这样￥
**参数**（`app/core/config.py:209-230`）:
- `chunk_max_context`: 32000（Embedding 模型上限）
- `chunk_target_size`: 24000（切分目标）
- `chunk_overlap_buffer`: 8000（重叠缓冲）
- `chunk_pack_threshold`: 32000（打包阈值）

**原因分析**:
1. **平衡语义完整性**: 尽量不切断语义单元
2. **利用模型上下文**: 接近 32k 但不超过
3. **重叠防止边界信息丢失**: 首尾重叠确保连续性
4. **可配置**: 所有参数都可在配置文件调整
