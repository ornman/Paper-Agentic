# 3-为什么这样 -- 决策记录

> 基于 Q3 问题框架：为什么这样设计？背后的权衡和约束是什么？
> 所有结论均有代码行号证据，区分"设计决策"与"临时方案"。

---

## 关键设计决策

### D1: 分布式 Qdrant（每篇论文一个 Collection）

| 维度 | 内容 |
|------|------|
| **决策** | 每篇论文创建独立 Qdrant Collection（命名 `paper_{id}`） |
| **原因** | 频繁更新/删除操作简单（删 Collection 即可，无需按 ID 逐点删除）；论文量 >=2000 时单 Collection 过大；天然支持按类型隔离（paper/video/note）；未来知识图谱扩展时 Collection 可作为图节点 |
| **权衡** | Collection 数量膨胀时 list 操作变慢；跨 Collection 检索需要并行遍历 |
| **证据** | `app/stores/qdrant_store.py:3-4`（文件头注释）、`app/stores/qdrant_store.py:29-37`（`_get_collection_name`）、`app/stores/qdrant_store.py:251-263`（`delete_paper` 直接删 Collection） |
| **风险** | ⚠️ 当论文数超过数千篇时，`search_all` 的全 Collection 遍历会成为性能瓶颈（见 `qdrant_store.py:226-249`） |

### D2: Embedding/Rerank 模型固定契约

| 维度 | 内容 |
|------|------|
| **决策** | Embedding 模型（Qwen3-Embedding-8B）、向量维度（1536）、Rerank 模型（Qwen3-Reranker-8B）写死为常量，启动时校验 |
| **原因** | 更换 Embedding/Rerank 模型会导致向量库中所有历史向量失效（维度/语义空间不同），检索结果和索引结构失去一致性 |
| **权衡** | 灵活性受限；未来换模型需要全量重建索引 |
| **证据** | `app/core/config.py:9-12`（`PINNED_*` 常量）、`app/core/config.py:108-110`（"系统硬约束"注释）、`app/core/config.py:318-330`（`validate_pinned_model_contract` 启动校验） |
| **风险** | 🔴 硬约束——无法在不重建索引的情况下切换模型 |

### D3: 客户端抽象接口（ABC 四件套）

| 维度 | 内容 |
|------|------|
| **决策** | VLMClient、LLMClient、EmbeddingClient、RerankClient 四个抽象基类，定义在 `vlm_client.py`，具体实现分别在 `kimi_client.py`、`embedding_client.py`、`rerank_client.py` |
| **原因** | MVP 阶段使用 Kimi Coding API（VLM/LLM）+ 硅基流动（Embedding/Rerank），但未来可能切换到 OpenAI/Azure 等其他供应商。抽象层保证业务代码不依赖具体实现 |
| **权衡** | 增加了一层间接；当前只有单一实现，抽象的价值尚未被多实现验证 |
| **证据** | `app/clients/vlm_client.py:1-5`（"支持多种实现"注释）、`app/clients/vlm_client.py:14-184`（四个 ABC 完整定义）、`app/clients/embedding_client.py:15-18`（"可替换为其他实现"注释） |
| **风险** | ⚠️ 两套 LLM 客户端并存：`clients/llm_client.py`（OpenAI 协议，Query 改写用）和 `clients/kimi_client.py`（Kimi Coding 协议，VLM/LLM 用），职责边界不清晰 |

### D4: SQLite + WAL 模式 + StaticPool

| 维度 | 内容 |
|------|------|
| **决策** | SQLite 作为元数据库，WAL 模式 + StaticPool + 双重检查锁单例引擎 |
| **原因** | MVP 阶段不需要独立数据库服务；SQLite 单进程场景下 WAL 模式允许读写并发；StaticPool 是 SQLite 单进程最佳连接池选择 |
| **权衡** | 无法多进程写入；未来多实例部署时需要迁移到 PostgreSQL |
| **证据** | `app/repositories/sqlite_repo.py:14-21`（`_get_engine` 注释说明三重优化）、`app/repositories/sqlite_repo.py:37-41`（`check_same_thread`、`StaticPool`、`pool_pre_ping`）、`app/repositories/sqlite_repo.py:43-50`（WAL PRAGMA 配置）、`app/repositories/sqlite_repo.py:57-72`（线程锁保护的单例模式） |
| **风险** | ⚠️ `sqlite_repo.py:65` 使用 `datetime.utcnow()`（已弃用，应改为 `datetime.now(datetime.UTC)`），不影响功能但属于技术债 |

### D5: 文档状态机（7 状态 + 显式迁移表）

| 维度 | 内容 |
|------|------|
| **决策** | 文档 7 个状态（pending/parsing/cleaning/indexing/completed/failed/deleting/deleted），迁移表显式定义允许的转换方向 |
| **原因** | 防止文档进入非法状态（如从 completed 直接跳到 parsing）；failed 状态只能回 pending 重试或进 deleting 删除，避免"僵尸状态" |
| **权衡** | 状态机约束严格，某些修复操作需要先手动改状态 |
| **证据** | `app/modules/library/models.py:19-29`（`DOCUMENT_STATUSES` 集中定义）、`app/modules/library/models.py:62-71`（`ALLOWED_STATUS_TRANSITIONS` 迁移表）、`app/modules/library/models.py:136-145`（`transition` 方法做校验并返回新对象） |
| **风险** | ⚠️ `parent_child` 索引模式已废弃但仍保留在 `DOCUMENT_INDEX_MODES` 中（`models.py:38`），可能误导使用者 |

### D6: 统一领域错误体系（DomainError + 5 特化类）

| 维度 | 内容 |
|------|------|
| **决策** | `DomainError` 基类包含 code/stage/message/detail 四个业务字段，5 个特化错误（ConfigError/IngestionError/IndexingError/RetrievalError/QAError）按阶段固定 stage |
| **原因** | 统一所有业务错误暴露的字段结构，避免各模块自行拼装错误对象；上层路由/服务/日志系统可以稳定读取统一字段 |
| **权衡** | 增加了 dataclass + Exception 继承的复杂度（需要处理 `__reduce__`、`__post_init__`、pickle 兼容等） |
| **证据** | `app/core/errors.py:33-41`（DomainError 设计目的注释）、`app/core/errors.py:48-65`（`__post_init__` 解决 dataclass 不自动调用 Exception.__init__ 的问题）、`app/core/errors.py:67-90`（`__reduce__`/`__reduce_ex__` 解决 pickle 重建问题） |
| **风险** | 无明显风险，设计合理 |

### D7: Query 改写 4 场景模板

| 维度 | 内容 |
|------|------|
| **决策** | 根据用户输入组合（prompt/已写内容/圈选文本）分为 4 种场景，各有独立改写 Prompt 模板和权重分配 |
| **原因** | 不同输入组合背后的检索意图不同；纯提问需要关键词提取，续写需要意图推断，圈选需要聚焦特定概念 |
| **权衡** | 4 套模板维护成本高；场景判断逻辑如果扩展会变得复杂 |
| **证据** | `app/services/query_rewrite_service.py:1-8`（设计来源 D042 决策，4 场景说明）、`app/services/query_rewrite_service.py:21-88`（4 套 Prompt 模板）、`app/services/query_rewrite_service.py:130-177`（`_select_prompt` 场景判断逻辑） |
| **风险** | ⚠️ 改写失败时静默降级到原始查询（`query_rewrite_service.py:127`），用户无法感知改写是否成功 |

### D8: 检索降级策略（Embedding 失败 -> BM25）

| 维度 | 内容 |
|------|------|
| **决策** | 向量检索失败时自动降级到 BM25 关键词检索，响应中标记 `degraded: true` |
| **原因** | Embedding 服务（硅基流动）可能因网络/限流/服务故障不可用；BM25 纯本地计算，无外部依赖，作为兜底保证基本可用 |
| **权衡** | BM25 检索质量远低于向量检索；降级 payload 简化为 `[BM25 检索] 文档 ID: {doc_id}`，丢失原文内容 |
| **证据** | `app/modules/retrieval/service.py:20-21`（"支持 Embedding 降级"注释）、`app/modules/retrieval/service.py:62-121`（try/except 降级逻辑）、`app/modules/retrieval/service.py:95`（`degraded: False`）、`app/modules/retrieval/service.py:120`（`degraded: True`） |
| **风险** | 🔴 降级到 BM25 时 `payload.content` 被替换为占位文本（`retrieval/service.py:110`），下游 QA 服务拿到的引用内容不可用 |

### D9: Kimi Coding API 的 User-Agent 伪装

| 维度 | 内容 |
|------|------|
| **决策** | 调用 Kimi Coding API 时必须设置 `User-Agent: claude-code` |
| **原因** | Kimi Coding API 的鉴权机制要求此 User-Agent，不带则请求被拒 |
| **权衡** | 依赖非标准鉴权行为；如果 Kimi 更改策略会导致所有 VLM/LLM 调用失败 |
| **证据** | `app/clients/kimi_client.py:3-4`（"必须伪装 User-Agent: claude-code 才能正常使用"）、`app/clients/kimi_client.py:24-29`（`_HEADERS` 包含 `User-Agent: claude-code`） |
| **风险** | 🔴 非标准鉴权依赖，随时可能失效 |

### D10: MinerU 结果 URL 安全校验

| 维度 | 内容 |
|------|------|
| **决策** | 只允许同源结果地址或白名单 CDN（`cdn-mineru.openxlab.org.cn`）；同源才携带 Bearer Token，CDN 不携带 |
| **原因** | 防止 SSRF：MinerU API 返回的 `result_url` 可能指向外部地址，如果直接携带 Token 请求会导致凭证泄露 |
| **权衡** | 白名单硬编码，MinerU 更换 CDN 时需同步更新代码 |
| **证据** | `app/modules/ingestion/mineru_client.py:239-241`（"安全目标"注释）、`app/modules/ingestion/mineru_client.py:445-481`（`_validate_result_url` 方法）、`app/modules/ingestion/mineru_client.py:470`（`allowed_result_hosts` 白名单） |
| **风险** | ⚠️ CDN 域名硬编码在代码中（`mineru_client.py:470`），应提取为配置项 |

### D11: 配置旧字段兼容层

| 维度 | 内容 |
|------|------|
| **决策** | Settings 类保留旧属性名（`llm_api_key`/`embedding_api_key` 等）作为 `@property`，映射到新字段 |
| **原因** | 重构配置字段名时（如 `llm_api_key` -> `deepseek_api_key`），旧客户端代码和路由如果同时崩掉，影响面太大。兼容层让迁移可以渐进式完成 |
| **权衡** | 兼容层增加了配置类的复杂度和维护成本 |
| **证据** | `app/core/config.py:332-333`（"旧字段兼容层"注释）、`app/core/config.py:334-373`（7 个 `@property` 兼容映射） |
| **风险** | ⚠️ 兼容层无过期计划，可能永久存在 |

### D12: 文档库服务层边界校验

| 维度 | 内容 |
|------|------|
| **决策** | 服务层显式校验输入（路径规范化、PDF 后缀、路径遍历检测），而非依赖路由层 |
| **原因** | 路由层不是唯一入口——测试和未来任务可能直接调用服务层；仓储层只负责落库，不应承担业务输入校验 |
| **权衡** | 服务层和路由层存在部分校验重复 |
| **证据** | `app/modules/library/service.py:45-48`（"路由层不是唯一入口"的校验原因说明）、`app/modules/library/service.py:138-183`（`_validate_import_pdf_path` 的 5 项安全检查）、`app/modules/library/service.py:169`（使用 `os.path` 而非 `Path` 处理 Windows 中文路径） |
| **风险** | 无明显风险 |

### D13: import_document 与 import_pdf 分离

| 维度 | 内容 |
|------|------|
| **决策** | `import_document` 只登记（最小语义），`import_pdf` 登记后触发完整导入链路 |
| **原因** | Task 3 的 `import_document` 语义是"只登记"；Task 4 新增的是"登记后继续导入"。拆成两个方法避免把原有行为悄悄改坏，测试边界也更清晰 |
| **权衡** | 两个公开方法需要文档说明各自的适用场景 |
| **证据** | `app/modules/library/service.py:72-79`（"为什么不直接复用 import_document"的详细原因）、`app/modules/library/service.py:39-43`（Task 3 只要求建立记录） |
| **风险** | 无明显风险 |

### D14: Qdrant HNSW 参数降低默认值

| 维度 | 内容 |
|------|------|
| **决策** | HNSW `m` 从默认 16 降到 12，`ef_construct` 从 100 降到 64，`full_scan_threshold` 从 10000 降到 5000 |
| **原因** | 降低内存占用；每篇论文 Collection 数据量相对较小（通常数百到数千 chunks），不需要高精度 HNSW 参数 |
| **权衡** | 检索精度略微下降；参数已在配置中暴露，可按需调整 |
| **证据** | `app/stores/qdrant_store.py:94-99`（注释说明降低策略和三个参数）、`app/core/config.py:222-239`（配置项定义，含默认值和范围约束） |
| **风险** | 无明显风险，参数已配置化 |

## 已知限制

- **L1**: 用户自选文献功能仅预留接口（`selected_papers` 参数），Collection 过滤逻辑被注释掉。证据：`app/modules/retrieval/service.py:56-60`
- **L2**: 用户自选数据类型（resource_types）仅预留接口，未实现过滤。证据：`app/modules/retrieval/service.py:40`
- **L3**: 检索降级到 BM25 时 payload 内容被替换为占位文本，下游 QA 无法使用。证据：`app/modules/retrieval/service.py:110`
- **L4**: KimiLLMClient 的 chat_stream 是伪流式（先全部获取再分块），非真正的 SSE 流式。证据：`app/clients/kimi_client.py:209-224`
- **L5**: Qdrant `search_all` 方法遍历所有 Collection，论文量大时性能堪忧。证据：`app/stores/qdrant_store.py:226-249`
- **L6**: 未实现真正的批量 Embedding + 索引流程。`services_archive/ingest_service.py` 存在旧实现，新架构下 `modules/ingestion/service.py` 已重写但端到端流程未完整验证
- **L7**: 前后端对接未完成，当前后端 API 独立运行，未与 Vue 前端集成
