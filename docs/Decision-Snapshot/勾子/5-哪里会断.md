## 高危风险

### 1. MinerU 同步轮询阻塞整个 FastAPI 事件循环

- **位置**: `app/modules/ingestion/mineru_client.py:159-233` (`poll_task` 方法)
- **触发条件**: 任何 PDF 导入请求（`/api/v1/library/import` 或 `/api/v1/library/import-path`）
- **影响**: MinerU 轮询使用 `time.sleep()` + 同步 `httpx.Client`，在 async 函数 `ingest_document` 内调用时，**整个 FastAPI 事件循环被阻塞**，所有其他请求（包括健康检查）全部挂起，直到 PDF 解析完成（最长 300 秒）
- **代码证据**:
  ```python
  # mineru_client.py:233
  self.sleep_fn(actual_interval)  # time.sleep() 在 async 上下文中
  # mineru_client.py:44
  self.http_client = http_client or httpx.Client(timeout=self.timeout)  # 同步客户端
  ```
  ```python
  # ingestion/service.py:60
  mineru_payload = self.mineru_client.run_pdf_task(current_record.file_path)  # 在 async def ingest_document 中调用同步方法
  ```
- **当前缓解**: 无。这是目前最严重的架构风险。
- **风险级别**: 高危

### 2. API Key 为空字符串时运行时才崩溃

- **位置**: `app/core/config.py:39-48`, `app/clients/kimi_client.py:25`, `app/clients/llm_client.py:16`
- **触发条件**: `.env` 文件中 `DEEPSEEK_API_KEY` / `KIMI_API_KEY` / `SILICONFLOW_API_KEY` 任一缺失或为空
- **影响**: **应用正常启动**，config 中这些字段默认值是空字符串 `""`，不会报错。但当用户第一次触发 LLM 问答 / VLM 描述 / Embedding 时才会崩溃
- **代码证据**:
  ```python
  # config.py:39-48 — 所有 API Key 默认值都是空字符串
  deepseek_api_key: str = Field(default="", ...)
  siliconflow_api_key: str = Field(default="", ...)
  kimi_api_key: str = Field(default="", ...)
  mineru_api_key: str = Field(default="", ...)
  ```
  ```python
  # kimi_client.py:25 — 空字符串被直接用作请求头
  _HEADERS = {
      "x-api-key": settings.kimi_api_key,  # 空字符串会导致 401
  ```
- **当前缓解**: 无启动校验。用户只能通过运行时报错发现问题。
- **风险级别**: 高危

### 3. Embedding 客户端 429 递归重试导致无限循环

- **位置**: `app/clients/embedding_client.py:188-194`
- **触发条件**: SiliconFlow API 返回 429（速率限制）
- **影响**: `_embed_batch_raw` 在捕获 429 后直接递归调用自身（`await self._embed_batch_raw(texts)`），没有任何重试次数限制。如果 API 持续返回 429，会导致无限递归，最终栈溢出
- **代码证据**:
  ```python
  # embedding_client.py:188-194
  except httpx.HTTPStatusError as e:
      if e.response.status_code == 429:
          await asyncio.sleep(1)
          return await self._embed_batch_raw(texts)  # 无限递归！
      raise
  ```
  注意：外层 `@retry_async(max_retries=3)` 不影响这个内层的无限递归，因为 429 被内部消化了，不会抛到外层触发 retry 装饰器
- **当前缓解**: 无
- **风险级别**: 高危

### 4. BM25 降级检索返回的 payload 缺少实际内容

- **位置**: `app/modules/retrieval/service.py:103-114`
- **触发条件**: Embedding 服务宕机或超时，触发降级到 BM25
- **影响**: BM25 降级路径返回的 payload 只有 `[BM25 检索] 文档 ID: {doc_id}` 的占位文本，**没有实际文档内容**。这意味着 RAG 问答的 LLM 收到的"参考文献"是空的，用户会得到无引用的回答，但不会收到任何降级提示
- **代码证据**:
  ```python
  # retrieval/service.py:110
  "payload": {"content": f"[BM25 检索] 文档 ID: {doc_id}"},  # 简化 payload — 没有实际内容！
  ```
- **当前缓解**: 有降级标记 `degraded: True`，但前端未消费此字段
- **风险级别**: 高危

### 5. QdrantStore 每次 API 请求都创建新客户端实例

- **位置**: `app/api/v1/routes/library.py:31-32`, `app/stores/qdrant_store.py:43-64`
- **触发条件**: 每次调用 library 相关 API
- **影响**: `_get_store()` 每次调用都 `return QdrantStore()`，创建新的 QdrantClient 实例。对于本地 Qdrant，这意味着每次请求都重新打开数据库文件、加载索引，造成严重的延迟和资源浪费
- **代码证据**:
  ```python
  # library.py:31-32
  def _get_store() -> QdrantStore:
      return QdrantStore()  # 每次新建实例
  ```
- **当前缓解**: 无
- **风险级别**: 高危

## 警告级风险

### 6. VLM 同步入口 `describe_image` 在 async 上下文中调用 `asyncio.run()` 会崩溃

- **位置**: `app/clients/kimi_client.py:322-324`
- **触发条件**: 在已有事件循环的环境中（如 FastAPI 请求处理中）调用同步 `describe_image()`
- **影响**: `asyncio.run()` 不能在已有事件循环中调用，会抛出 `RuntimeError: This event loop is already running`
- **代码证据**:
  ```python
  # kimi_client.py:324
  def describe_image(...) -> str:
      import asyncio
      return asyncio.run(describe_image_async(...))  # 如果在 FastAPI 中调用就会崩溃
  ```
- **当前缓解**: 调用方（`cleaning.py`）已经使用 async 版本，但如果有人误用同步版本会直接崩溃
- **风险级别**: 警告

### 7. SQLite 全局单例引擎可能被并发写入竞争

- **位置**: `app/repositories/sqlite_repo.py:56-72`
- **触发条件**: 多个并发请求同时触发写入操作（如同时创建会话 + 导入 PDF + 保存消息）
- **影响**: 虽然已配置 `check_same_thread=False` + `StaticPool` + WAL 模式，但 StaticPool 只有**一个连接**，并发写入实际上会串行化。如果写入耗时较长，后续写入请求会阻塞等待
- **代码证据**:
  ```python
  # sqlite_repo.py:39
  poolclass=StaticPool,  # SQLite 单连接池
  ```
- **当前缓解**: WAL 模式允许读写并发，但写-写仍然是串行的
- **风险级别**: 警告

### 8. PDF 上传接口无文件大小限制

- **代码证据**:

### 17. LLMClient 和 KimiLLMClient 同时存在，职责重叠

- **位置**: `app/clients/llm_client.py` 和 `app/clients/kimi_client.py`
- **触发条件**: 不同模块使用不同的 LLM 客户端
- **影响**: `QAService` 使用 `KimiLLMClient`，而 `QueryRewriteService` 使用 `LLMClient`（基于 OpenAI SDK 的 DeepSeek 实现）。两个客户端使用不同的 API Key（Kimi vs DeepSeek），配置不同（timeout 120s vs 60s），如果其中一个 API 不可用，会影响不同功能
- **代码证据**:
  ```python
  # qa/service.py:75 — QA 使用 KimiLLMClient
  self.llm_client = llm_client or KimiLLMClient()

  # query_rewrite_service.py:95 — Query 改写使用 LLMClient（DeepSeek）
  self._llm = LLMClient()
  ```
- **当前缓解**: 两者都有降级逻辑（改写失败回退原始查询，LLM 失败回退错误消息）
- **风险级别**: 警告

### 18. 前端 SSE 重试机制会重复已接收的 chunks

- **位置**: `project/MVP/frontend/src/services/sse-client.ts:483-486`
- **触发条件**: SSE 连接中断后自动重试
- **影响**: 重试时会重新发送 `handlers.onChunk?.(receivedChunks.join(''))`，但这只是把之前收到的所有 chunk 拼在一起重新传给 handler。而后端没有断点续传机制——重试会重新发起完整请求，导致前端**重复接收**之前已经显示的内容
- **代码证据**:
  ```typescript
  // sse-client.ts:483-486
  if (receivedChunks.length > 0) {
      console.log(`恢复已接收的 ${receivedChunks.length} 个 chunks`)
      handlers.onChunk?.(receivedChunks.join(''))  // 和新请求的 chunk 叠加
  }
  ```
- **当前缓解**: 无。前端 store 层直接 append chunk，会导致内容重复
- **风险级别**: 警告

## 优先修复建议（按影响排序）

1. **MinerU 同步改异步**（风险 #1）— 将 `httpx.Client` 改为 `httpx.AsyncClient`，`time.sleep` 改为 `asyncio.sleep`，避免阻塞事件循环
2. **API Key 启动校验**（风险 #2）— 在 `Settings.validate_pinned_model_contract` 中增加对非空 API Key 的校验，启动时 fast-fail
3. **Embedding 429 无限递归**（风险 #3）— 添加重试次数计数器或移除内层的递归重试，让外层 `@retry_async` 统一处理
4. **BM25 降级 payload**（风险 #4）— BM25Repo 需要存储文档原文，降级时返回实际内容
5. **QdrantStore 单例化**（风险 #5）— 使用依赖注入或全局单例，避免每次请求新建实例
6. **query.py 缺少 import asyncio**（风险 #15）— 在文件顶部添加 `import asyncio`
7. **SSE 重试去重**（风险 #18）— 重试时清空已接收 chunks 或让后端支持断点续传
