# 3-为什么 — 设计决策记录

## 为什么选择 FastAPI？

**证据**：
- `app/main.py:14` - FastAPI 应用入口
- `app/api/v1/routes/*.py` - 路由模块化设计

**决策原因**（需人工补充完整业务背景）：
1. **异步原生支持** - RAG 链路涉及多个 IO 密集操作（MinerU 调用、Embedding、VLM 描述、LLM 流式输出），FastAPI 的 `async/await` 可避免阻塞
2. **类型安全** - Pydantic 自动校验请求/响应，减少运行时错误
3. **OpenAPI 集成** - 自动生成 API 文档，便于前后端协作

**风险**：
- ⚠️ 同步依赖（如 SQLAlchemy）在异步上下文需特殊处理（当前代码未体现 ORM 集成，仅用 SQLite 仓储）

## 为什么 Qdrant 采用"一论文一 Collection"架构？

**证据**：
- `app/stores/qdrant_store.py:29-37` - Collection 命名规则 `paper_{paper_id}`
- `app/stores/qdrant_store.py:66-92` - `create_paper_collection()` 为每篇论文创建独立 Collection
- `app/stores/qdrant_store.py:196-235` - `search_all()` 遍历所有 Collection

**决策原因**：
1. **论文级隔离** - 删除单篇论文时可直接删除整个 Collection，无需精确筛选向量点
2. **权限控制预留** - 未来可按 Collection 设置访问权限（如付费论文隔离）
3. **元数据查询简化** - 不需在 payload 中存储 `paper_id` 字段（Collection 名称已隐含）

**代价**：
- 🔴 跨论文检索需遍历多个 Collection，当论文数 >1000 时性能会退化（代码未体现批量优化）
- ⚠️ Collection 数量受 Qdrant 集群限制（需查阅官方文档）

**未来扩展证据**：
- `app/modules/retrieval/service.py:45-49` - 预留 `selected_papers` 过滤逻辑（注释状态）

## 为什么 Embedding/Rerank 模型硬编码为固定契约？

**证据**：
- `app/core/config.py:8-12` - 模型名称定义为模块级常量
- `app/core/config.py:103-132` - `embedding_model`/`rerank_model` 字段带 `@model_validator` 校验
- `app/core/config.py:286-298` - 启动时强制校验模型契约

**决策原因**：
1. **向量一致性** - 不同模型的 Embedding 空间不可互操作，历史向量会失效
2. **检索稳定性** - Rerank 模型更换会导致排序逻辑漂移，影响检索质量
3. **成本控制** - SiliconFlow 统一管理 Embedding/Rerank，避免多服务商混用导致契约漂移

**风险**：
- 🔴 无法快速切换到更好的模型（如需升级需手动重建所有索引）
- ⚠️ 测试环境无法使用廉价模型（如 `text-embedding-3-small`）降级成本

## 为什么采用"六阶段导入链路"而非"单函数暴力处理"？

**证据**：
- `app/modules/ingestion/service.py:50-114` - 清晰的阶段划分（parsing → cleaning → VLM 描述 → 切分 → Embedding → 存储）
- `app/modules/library/models.py` - DocumentRecord 状态机（pending → parsing → cleaning → indexing → completed）

**决策原因**：
1. **断点续传** - 每阶段独立记录状态，失败后可从断点恢复（见 `app/modules/library/service.py:92-128` `resume_import()`）
2. **错误隔离** - 单个阶段失败不影响其他阶段（如 VLM 失败可降级跳过）
3. **可观测性** - 每阶段打印日志（`[INGEST] 阶段X: ...`），便于定位瓶颈

**代价**：
- ⚠️ 状态迁移增加代码复杂度（需维护 `transition()` 方法）
- 🔴 没有中间缓存（INGEST_DESIGN.md 提到需 JSON 缓存，但代码未实现）

## 为什么 Chunker 采用"混合语义切分"而非固定窗口？

**证据**：
- `app/processing/chunker.py:1-8` - 策略注释（打包阈值 32k，目标大小 24k，重叠缓冲 8k）
- `app/processing/chunker.py:84-136` - `pack_chunks_if_fit()` 先尝试打包
- `app/processing/chunker.py:138-202` - `split_long_chunk()` 超长时平均切分 + 重叠

**决策原因**：
1. **语义完整性** - 两个小章节（如 3.1 和 3.2）合计 ≤32k 时打包，避免截断论述
2. **上下文保留** - 超长块切分时首尾重叠接近 32k（Embedding 模型上限），保留跨块引用
3. **适配中文** - 用 `default_token_counter()` 按"中文×1.5 + 英文×0.75"估算，避免 tiktoken 依赖

**风险**：
- ⚠️ Token 估算不精确（实测误差 ±10%，需人工补充验证数据）
- 🔴 重叠策略可能产生冗余向量（检索时同一段落可能出现多次）

## 为什么前端采用"流式 SSE"而非 WebSocket？

**证据**：
- `frontend/src/services/sse-client.ts:1` - 文件命名即为 `sse-client`
- `frontend/src/services/sse-client.ts:371-394` - `postAskInspirationStream()` 使用 `Accept: text/event-stream`
- `backend/app/api/v1/routes/query.py:79-87` - `StreamingResponse` 媒体类型 `text/event-stream`

**决策原因**：
1. **单向流式** - 问答场景只需服务端→客户端推送，无需客户端→服务端反向通信
2. **HTTP 兼容** - SSE 走标准 HTTP，无需额外协议升级（代理友好）
3. **自动重连** - 浏览器原生支持断线重连（代码未体现，但为 SSE 标准特性）

**局限**：
- ⚠️ 无法实现双向实时通信（如协作文档同步）
- 🔴 单连接限制（浏览器同域 SSE 连接数上限为 6，需补充连接池管理）

## 为什么 WPS 轮询周期固定为 5 秒？

**证据**：
- `frontend/src/services/wps-host.ts:10` - `WPS_POLLING_INTERVAL_MS = 5000` 导出常量
- `frontend/src/services/wps-host.ts:284-306` - `startPolling()` 使用 `setInterval(..., 5000)`

**决策原因**（需人工补充）：
1. **用户体验平衡** - 5 秒在"即时性"和"性能开销"间折中
2. **WPS JS API 限制** - 频繁调用可能导致宿主卡顿（需查阅 WPS 官方文档）
3. **避免重复计算** - 用户打字速度通常 <200 字/分钟，5 秒轮询足够捕获变化

**风险**：
- ⚠️ 固定周期无法自适应（如用户静止时应降低频率）
- 🔴 多文档场景需维护多个轮询定时器（代码未体现资源清理机制）

## 为什么前端 Store 采用"不可变更新"模式？

**证据**：
- `frontend/src/stores/conversation.ts:244-254` - `updateAssistantMessage()` 使用 `.map()` 返回新数组
- `frontend/src/stores/conversation.ts:77-85` - `cloneSources()` 深拷贝来源卡片

**决策原因**：
1. **可预测性** - 消除隐式副作用，便于调试（Vue DevTools 可追踪状态变更）
2. **时间旅行** - 配合 Pinia 支持撤销/重做（未来功能预留）
3. **并发安全** - SSE 流式更新时避免竞态条件（多 chunk 并发到达）

**代价**：
- ⚠️ 性能开销（大型消息列表频繁复制数组，需补充性能测试）
- 🔴 代码冗余（每次更新都需手动解构，可用 Immer 优化）

## 为什么 Query 改写服务使用 LLM 而非规则引擎？

**证据**：
- `backend/app/services/query_rewrite_service.py:91-128` - `rewrite()` 直接调用 LLMClient
- `backend/app/services/query_rewrite_service.py:20-88` - 四个 Prompt 模板覆盖不同输入场景

**决策原因**：
1. **语义理解** - 用户口语化提问需转换为学术检索术语（如"怎么写" → "方法论框架"）
2. **场景多样性** - 四种输入组合（仅 prompt/仅已写/已写+圈选/全选）需不同改写策略
3. **快速迭代** - 调整 Prompt 模板比维护规则树更快

**风险**：
- 🔴 成本高昂（每次问答额外一次 LLM 调用）
- ⚠️ 延迟增加（改写失败降级逻辑见 `backend/app/services/query_rewrite_service.py:125-128`）

## 为什么 API 客户端强制限制为本机回环地址？

**证据**：
- `frontend/src/services/api-client.ts:44` - `LOCAL_API_HOSTNAME_ALLOWLIST` 白名单
- `frontend/src/services/api-client.ts:76-109` - `normalizeConfiguredApiBaseUrl()` 强制校验
- `frontend/src/services/api-client.ts:66-69` - 注释说明："避免本地路径被发出机器"

**决策原因**：
1. **安全边界** - 导入接口会携带本地 PDF 绝对路径，误配外部 API 会导致路径泄露
2. **开发体验** - 默认 `http://127.0.0.1:8000` 保证开箱即用
3. **CORS 简化** - 后端 `app/main.py:21-27` 允许所有来源，仅依赖前端白名单收口

**风险**：
- 🔴 生产环境需手动配置（当前无环境变量覆盖机制）
- ⚠️ 无法部署到子路径（如 `https://example.com/assistant`）

## 为什么重试机制采用"指数退避"？

**证据**：
- `backend/app/core/retry.py:15-18` - `MAX_RETRIES=3, INITIAL_DELAY=1.0, BACKOFF_FACTOR=2.0`
- `backend/app/core/retry.py:78-104` - `async_retry()` 实现 1s → 2s → 4s 退避

**决策原因**：
1. **API 限流友好** - SiliconFlow/DeepSeek 等服务有速率限制，退避可避免触发封禁
2. **容错弹性** - 网络抖动时自动重试，减少用户侧错误暴露
3. **成本控制** - 最多重试 3 次，避免无限重试消耗配额

**局限**：
- ⚠️ 固定策略无法自适应（如检测到 429 状态码应延长退避时间）
- 🔴 同步重试会阻塞事件循环（`retry_sync()` 应标记为 deprecated）

## 为什么清洗模块使用"精确噪声列表"而非通用停用词？

**证据**：
- `backend/app/processing/cleaner.py:19-41` - `_NOISE_EXACT` 集合（"Expand table", "Tip", ".NET CLI"...）
- `backend/app/processing/cleaner.py:194-200` - `SKIP_TYPES` 明确列出噪声类型

**决策原因**：
1. **文档特化** - 当前主要处理计算机科学论文，需去除 MSDN/文档网站特有噪声
2. **精度优先** - 精确匹配避免误删正文（如"Note: "在实验结果中可能有意义）
3. **可扩展** - `frozenset` 保证 O(1) 查询，添加新噪声不影响性能

**风险**：
- 🔴 领域局限（迁移到人文社科时需重构噪声列表）
- ⚠️ 硬编码中文/英文混合（`_SPACED_ABBREVS` 需人工维护）

## 架构债务记录

**高优先级债务**：
1. 🔴 无中间缓存（INGEST_DESIGN.md 提到 JSON 缓存设计，但代码未实现）
2. 🔴 Qdrant 跨 Collection 检索性能未测试（需补充 1000+ 论文基准）
3. 🔴 SSE 无重连机制（`frontend/src/services/sse-client.ts` 无心跳检测）

**中优先级债务**：
1. ⚠️ Token 估算误差（`chunker.py` 的 `default_token_counter` 需校准）
2. ⚠️ 固定轮询周期无法自适应（`WPS_POLLING_INTERVAL_MS` 应动态调整）
3. ⚠️ 不可变更新性能开销（大型会话列表需虚拟滚动优化）

**低优先级债务**：
1. 噪声列表硬编码（`cleaner.py` 需提取为配置文件）
2. 同步重试函数应废弃（`retry_sync` 在异步项目中无意义）
3. CORS 全开放（生产环境需收紧 `allow_origins`）
