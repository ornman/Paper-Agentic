# 3-为什么这样 — 设计意图 vs 实际偏差

## 清晰的设计意图

### 1. 模型契约硬锁定（config.py）

config.py 用三层机制防止 Embedding/Rerank 模型漂移：

- **常量锁定**：`PINNED_EMBEDDING_MODEL`、`PINNED_EMBEDDING_DIMENSIONS`、`PINNED_RERANK_MODEL` 三个常量独立于配置字段。
- **启动校验**：`validate_pinned_model_contract()` 在 Settings 实例化时强制检查，任何漂移直接 `raise ValueError`。
- **注释说清原因**：`"一旦模型名或维度漂移，历史向量、检索结果和索引结构都会失去一致性。"`

注释里明确用了"硬约束"而非"推荐值"：

> `"这里不是'推荐值'，而是'系统硬约束'。"`

这是踩过坑后的设计。INGEST_DESIGN.md 记录了具体事故：`"测试时误将 EMBEDDING_DIMENSION 从 1536 改为 4096，后果：向量空间稀疏，检索质量下降"`。

### 2. 旧字段兼容层（config.py:300-341）

Task 1 只允许改 config.py，所以用 `@property` 做 7 个字段映射，把旧的通用名（`llm_api_key`、`embedding_api_key`、`rerank_api_key` 等）映射到新的服务商专属名（`deepseek_*`、`siliconflow_*`）。注释原文：

> `"Task 1 只允许改 config.py，因此这里保留旧属性，避免现有客户端与路由同时崩掉。"`

这是渐进重构策略：新代码用新名字，旧代码不被打断。

### 3. 服务商职责分离（config.py）

config.py 把 LLM 和 Embedding/Rerank 分到不同服务商，注释说清了原因：

> `"Embedding 与 Rerank 统一走 SiliconFlow，避免不同服务商混用导致契约漂移。"`
> `"主问答链路固定使用 DeepSeek 官方 API。"`

这是有意的架构决策：问答链路和向量链路解耦，避免一个服务商宕机连累另一个链路。

### 4. 渐进式架构演进（library/service.py）

`import_document`（只登记）和 `import_pdf`（登记+导入）是两个独立方法，注释解释了为什么不复用：

> `"为什么不直接复用 import_document 作为公开入口名：Task 3 的 import_document 语义是'只登记'。Task 4 新增的是'登记后继续导入'。拆成 import_pdf 可以避免把原有行为悄悄改坏，测试边界也更清晰。"`

`reindex_document` 也做了占位而非空实现，注释说：

> `"Task 3 还没有真正的索引模块，因此这里必须避免把健康的 completed 文档推进到 indexing，否则会制造一个当前任务无法自愈的卡死状态。"`

### 5. services_archive 的废弃原因

四个废弃文件各有明确的替代者：

| 废弃文件 | 功能 | 替代者 | 废弃原因 |
|----------|------|--------|----------|
| `qa_service.py` | 纯 LLM 问答（无 RAG） | `modules/qa/service.py` (QAService) | 旧版只支持 4 种 Prompt 模板，新架构集成了 RAG + Query 改写 |
| `qa_service_rag.py` | RAG 问答（Kimi 直连） | `modules/qa/service.py` (QAService) | 旧版直接 `KimiLLMClient()`，新架构通过依赖注入 |
| `retrieval_service.py` | 分布式多 Collection 检索 + RRF | `modules/retrieval/service.py` | 旧版设计了复杂的跨 Collection RRF，实际 MVP 只用单 Collection + Rerank |
| `ingest_service.py` | PDF 导入工作流 | `modules/ingestion/service.py` (IngestionService) | 旧版无状态机、无错误恢复，新版有状态迁移和断点续传 |

### 6. 旧检索服务的过度设计被砍掉

`services_archive/retrieval_service.py` 设计了完整的"分布式多 Collection + 跨 Collection 并行搜索 + RRF 融合 + Rerank"四步流程，包含详细的 Collection 命名规范（paper_{uuid}、video_{uuid}、note_{uuid}...）和大量"未来扩展"注释。

但新版 `modules/retrieval/service.py` 只保留了"向量检索 + Rerank"两步。原因显然是 MVP 阶段论文数量不多，跨 Collection RRF 融合的复杂度不值得。

## 意图与实际偏差

### 1. BM25 混合检索：写好了但没接入

**意图**：config.py 预留了 `bm25_index_path` 和 `retrieval_bm25_top_k` 配置，`repositories/bm25_repo.py` 实现了完整的 BM25 索引（add、query、delete、save、load），INGEST_DESIGN.md 明确写了"入库 → ChromaDB + BM25"。

**实际**：当前活跃的 `modules/retrieval/service.py` 完全没有引用 BM25。`app/repositories/bm25_repo.py` 在整个 `app/` 目录下零引用（只在 `tests/` 中被间接使用）。

**偏差原因**：从 ChromaDB 迁移到 Qdrant 后，BM25 的集成点丢失了。新架构只走了"向量检索 → Rerank"的路径。

### 2. ChromaDB → Qdrant 迁移：文档没跟上

**意图**：INGEST_DESIGN.md 详细描述了 ChromaDB 作为向量库的方案（三阶段导入、缓存目录 `data/chroma/`、ChromaDB+BM25 双索引）。

**实际**：代码已经全面使用 Qdrant（`QdrantStore`、`qdrant_path` 配置），但 INGEST_DESIGN.md 还在写"ChromaDB"，`.env.example` 还保留着 `CHROMA_PERSIST_DIR`，`README.md` 还写着"向量库: ChromaDB"，`health.py` 注释还写着"ChromaDB 本地，始终可用"。

**偏差原因**：迁移 Qdrant 是在 Task 4 或更后期做的，但设计文档没有同步更新。

### 3. 测试引用了不存在的模块

**实际**：`tests/indexing/test_index_modes.py` 引用了 `app.modules.indexing.bm25_repo` 和 `app.modules.indexing.chroma_repo`，但 `app/modules/indexing/` 目录根本不存在。

**偏差原因**：这些测试是为早期设计的索引模块写的（Task 3 时期），架构改为 Qdrant 后测试没有跟着改，目前处于无法运行的状态。

### 4. 旧 services_archive 没有清理

**意图**：废弃服务已经移到 `services_archive/` 目录，新架构通过 `modules/` 模块化。

**实际**：`services_archive/` 中的代码没有任何地方导入它（`grep` 确认零引用），但它仍然留在代码库中。虽然不造成运行时影响，但会误导阅读者以为这些文件还在使用。

### 5. "用户自选文献"参数到处透传但从未实现

`selected_papers` 和 `resource_types` 从路由层（query.py）→ QA 服务 → 检索服务一路透传，每层都有注释标注"暂未实现"或"预留接口"。新版检索服务的 retrieve 方法签名里接受这两个参数但内部完全忽略。

这是一个有意为之的 MVP 剪裁，但从代码阅读角度，大量的"未来扩展"注释块（包括 ASCII 前端 UI 示意图）增加了认知负担。

### 6. 断点续传：INGEST_DESIGN.md 设计了但新版实现方式不同

**设计文档**的三阶段断点续传：清洗 → JSON 缓存（status 字段）→ 向量化 → 入库，每个阶段输出独立 JSON 文件。

**实际实现**：`modules/ingestion/service.py` 通过 SQLite 状态机（pending → parsing → cleaning → indexing → completed/failed）实现断点续传，不走 JSON 文件。`library/service.py` 的 `resume_import` 方法只支持从 failed 状态恢复，而非设计文档中描述的任意阶段续传。

## 设计债务

### 高优先级

1. **测试与代码脱节**：`tests/indexing/test_index_modes.py` 引用 `app.modules.indexing.*` 模块不存在，整个测试文件无法运行。这可能是 CI 中的定时炸弹。

2. **文档与代码不同步**：INGEST_DESIGN.md、README.md、.env.example、health.py 中仍然引用 ChromaDB，实际已迁移到 Qdrant。新开发者会因文档误导。

3. **BM25 死代码**：`repositories/bm25_repo.py` 完整实现但零调用。config.py 中 `bm25_index_path` 和 `retrieval_bm25_top_k` 配置仍然存在。要么接入 BM25 混合检索，要么清理掉避免混淆。

### 中优先级

4. **services_archive 未清理**：4 个废弃文件 + `__init__.py`，零引用，建议移除或移到 docs 中作为参考。

5. **检索服务的降级策略硬编码**：`modules/qa/service.py` 中 RAG 检索失败时 `print` 一句就静默降级为纯问答，没有日志框架、没有指标、没有通知。用户不会知道检索失败了。

6. **旧检索服务的 RRF 算法丢失**：`services_archive/retrieval_service.py` 中精心实现的 `_reciprocal_rank_fusion` 算法在迁移时被丢弃了。当论文数量增长需要跨 Collection 检索时，需要重新实现。

### 低优先级

7. **4 个外部 API 凭空管理**：DeepSeek（问答）+ SiliconFlow（Embedding/Rerank）+ Kimi（VLM/Chat）+ MinerU（PDF 解析）。如果任何一个服务商改接口或宕机，系统静默失效，没有统一的健康检查或熔断机制。

8. **大量"未来扩展"注释块**：`services_archive/retrieval_service.py` 和 `modules/qa/service.py` 中包含多段带 ASCII 示意图的"未来扩展"注释（用户自选文献的 UI 想象图等），这些更适合放在产品文档而非代码中。

9. **配置的 AliasChoices 过度兼容**：每个配置字段都有 3-5 个别名（如 `deepseek_api_key` 同时响应 `llm_api_key`、`DEEPSEEK_API_KEY`、`LLM_API_KEY`），这是多次重构的痕迹，应在稳定后收敛到一个命名。
