# 2-做什么 — 行为图

## 核心流程

### 导入链路（PDF -> 知识库）

6 个阶段，由 `IngestionService.ingest_document()` 编排，状态机驱动：`pending -> parsing -> cleaning -> indexing -> completed`（失败则 `-> failed`）。

```
用户操作：POST /api/v1/library/import（上传文件）或 POST /api/v1/library/import-path（本地路径）
    |
    v
[LibraryService.import_pdf()]
  1. 校验文件路径（仅本地 .pdf，拒绝 URL/UNC）
  2. 创建 DocumentRecord（status=pending），写入 SQLite
  3. 调用 IngestionService.ingest_document(record)
    |
    v
[阶段1: MinerU 解析] status=parsing
  - MineruClient.run_pdf_task(file_path)
  - 内部流程：申请上传链接 -> PUT 上传文件 -> 轮询状态（time.sleep 阻塞）-> 下载 ZIP -> 解压 layout.json + 图片
  - 外部服务：MinerU API（自部署或云端）
  - 输出：{ pages: [{ page, blocks: [...] }] } + 本地图片文件（data/papers/{task_id}/images/）
  - 失败处理：IngestionError（超时/格式错误/任务失败）
    |
    v
[阶段2: 清洗] status=cleaning
  - clean_mineru_payload()（cleaning.py）
  - 从 MinerU payload 提取 raw blocks（_extract_raw_blocks）
  - 分离文本块和图片块
  - 文本过滤规则：
    a. _is_header_or_footer：跨页重复短行 -> 丢弃
    b. _is_page_number：纯页码格式 -> 丢弃
    c. _is_short_noise：<=2 字符 -> 丢弃
    d. 文本去重（seen_texts set）
  - 图片块：收集但暂不处理（由阶段3处理）
  - 输出：CleanedDocument（包含 cleaned_blocks 列表）
  - 如果清洗后 0 个有效块 -> IngestionError
    |
    v
[阶段3: VLM 图片描述]
  - 判断：如果 payload 中有 image/figure/picture 类型的块
  - 调用路径分歧：
    - 有图片：_describe_images() -> describe_chunks_async()
      - 并发调用 KimiVLMClient.describe_image()（semaphore 控制并发数）
      - 图片 base64 编码后发送给 Kimi VLM API
      - 结果缓存到 image_descriptions.json
    - 无图片：_blocks_to_chunks() 直接转换
  - 外部服务：Kimi VLM API（需要 API Key + User-Agent 伪装 claude-code）
  - 输出：list[Chunk]（图片块的 content 被替换为 VLM 描述）
  - 重试：retry_async(max_retries=3)
  - 注意：cleaning.py 内部也有独立的 _describe_images()，会在清洗阶段直接为图片块生成描述
    |
    v
[阶段4: 混合语义切分]
  - chunk_by_semantic_units()（chunker.py）
  - Chunk -> SemanticChunk -> TextChunk
  - 切分策略（基于 token 估算，中文*1.5 + 英文*0.75）：
    a. 相邻两块合计 <= pack_threshold（默认 32k）-> 合并打包
    b. 超过但单块不超 max_context -> 分开，不重叠
    c. 单块超过 max_context -> split_long_chunk()，按 target_size 切分 + overlap
  - 输出：list[TextChunk]
    |
    v
[阶段5: Embedding] status=indexing
  - EmbeddingClient.embed()（SiliconFlow 实现）
  - 批量向量化，内部按 batch_size 分批
  - 调用 SiliconFlow API（Qwen3-Embedding-8B）
  - 输出：list[list[float]]（默认 1536 维）
  - 重试：retry_async(max_retries=3)
    |
    v
[阶段6: Qdrant 存储]
  - QdrantStore.add_chunks()
  - 每篇论文创建独立 collection（paper_{id}）
  - 自动创建 collection（HNSW 索引，内存优化配置）
  - payload 保存：content, section, page, image_path, chunk_type, paper, original_id
  - 输出：collection 中新增的向量点
  - 最终：record.transition("completed")，写入 SQLite
```

**SQLite 记录的完整状态机**：`pending -> parsing -> cleaning -> indexing -> completed`，任意阶段可 -> `failed`。失败后可通过 `POST /api/v1/library/resume/{document_id}` 恢复。

### 问答链路（提问 -> SSE 流式回答）

```
用户操作：前端 SSE 请求 POST /api/v1/query/ask
  请求体：AskRequest { session_id, query, context?: { written_content, selected_text, prompt }, resource_types?, selected_papers? }
    |
    v
[query.py: ask()]
  - 打印调试信息（context 各字段预览）
  - 创建 event_generator() 异步生成器
  - 返回 StreamingResponse(media_type="text/event-stream")
    |
    v
[QAService.ask_stream_with_rag()]
    |
    +--[1. Query 改写]
    |   QueryRewriteService.rewrite(query)
    |   - 4 种场景模板（仅 prompt / 仅已写 / 已写+圈选 / 全部）
    |   - 调用 LLMClient（OpenAI 兼容协议，配置的 llm_base_url）生成 1-3 个检索查询
    |   - 发送 SSE event: rewrite { original, rewritten }
    |   - 失败降级：直接用原始 query
    |
    +--[2. RAG 检索]
    |   RetrievalService.retrieve(rewritten_queries[0], top_k=10)
    |   |
    |   +--[2a. 向量检索]
    |   |   EmbeddingClient.embed_single(query) -> query_vector
    |   |     外部服务：SiliconFlow Embedding API
    |   |   QdrantStore.search_all(query_vector, limit=top_k*2)
    |   |     遍历所有 paper_* collection，每个 collection 搜索 limit 条
    |   |     合并后按 score 排序，取 top
    |   |
    |   +--[2b. Rerank]
    |       如果向量结果 > top_k：
    |         RerankClient.rerank(query, documents, top_k)
    |           外部服务：SiliconFlow Rerank API（Qwen3-Reranker-8B）
    |         按 rerank_score 重新排序
    |   |
    |   发送 SSE event: sources { sources: [...] }
    |   失败降级：sources=[]，回退到纯问答
    |
    +--[3. 构建消息]
    |   _build_messages(session_id, query, sources)
    |   - 系统提示词："你是一个专业的学术写作助手"
    |   - 加载历史消息（最近 20 条 / 10 轮）from SQLite
    |   - 如果有 sources：拼接 RAG 上下文到用户消息（含引用格式 [来源:论文名 页码X]）
    |   - 如果无 sources：直接用 query
    |
    +--[4. 流式生成]
    |   KimiLLMClient.chat_stream(messages)
    |   - 注意：KimiLLMClient 的 chat_stream 是"伪流式"
    |     先调用 chat() 获取完整响应，再按 50 字符切块 yield
    |   - 外部服务：Kimi API（需要 API Key + User-Agent 伪装 claude-code）
    |   - 重试：retry_async 包裹
    |   - 每个 chunk 发送 SSE event: chunk { content, index }
    |
    +--[5. 保存消息]
    |   add_message(session_id, "user", query) -> SQLite
    |   add_message(session_id, "assistant", full_response, sources=sources) -> SQLite
    |
    +--[6. 完成]
        SSE event: done { total_tokens }

    异常路径：
    - LLM 生成失败 -> SSE event: error { code: 9001, message }
    - 整个 generator 异常 -> SSE event: error { code: 9001, message }
```

**前端 SSE 客户端处理**（sse-client.ts）：
1. `postAskInspirationStream()` 发起 POST 请求，Accept: text/event-stream
2. `readSseStream()` 逐帧解析 SSE（按 `\n\n` 分割）
3. 事件分发：`chunk` -> onChunk（追加文本），`sources` -> onSources（展示引用卡片），`done` -> onDone，`error` -> onErrorEvent
4. 安全限制：只允许连接本地回环地址（127.0.0.1, localhost, ::1）
5. 来源卡片映射：后端 { document, content, page } -> 前端 SourceCard { title, snippet, page }

## 未接通的管道

### 1. BM25 关键词检索（写好了但没接入检索链路）

- 文件：`repositories/bm25_repo.py`（完整实现：add, add_batch, query, delete, save/load 持久化）
- 依赖：rank-bm25, jieba（已在依赖中）
- 配置项已存在：`bm25_index_path`, `retrieval_bm25_top_k`
- **但没有被任何服务调用**：
  - 导入链路中 `IngestionService` 不会向 BM25Repo 写入
  - 检索链路中 `RetrievalService.retrieve()` 只走向量 + Rerank，没有 BM25 召回
  - 原始设计是"向量 + BM25 混合检索 + RRF 融合 + Rerank"，当前只实现了"向量 + Rerank"

### 2. Query 改写的 context 参数（接了线但没传信号）

- `AskRequest` 接收 `context: QueryContext`（含 written_content, selected_text, prompt）
- 路由层正确解析并打印了 context 各字段
- 但 `QAService.ask_stream_with_rag()` 调用 `self.query_rewrite_service.rewrite(query)` 时**没有传递 context**
- `QueryRewriteService.rewrite()` 的第二个参数 `context: Optional[QueryContext] = None` 从未被使用
- 4 种场景改写模板（仅 prompt / 仅已写 / 已写+圈选 / 全部）全部闲置，每次都走"仅 prompt"场景
- 前端已经通过 WPS JS SDK 获取了 written_content 和 selected_text 并传给后端，但后端没有利用

### 3. resource_types 和 selected_papers 过滤（预留接口，未实现）

- `AskRequest` 和 `RetrieveRequest` 都定义了 `resource_types` 和 `selected_papers`
- `RetrievalService.retrieve()` 接受这两个参数，但内部注释标注"暂未实现"
- `QdrantStore.search_all()` 有 `paper_filter` 参数但上层没传
- 代码中被注释掉的逻辑：`if selected_papers: collections = [f"paper_{pid}" for pid in selected_papers]`

### 4. LangChain（依赖存在但未被使用）

- 项目技术栈声明了 LangChain
- 但整个代码库中没有任何 LangChain import 或使用
- QA 服务直接用自建的 `KimiLLMClient` + 手动拼接消息
- 不影响功能，但增加了依赖体积

### 5. KimiLLMClient 的 chat_stream 是伪流式

- `KimiLLMClient.chat_stream()` 内部先调用 `chat()` 获取完整响应，再按 50 字符切块 yield
- 不是真正的 SSE 流式输出（Kimi API 可能支持流式，但当前实现没有用）
- 另有一个独立的 `LLMClient`（llm_client.py）使用 AsyncOpenAI 真正流式，但 `QueryRewriteService` 用的是这个，`QAService` 用的是 `KimiLLMClient`
- `config.py` 的连通性测试用的也是 `LLMClient`，不是 `KimiLLMClient`

### 6. 双重 VLM 图片描述路径

- `IngestionService._describe_images()`：阶段3 中调用 `describe_chunks_async()` 处理图片
- `cleaning.py._describe_images()`：清洗阶段也有独立的图片描述逻辑
- 两条路径都会创建 `KimiVLMClient` 实例并调用 VLM
- 可能导致同一批图片被描述两次，浪费 API 调用
