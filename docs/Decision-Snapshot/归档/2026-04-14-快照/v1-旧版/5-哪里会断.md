## 🔴 P0: 导入链路同步阻塞整个 FastAPI 事件循环

**代码路径**：`service.py:60` → `mineru_client.py:46` → `mineru_client.py:134-193`

```python
# service.py:60 — async 函数内部调用同步方法
mineru_payload = self.mineru_client.run_pdf_task(current_record.file_path)

# mineru_client.py:46 — run_pdf_task 是同步函数
def run_pdf_task(self, file_path: str) -> dict[str, Any]:

# mineru_client.py:134-193 — poll_task 内部同步轮询
def poll_task(self, task_id: str) -> MineruTaskState:
    while True:
        elapsed = self.monotonic_fn() - started_at
        if elapsed > self.timeout:  # 默认 300 秒
            raise IngestionError(...)
        ...
        self.sleep_fn(self.poll_interval)  # time.sleep(5) — 同步阻塞
```

**为什么危险**：
- `IngestionService.ingest_document()` 是 `async` 函数（`service.py:50`），但 `MineruClient` 全部是同步实现
- `poll_task` 内部 `time.sleep(5)` 在 async 事件循环中 = 阻塞整个线程
- `httpx.Client`（同步）不是 `httpx.AsyncClient`，所有网络 IO 也是同步的
- 大 PDF 解析可能 5-10 分钟（`mineru_timeout=300`），这期间整个 FastAPI 服务无法响应任何请求

**依赖链**：用户上传 PDF → `ingest_document()` → `run_pdf_task()` → `poll_task()` → `time.sleep()` 循环 → 事件循环冻结 → SSE 流断开 → 前端无限 loading → 所有其他用户的请求也排队等待

**断点触发**：任何用户导入超过几页的 PDF。不是极端情况，是正常使用就会触发。

## 🔴 P0: SSE 流无超时保护，前端永久挂起

**代码路径**：`sse-client.ts:371-394`

```typescript
// sse-client.ts:375 — fetch 没有 AbortController，没有 timeout
const response = await fetch(buildApiUrl(ASK_ENDPOINT), {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    Accept: 'text/event-stream',
  },
  body: JSON.stringify(payload),
})
// ← 没有 signal: abortController.signal
// ← 没有外层 Promise.race([fetch, timeout])

// sse-client.ts:349-360 — ReadableStream 读取也无超时
while (!terminalEventReached) {
  const { done, value } = await reader.read()  // ← 无限等待
  if (done) { ... break }
  buffer += decoder.decode(value, { stream: true })
  await drainBuffer(false)
}
```

**为什么危险**：
- 后端因 MinerU 阻塞挂起（P0 上面的断点）→ 前端 `fetch` 永远不返回
- 即使后端恢复了，TCP 连接可能已经超时断开，但 `reader.read()` 不会触发 `done`
- 用户只能刷新页面，无法知道是"还在处理"还是"已经挂了"
- 没有 AbortController 意味着用户切换页面时请求不会被取消

**依赖链**：后端卡住（MinerU/DeepSeek/Kimi 任一外部服务超时）→ SSE 流无数据 → `reader.read()` 永远挂起 → UI 无限 loading → 用户无法恢复

**断点触发**：任何外部 API 超时或网络抖动。与 P0 同步阻塞叠加时 100% 触发。

## 🔴 P0: CORS `allow_origins=["*"]` + `allow_credentials=True` — 安全悖论

**代码位置**：`main.py:21-27`

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,   # ← 与通配符冲突
    allow_methods=["*"],
    allow_headers=["*"],
)
```

**为什么危险**：
- 浏览器规范：`allow_origins=["*"]` + `allow_credentials=True` 时，浏览器实际上**不会发送 credentials**。所以 `allow_credentials=True` 是无效的
- 真正的危险：如果部署到非 localhost 环境，任何网站可以无凭证调用 API
- `config.py:256-259` 已经有 `is_development` / `is_production` 属性，但这里完全没用
- 没有环境判断逻辑，意味着部署到生产时这个配置会原封不动地带过去

**依赖链**：论文助手 API 暴露在非回环地址 → 恶意网页 `<script>fetch('http://...')` → 窃取论文内容/操作 API

**断点触发**：`app_host` 改为 `0.0.0.0` 或部署到服务器时。本地单机使用时影响较小，但 MVP 之后必然要面对。

**断点触发**：MinerU 服务抖动时。外部 API 不稳定是常态。

## 🔴 P1: Embedding 客户端每批创建新的 httpx.AsyncClient

**代码位置**：`embedding_client.py:87`

```python
async def _embed_batch_raw(self, texts: list[str]) -> list[list[float]]:
    async with httpx.AsyncClient(timeout=self._timeout) as client:  # ← 每批新建+销毁
        response = await client.post(...)
```

**为什么危险**：
- `embed()` 方法按 `batch_size=32` 分批处理（行 47-51）
- 每批都创建一个新的 `AsyncClient` 实例，意味着每批都新建 TCP 连接 + TLS 握手
- 100 个 chunks = 4 批 = 4 次 TCP 连接建立/销毁
- 更严重的是：如果 `self._timeout` 是 int（`config.py:96` `siliconflow_timeout: int = 120`），httpx 会用默认的连接超时，可能在并发场景下出现连接池耗尽

**依赖链**：大批量 chunks → 频繁创建/销毁连接 → 连接泄漏或耗尽 → Embedding 阶段超时 → 导入失败

**断点触发**：论文 chunk 数量较多时（>64 个 chunk）。普通论文可能触发。

## 断点优先级总结

| 等级 | 断点 | 触发概率 | 影响范围 |
|------|------|----------|----------|
| 🔴 P0 | MinerU 同步阻塞事件循环 | 高（每次导入必触发） | 全服务不可用 |
| 🔴 P0 | SSE 无超时保护 | 高（与 P0 叠加） | 前端永久挂起 |
| 🔴 P0 | CORS 通配符 | 低（本地使用时） | API 被恶意调用 |
| 🔴 P1 | Kimi User-Agent 伪装 | 中（迟早发生） | 图片描述功能归零 |
| 🔴 P1 | API Key 启动不校验 | 高（首次部署） | 所有外部调用失败 |
| 🔴 P1 | MinerU 无重试 | 中（API 不稳定时） | 导入链路中断 |
| 🔴 P1 | Embedding 每批新建连接 | 中（大文档时） | 连接耗尽/超时 |
| 🔴 P1 | 导入中断无清理 | 中（重试场景） | 数据混杂 |
| ⚠️ P2 | 异常信息泄露 | 低 | 信息安全 |
| ⚠️ P2 | Qdrant 每 Paper 一个 Collection | 低（MVP 阶段） | 规模化瓶颈 |
| ⚠️ P2 | SSE 缓冲区无上限 | 低 | 前端内存溢出 |
| ⚠️ P2 | VLM 并发未实际限制 | 中（多图论文） | 图片描述质量归零 |
| ⚠️ P2 | Config 缓存不刷新 | 低 | 调试困惑 |
