# 1-有什么 — 结构图

## 一句话定性

前后端分离的 RAG 学术写作助手 MVP，后端 71 个 Python 文件（FastAPI），前端 26 个源文件（Vue 3 + TS），本地部署，WPS 侧边栏插件形态。当前状态：核心链路（PDF 导入 -> 向量化 -> 检索 -> 流式问答）已打通，存在少量废弃代码和未实现接口。

## 值得注意的

1. **BM25 建了但没用。** `bm25_repo.py` 完整实现了增删查，但 `RetrievalService` 只走了向量检索 + Rerank，BM25 完全没接入检索链路。config 里定义了 `retrieval_bm25_top_k=30` 和 `retrieval_rrf_k=60`，但没有任何代码使用这些参数。

2. **`services_archive/` 有 5 个废弃文件没清。** 被 `modules/` 新架构替代的旧服务（embedding/ingest/qa/qa_rag/retrieval），仍留在仓库中。

3. **Kimi LLM 的 `chat_stream` 是假流式。** `kimi_client.py:218-224` 实际是一次性拿完整响应，然后按 50 字符分块 yield，不是真正的 token 级流式。

4. **LLM 客户端存在双轨。** `llm_client.py`（DeepSeek/OpenAI）被 `QueryRewriteService` 使用，`KimiLLMClient` 被 `QAService` 使用。问答链路和改写链路用的是不同 LLM 后端。

5. **`workflows/` 目录是空壳。** 只有一个空的 `__init__.py`，没有任何实际内容。

6. **前端 SSE 客户端有严格的安全边界。** `sse-client.py:36-37` 只允许 `127.0.0.1`/`localhost`/`[::1]` 的 API 地址，非本机地址直接回退默认值。API 基地址白名单校验约 60 行代码。

7. **Embedding/Rerank 模型是硬锁的。** `config.py:10-13` 定义了 `PINNED_*` 常量，`model_validator`（`config.py:287-298`）在启动时强制校验，防止模型名或维度漂移导致历史向量失效。

8. **文档导入有断点续传。** `LibraryService.resume_import()`（`library/service.py:92-128`）只允许从 `failed` 状态恢复，重置到 `pending` 后重新走完整链路。不是真正的阶段性断点续传。

9. **docker-compose.yml 的环境变量名已过时。** 仍使用旧的 `LLM_API_KEY`/`LLM_BASE_URL`/`LLM_MODEL`/`EMBEDDING_API_KEY`/`RERANK_API_KEY`，但 `config.py` 已迁移到 `deepseek_*`/`siliconflow_*` 命名。虽然 `config.py:300-341` 有兼容层 property 做映射，但 docker-compose 和配置层之间的命名对齐是隐式的。
