## 典型用户场景

基于 Q1-Q5 分析，论文助手是一个面向学术写作场景的 RAG 系统，以下是推断的典型场景（需人工验证）：

### 场景1：首次使用 - 文献库初始化
- **用户行为**：安装系统 → 配置 API Key → 导入第一篇 PDF 论文
- **预期体验**：导入进度可视化 → 等待清洗/切分/向量化完成 → 看到成功提示
- **未知点**：
  - 用户从哪里获取论文？本地文件夹 / 在线数据库 / 文献管理软件（Zotero/EndNote）？
  - 批量导入还是单篇导入？单次导入多少篇符合预期？
  - 导入失败时的错误提示是否友好？

### 场景2：写作中遇到问题 - 灵感搜索
- **用户行为**：在 WPS 中写论文 → 遇到问题 → 圈选关键词 → 打开插件 → 输入问题
- **预期体验**：系统自动结合已写内容 + 圈选文本 → 返回带引用的答案 → 复制到文档
- **未知点**：
  - 用户通常在什么阶段使用？选题/文献综述/方法论/结论？
  - 最常问的问题类型是什么？概念解释/方法对比/数据支持/相关研究？
  - 用户如何验证答案的准确性？是否需要跳转到原文？

### 场景3：多篇文献对比 - 深度分析
- **用户行为**：选中 3-5 篇相关论文 → 提问"这些论文对 XX 问题有何不同观点？"
- **预期体验**：系统只从选中文献中检索 → 返回对比表格或分点论述
- **未知点**：
  - 用户如何选择文献？是否需要预览摘要？
  - 对比结果如何呈现？表格 / 时间线 / 观点聚类？
  - 是否支持导出对比结果到文档？

### 场景4：会话中断 - 恢复历史
- **用户行为**：关闭 WPS → 重新打开 → 查看上次的问答记录
- **预期体验**：历史记录自动加载 → 可继续追问或查看引用
- **未知点**：
  - 用户期望保留多长的历史？1天/1周/1个月？
  - 是否需要会话命名/分类/搜索？
  - 是否需要导出会话记录？

### 核心流程：学术写作辅助

```
┌─────────────────────────────────────────────────────────────┐
│  1. 准备阶段                                                 │
│     ┌───────────────────────────────────────────────────┐   │
│     │ • 下载/收集论文 PDF                                │   │
│     │ • 配置系统（API Key、Qdrant 路径）                 │   │
│     │ • 导入文献库（单篇或批量）                         │   │
│     └───────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────┐
│  2. 写作阶段                                                 │
│     ┌───────────────────────────────────────────────────┐   │
│     │ • 在 WPS 中撰写论文                                │   │
│     │ • 遇到问题时打开插件                               │   │
│     │ • 圈选关键文本（可选）                             │   │
│     │ • 输入问题（如"这段引用的原始论证是什么？"）       │   │
│     └───────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────┐
│  3. 检索与生成阶段                                           │
│     ┌───────────────────────────────────────────────────┐   │
│     │ • 系统分析用户输入（已写内容 + 圈选 + prompt）    │   │
│     │ • Query 改写（生成 1-3 个检索查询）               │   │
│     │ • 混合检索（向量 + BM25 + Rerank）                │   │
│     │ • LLM 生成答案（流式输出）                         │   │
│     │ • 返回引用来源（文档名 + 页码 + 摘要）             │   │
│     └───────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────┐
│  4. 验证与应用阶段                                           │
│     ┌───────────────────────────────────────────────────┐   │
│     │ • 查看答案 + 引用来源                              │   │
│     │ • 点击来源卡片跳转到原文（Qdrant payload 信息）    │   │
│     │ • 复制答案到文档                                   │   │
│     │ • 继续追问或开始新话题                             │   │
│     └───────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

## 痛点与需求（推断 + 待验证）

| 痛点 | 对应方案（代码证据） | 用户真实需求？ |
|------|---------------------|---------------|
| **文献太多，找不到相关内容** | 混合检索（向量 + BM25 + Rerank） | 检索准确率是否满足需求？ |
| **答案没有引用，无法验证** | 返回来源卡片（文档名 + 页码 + 摘要） | 是否需要直接跳转到原文？ |
| **问题太模糊，检索不准** | Query 改写（4 种场景权重） | 用户是否感知到改写效果？ |
| **答案太长，难以消化** | 流式输出（SSE） + 引用分离 | 是否需要摘要/要点模式？ |
| **需要对比多篇文献** | 预留接口（`selected_papers`） | 用户是否真的需要这个功能？ |
| **导入 PDF 太慢** | 六阶段流程 + 断点续传 | 导入 100 篇论文需要多久？ |
| **网络不稳定，答案中断** | SSE 流式（无重连） | 是否需要自动重连？ |

## 需人工补充的关键信息

### 1. 用户画像
- **目标用户**：研究生/博士生/青年教师？
- **技术熟练度**：是否需要 API Key 配置指引？
- **使用频率**：每天/每周/每月使用几次？
- **文献库规模**：平均管理多少篇论文？（10/50/200/1000+）

### 2. 使用场景细节
- **写作阶段**：选题 → 文献综述 → 方法设计 → 实验分析 → 结论，哪个阶段最常用？
- **问题类型**：
  - 概念解释（"什么是注意力机制？"）
  - 方法对比（"BERT 和 GPT 的区别？"）
  - 数据支持（"有没有论文支持这个假设？"）
  - 引用推荐（"相关研究有哪些？"）
  - ...？
- **输入方式**：
  - 仅文字提问占比？
  - 圈选文本 + 提问占比？
  - 结合已写内容占比？

### 3. 成功标准
- **检索准确率**：用户认为多少比例的答案是准确的？（80%/90%/95%？）
- **响应速度**：多少秒内返回答案可接受？（<3s/<5s/<10s？）
- **导入速度**：导入一篇论文（平均 20 页）需要多久？
- **留存率**：用户是否会持续使用？（一天/一周/一个月后？）

### 4. 竞品对比
- **用户当前解决方案**：
  - 手动搜索 + 阅读（传统方式）
  - 其他 AI 工具（ChatGPT/Claude/Kimi）
  - 其他科研工具（Semantic Scholar/Connected Papers）
- **切换成本**：从旧方案迁移到论文助手，用户最大的顾虑是什么？

### 5. 未覆盖场景
- **协作需求**：多人共享文献库？会话同步？
- **移动端**：是否需要手机/平板访问？
- **离线模式**：无网络时是否可用？（至少本地检索）
- **多语言**：是否需要支持英文/其他语言论文？

## 数据收集建议

### 定性研究
1. **用户访谈**（5-10 人）：
   - 深度了解写作流程
   - 观察实际使用过程
   - 收集痛点与期望

2. **可用性测试**：
   - 首次导入文献任务
   - 典型问答任务
   - 错误恢复任务

### 定量研究
1. **埋点统计**：
   - DAU/WAU/MAU（日/周/月活）
   - 功能使用频率（导入/问答/会话管理）
   - 检索成功率（SSE 完成率）
   - 平均会话时长

2. **问卷调研**：
   - 用户画像（年级/专业/研究方向）
   - 满意度评分（1-5 分）
   - NPS（净推荐值）

## 与 Q1-Q5 的关联

- **Q1（有什么）**：系统提供了 WPS 插件 + 文献库 + RAG 问答
- **Q2（做什么）**：导入文献 → 检索 → 生成答案 → 返回引用
- **Q3（为什么）**：采用分布式架构、混合检索、Query 改写等技术决策
- **Q4（答应了什么）**：配置校验、状态机、API 契约等硬约束
- **Q5（哪里会断）**：外部 API 故障、SSE 中断、状态无持久化等风险
- **Q7（经历了什么）**：**用户如何实际使用？是否符合预期？** ← 需人工补充

---

**结论**：代码只能告诉我们"系统做了什么"，无法推导"用户经历了什么"。Q7 必须通过用户研究（访谈/观察/问卷）来补充。建议优先收集用户画像和典型场景，再迭代优化产品。
