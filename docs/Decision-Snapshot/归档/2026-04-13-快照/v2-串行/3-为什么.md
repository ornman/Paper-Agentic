# 3-为什么这样 — 设计意图 vs 实际偏差

## 清晰的设计意图

1. **模型契约锁定** — `config.py` 用 `PINNED_*` 常量 + `model_validator` 防止 Embedding 维度和模型漂移。注释明确："一旦模型名或维度漂移，历史向量、检索结果和索引结构都会失去一致性。" 这是经过踩坑的设计。

2. **旧字段兼容层** — `config.py:300-341` 用 `@property` 把旧的 `llm_api_key` 映射到新的 `deepseek_api_key`，避免客户端同时崩掉。注释说"Task 1 只允许改 config.py"。说明有渐进重构的策略。

3. **SSE 协议设计** — 前端 `sse-client.ts` 的注释很详细（为什么要单独定义 payload 类型、为什么暂存 sources），说明经历过字段漂移的坑。

4. **API 基地址安全边界** — `sse-client.ts:36-98` 大段代码专门处理 URL 规范化和本机回环地址白名单，防止把论文正文误发到外部主机。

## 意图与实际偏差

| 设计意图 | 实际状态 | 偏差原因 |
|----------|----------|----------|
| LangChain 做 QA 编排 | `pyproject.toml` 有依赖，代码里没 import | 可能中途换了 Kimi 直连，忘了清理依赖 |
| BM25 + 向量混合检索 | `bm25_repo.py` 写好了，`RetrievalService` 没调用 | 预留了但没接入 |
| 用户自选文献检索 | `selected_papers` 参数到处透传但注释标注"暂未实现" | MVP 砍掉了 |
| `services_archive/` 废弃 | 4 个旧文件还在代码库 | 没清理 |

## 设计债务

- **4 个外部 API 凭空管理**（DeepSeek + SiliconFlow + Kimi + MinerU）— 如果任何一个服务商改接口，系统静默失效
- **SQLite + Qdrant + BM25 文件 三套存储** — 没有统一的事务或一致性保证
