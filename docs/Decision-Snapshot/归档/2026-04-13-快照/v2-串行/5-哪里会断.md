# 5-哪里会断 — 风险全貌

**最高优先级文件，按风险等级排序。**

## 🔴 P0: MinerU 同步阻塞整个服务

**位置**：`backend/app/clients/mineru_client.py:29-52` — `MinerUTask.poll()`

```python
while True:
    if time.time() - start > timeout:
        raise TimeoutError(...)
    resp = httpx.get(self.status_url, timeout=30)
    ...
    time.sleep(interval)  # ← 同步阻塞
```

**为什么危险**：
- `IngestionService.ingest_document()` 是 `async` 函数，但调用的 `mineru_client.run_pdf_task()` → `MinerUTask.poll()` 是**同步的**
- `time.sleep()` 在 async 事件循环里 = 卡死整个服务
- 大 PDF 解析 5-10 分钟，这期间所有 HTTP 请求排队等待
- `httpx.get()` 也是同步调用（不是 `httpx.AsyncClient`）

**断点**：任何用户导入大 PDF → 全服务不可用 → SSE 连接超时 → 前端转圈无响应

## 🔴 P0: SSE 流无超时保护

**位置**：`frontend/src/services/sse-client.ts:374-394`

```typescript
const response = await fetch(buildApiUrl(ASK_ENDPOINT), {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', Accept: 'text/event-stream' },
  body: JSON.stringify(payload),
})
// ← 没有 AbortController, 没有 timeout
// ← ReadableStream reader 也没有超时
```

**为什么危险**：
- 后端挂起（如 MinerU 阻塞）→ 前端永远转圈
- 没有 `AbortController` 或 `setTimeout` 做兜底
- 网络断开时 `ReadableStream` 可能永远不触发 `done`

**断点**：后端卡住 → 前端无限 loading → 用户只能刷新页面

## 🔴 P1: Kimi API 伪装 User-Agent

**位置**：`backend/app/core/config.py:146` 注释 + `.env.example:14`

```
# 关键：必须使用 User-Agent: claude-code 伪装
```

**为什么危险**：
- 依赖 Kimi 的 `kimi-for-coding` 端点的反爬漏洞
- Kimi 升级 WAF 校验 → 所有 VLM 图片描述 + Chat 功能静默失效

**断点**：Kimi 反爬升级 → 导入带图片的论文必失败

## ⚠️ P2: Qdrant 每 Paper 一个 Collection

**位置**：`backend/app/stores/qdrant_store.py`

每篇论文创建独立 Collection。2000 篇论文 = 2000 个 Collection + 2000 个 HNSW 索引。内存非线性增长，没有冷热分层。

**断点**：论文量达到一定规模 → Qdrant 内存溢出 → 检索不可用
