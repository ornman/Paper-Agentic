# 2-做什么 — 行为全貌

**一句话**：PDF 导入 → 清洗切分 → 向量化存储 → 混合检索 → 流式问答。WPS 插件提供论文上下文。

## 核心数据流

```
用户上传 PDF
    ↓
[导入链路] IngestionService.ingest_document()
    ├─ 阶段1: MinerU 远程解析 PDF → JSON + 图片
    ├─ 阶段2: clean_mineru_payload() 清洗（去噪、归一化）
    ├─ 阶段3: Kimi VLM 图片描述（并行，最多 10 并发）
    ├─ 阶段4: chunk_by_semantic_units() 混合语义切分
    ├─ 阶段5: SiliconFlow Embedding 批量向量化
    └─ 阶段6: Qdrant 存储（每篇论文一个 Collection）
    ↓
[QA 链路] QAService.ask_stream_with_rag()
    ├─ QueryRewriteService 改写查询
    ├─ RetrievalService 向量检索 → Rerank
    ├─ 拼接 RAG 上下文 + 历史消息（最近 10 轮）
    └─ KimiLLMClient 流式生成 → SSE 推送到前端
    ↓
[前端] Vue 3 + Pinia 状态机
    idle → requesting → streaming → done/error
    SSE 解析: chunk/sources/done/error 四类事件
```

## API 端点（5 组）

| 路由 | 功能 |
|------|------|
| `/api/v1/health` | 健康检查 |
| `/api/v1/session/*` | 会话管理 |
| `/api/v1/query/ask` | SSE 流式问答（核心） |
| `/api/v1/config/*` | 配置查询 |
| `/api/v1/library/*` | 论文库管理（导入/列表/删除） |

## WPS 插件做的事

- 侧边栏嵌入 WPS Office
- 轮询获取当前编辑的论文正文 → 作为 RAG context 传给后端
- 支持圈选文本 → 作为 `selected_text` 传给后端

## 值得注意

- **导入链路是同步阻塞的** — `MinerUTask.poll()` 在 async 函数里用 `time.sleep()` 轮询，大 PDF 会卡死整个服务
- **BM25 在依赖里但检索链路没接入** — `RetrievalService` 只走向量 + Rerank，`bm25_repo.py` 写好了但没用上
- **LangChain 在依赖里但代码里没 import** — QA 服务直接用 `KimiLLMClient`，没有走 LangChain 的 chain/agent
- **`resource_types` 和 `selected_papers` 是预留接口** — `RetrievalService.retrieve()` 接受参数但注释标注"暂未实现"
