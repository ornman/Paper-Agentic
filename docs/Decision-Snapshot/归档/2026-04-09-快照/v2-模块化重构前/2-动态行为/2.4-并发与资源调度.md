## 无限并发风险清单

### 风险点 1：多 Collection 并行检索

**位置**: `retrieval_service.py:152`

```python
results = await asyncio.gather(*tasks, return_exceptions=True)
```

**问题**: 每个 Collection 创建一个搜索 Task，当论文库有 N 篇论文时，会同时发起 N 个 Qdrant 搜索请求。

| 论文数量 | 并发连接数 | Qdrant 压力 | 风险等级 |
|---------|-----------|------------|---------|
| 10 | 10 | 低 | 低 |
| 100 | 100 | 中 | 中 |
| 1000 | 1000+ | **高** | **Critical** |
| 5000 | 5000+ | **极高** | **Critical** |

**影响**: Qdrant 端文件句柄耗尽，或网络层被视为异常流量。

**缓解方案**: 引入 `asyncio.Semaphore` 限制最大并发数（如 50）。

### 详细分析：MinerU 同步阻塞

MinerU 客户端是**完全同步实现**（`mineru_client.py`）：
- `submit_pdf()`: 同步 HTTP POST
- `poll()`: 同步 `time.sleep` 循环
- `parse_pdf()`: 同步下载 + 解压

调用链：`ingest_service.ingest_pdf()` → `MinerUClient.parse_pdf()` → 阻塞事件循环

当 PDF 解析耗时 5-10 分钟时，**整个 FastAPI 事件循环被卡死**，所有其他请求无法响应。

一、多 Collection 并行检索的扇出爆炸问题
1. 当前模型的问题抽象
现状：对于每个用户的每次检索请求，系统会创建 N 个并发异步任务，每个任务对应一个 PDF Collection，向 Qdrant 发起独立的搜索请求。
复杂度：O(N) 网络连接、O(N) Qdrant 内部索引扫描开销。
临界点：当 N 从 100 增长到 10000 时，系统面临：
客户端侧：文件描述符耗尽、端口耗尽、内存中协程爆炸。
Qdrant 侧：并发请求数等于论文数，Qdrant 的 CPU 被大量上下文切换和内耗淹没，实际有效吞吐量反而下降。
网络层：瞬时流量洪峰可能触发防火墙 SYN Flood 防护或限流策略。
2. 根本矛盾
检索场景需要全局排序，而您选择了物理分片（每 PDF 一个 Collection）。这导致全局排序必须在应用层通过扇出 + 归并实现，违背了向量数据库的“单 Collection 内高效 ANN 搜索 + 过滤”的设计初衷。
3. 架构修正路线
方案一（首选）：彻底重构为单 Collection + Payload 过滤
这是唯一符合 Qdrant 最佳实践的方案。
迁移步骤：

创建单一 papers Collection。
插入向量时，在 Payload 中携带 document_id 字段。
检索时，使用 Qdrant 的 SearchRequest 配合 filter 实现跨文档检索，无需指定文档 ID 时即为全库搜索。
如果需要限制在特定文档集合内，使用 filter 进行筛选。
性能对比：

原方案：10,000 个 Collection → 10,000 次网络 RPC → 应用层做 10,000 路归并排序。

新方案：1 次网络 RPC，Qdrant 内部利用 HNSW 图在单次遍历中完成全局 Top-K 检索。

预期提升：延迟降低 100~1000 倍，资源消耗降低 99%。


二、HTTP 客户端连接池滥用与生命周期管理混乱
1. 问题定性
您列出的五个 HTTP 客户端实例中，存在两类完全不同的使用模式：

客户端	预期复用模式	当前现实
DeepSeek（LLM 对话）	长连接复用（高频小请求，需利用 HTTP Keep-Alive 和连接池降低延迟）	短生命周期，每次新建
SiliconFlow Embedding	长连接复用（批量向量化，吞吐优先）	短生命周期
SiliconFlow Rerank	长连接复用	短生命周期
Kimi VLM	短连接（单次大文件上传，连接保持时间长但请求频率低）	短生命周期（无妨）
MinerU	短连接（异步任务，无需复用）	短生命周期（无妨）
根本问题：将所有 HTTP 客户端均等对待，且未正确管理 httpx.AsyncClient 的生命周期。

2. 为什么“每次新建 Client”是灾难？
连接池闲置：每次请求结束后 Client 被销毁，TCP 连接随之关闭，下次请求重新 TCP 三次握手 + TLS 四次握手。

端口耗尽：高并发下，短时间内大量 TIME_WAIT 状态的端口堆积，导致无法建立新连接。

无法利用 HTTP/2 多路复用：现代 API 普遍支持 HTTP/2，复用 Client 可以单 TCP 连接承载数百并发流，而每次新建 Client 则退化到 HTTP/1.1 每个请求一个连接。

3. 按需修正方案（第一性原理：连接复用与否取决于请求频次与连接建立开销占比）
A. 对于 DeepSeek / Embedding / Rerank（高频请求）

必须使用应用级全局单例的 httpx.AsyncClient。

python
# llm_client.py 改进
import httpx
from contextlib import asynccontextmanager

class DeepSeekClient:
    _client: httpx.AsyncClient = None

    @classmethod
    async def get_client(cls) -> httpx.AsyncClient:
        if cls._client is None:
            # 配置连接池参数
            limits = httpx.Limits(
                max_keepalive_connections=20,   # 保持 20 个空闲长连接
                max_connections=100,             # 最大并发连接数
                keepalive_expiry=30.0           # 空闲连接 30 秒后关闭
            )
            timeout = httpx.Timeout(
                connect=5.0,
                read=60.0,
                write=10.0,
                pool=5.0
            )
            cls._client = httpx.AsyncClient(
                limits=limits,
                timeout=timeout,
                http2=True  # 若 API 支持 HTTP/2
            )
        return cls._client

B. 对于 Kimi VLM / MinerU（低频大请求）
Kimi VLM：通常涉及文件上传，且单次请求耗时较长（数十秒），不适合占用长连接池中的有限连接。建议每次请求创建独立 Client，但需设置合理的超时和连接关闭策略。
MinerU：异步任务提交，请求体积极小，连接复用收益不大，也可采用独立 Client。
C. 关于 DeepSeek 缓存的特别说明
“希望 DeepSeek 可复用命中缓存”。这里需要区分两层缓存：
HTTP 连接复用：如上述，通过全局 Client 实现，减少 RTT。
语义缓存（Semantic Cache）：如果相同或相似的问题频繁出现，应在业务层引入缓存存储（如 Redis），Key 为问题文本的哈希，Value 为 LLM 响应。这与 HTTP 连接池无关，是独立的优化手段。

三、综合行动清单（按紧急程度排序）
优先级	行动项	预期收益
P0	重构 Qdrant 为单 Collection + Payload 过滤	消除扇出爆炸根源，检索性能提升 100 倍以上
P1	修复高频 HTTP Client（LLM/Embedding/Rerank）为全局单例	避免端口耗尽，降低 P99 延迟 30%~50%
P2	为 Qdrant 客户端实现单例模式	减少 TCP 握手开销，提升批量写入效率
P3	为 DeepSeek 引入语义缓存层	降低 API 费用，提升重复查询响应速度

这等同于买了一辆赛车，却每次比赛都重新启动引擎并热车一圈。 对于 Embedding 和 Rerank 这类高频、小请求体的 API，连接建立开销可能占据总请求时间的 20%~40%（尤其是跨地域 HTTPS）。修复这一项，无需改动业务逻辑，即可获得显著的性能提升和资源节省。
最终结论：当前系统的性能瓶颈并非硬件资源不足，而是架构模式选择（多 Collection） 与 基础设施使用方式（短生命周期 Client） 的双重失误。纠正这两点后，系统容量可支撑 10000+ 论文库 的实时检索。
