### 全局异常处理（`main.py:27-36`）

```python
@app.exception_handler(Exception)
async def general_exception_handler(request, exc):
    return JSONResponse(status_code=500, content={
        "code": 9001, "data": None, "message": f"内部错误：{str(exc)}"
    })
```

**问题**: 所有未捕获异常统一返回 500 + 错误信息字符串，**可能泄露内部实现细节**（如 API Key 错误信息、SQL 语句等）。
￥对于早期开发而言，问题不大，但是我认为应该返回的错误分好类别，不应该这么杂乱；但对于如何详细的捕捉异常，我也不是很有头绪，一般来说单元测试跑通，说明异常的可能性很小；就算有，这种全局异常处理也不会很多￥
￥
【回答】：你的判断是对的。MVP 阶段统一返回 500 + 错误信息是可以接受的，但确实应该分类错误码。

**当前状态**：`main.py:25-28`
```python
async def general_exception_handler(request, exc):
    return JSONResponse(status_code=500, content={
        "code": 9001, "data": None, "message": f"内部错误：{str(exc)}"
    })
```

**建议的错误分类**（🔮 未来扩展）：
| 错误码 | 类型 | HTTP状态码 | 示例 |
|--------|------|-----------|------|
| 1001 | 参数错误 | 400 | 论文ID为空 |
| 2001 | 认证失败 | 401 | API Key无效 |
| 2003 | 权限不足 | 403 | 无权访问该论文 |
| 3001 | 资源不存在 | 404 | 论文不存在 |
| 4001 | 外部服务错误 | 502 | MinerU API 超时 |
| 4002 | 外部服务限流 | 429 | Embedding API 限流 |
| 9001 | 内部错误 | 500 | 其他未捕获异常 |

**实现方案**：
1. 定义异常类：`class APIException(ABC)`
2. 子类化：`class ParamError(APIException)`、`class ExternalServiceError(APIException)`
3. 全局处理器中根据异常类型返回不同的错误码

**优先级**：P2（MVP 后优化）

## 第一阶段：立即优化（P0）

### 1. 自动重试机制（参考 `withRetry`）

**改动位置**：
- 新增：`app/utils/retry.py`
- 修改：`app/clients/embedding_client.py`、`app/clients/rerank_client.py`、`app/clients/vlm_client.py`

**原有代码**（`embedding_client.py:84`）：
```python
async with httpx.AsyncClient(timeout=60.0) as client:
    response = await client.post(...)
    response.raise_for_status()  # ← 失败直接抛异常
```

**修改后**：
```python
from app.utils.retry import async_retry_with_backoff

@async_retry_with_backoff(max_retries=3, base_delay=1.0)
async def _embed_batch_raw(self, texts: list[str]):
    async with httpx.AsyncClient(timeout=60.0) as client:
        response = await client.post(...)
        response.raise_for_status()
```

**影响**：
- ✅ 网络抖动时自动重试，用户体验提升
- ⚠️ 需要区分可重试错误（网络、超时、429）和不可重试错误（401、404）
- ⚠️ 增加请求总时长（最多 1+2+4=7 秒额外延迟）

### 2. 错误分类与用户友好提示

**改动位置**：
- 新增：`app/core/exceptions.py`（定义异常类）
- 修改：`app/main.py`（全局异常处理器）

**原有代码**（`main.py:25-28`）：
```python
async def general_exception_handler(request, exc):
    return JSONResponse(status_code=500, content={
        "code": 9001, "message": f"内部错误：{str(exc)}"
    })
```

**修改后**：
```python
from app.core.exceptions import (
    ExternalServiceError,
    ParamError,
    NotFoundError,
)

async def general_exception_handler(request, exc):
    if isinstance(exc, ExternalServiceError):
        return JSONResponse(status_code=502, content={
            "code": 4001, "message": f"外部服务错误：{exc.service_name} 暂时不可用，请稍后重试"
        })
    elif isinstance(exc, ParamError):
        return JSONResponse(status_code=400, content={
            "code": 1001, "message": f"参数错误：{exc.field_name} 不能为空"
        })
    # ... 其他分类
```

**影响**：
- ✅ 用户看到更友好的错误提示
- ⚠️ 需要在服务层抛出具体的异常类型（不能都用 `raise Exception`）

**改动位置**：

**功能**：

## 总结

**推荐实施路径**：
1. **立即**：第一阶段（自动重试、错误分类）
2. **中期**：第二阶段（Token 预算、查询缓存）
3. **暂缓**：第三阶段（上下文压缩、多轮对话状态机）

**关键原则**：
- 保持简单：只引入必需的功能
- 渐进式重构：每次改动后测试，避免大规模重写
- 监控影响：每个改动都要记录性能指标（响应时间、错误率）

**优先级**：P0 > P1 > P2

## 幂等性分析

| 操作 | 幂等性 | 说明 |
|------|--------|------|
| PDF 导入 | **非幂等** | 重复导入同一 PDF 会创建新的 Collection（以 PDF 名的 hash 为 ID） |
| Qdrant upsert | **幂等** | 相同 point_id 会覆盖 |
| 会话创建 | **非幂等** | 每次调用创建新会话 |
| 消息存储 | **非幂等** | 每次调用插入新消息 |
| PDF 删除 | **幂等** | 删除不存在的 Collection 不报错（Qdrant 行为） |
| Embedding 生成 | **有条件幂等** | `ingest_service` 中有缓存状态检查 |
