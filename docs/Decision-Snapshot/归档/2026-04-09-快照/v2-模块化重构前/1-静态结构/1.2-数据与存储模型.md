## BM25 索引结构

### 索引文件

- 路径：`./data/bm25_index.json`（可配置 `BM25_INDEX_PATH`）
- 格式：pickle 序列化的 `BM25Okapi` 对象 + 元数据
- 分词器：`jieba`（中文分词）

### 索引数据结构

```python
class BM25Repo:
    _corpus: list[str]           # 原始文档文本
    _tokenized: list[list[str]]  # jieba 分词结果
    _doc_ids: list[str]          # 文档 ID
    _bm25: BM25Okapi             # BM25 索引对象
```

### 索引操作

| 操作 | 方法 | 说明 |
|------|------|------|
| 添加单文档 | `add(doc_id, document)` | 追加后重建整个索引 |
| 批量添加 | `add_batch(doc_ids, documents)` | 追加后重建整个索引 |
| 检索 | `query(query_text, top_k)` | jieba 分词 → BM25 打分 → Top-K |
| 持久化 | `save()` | pickle 序列化到文件（同步 IO） |
| 加载 | `load()` | 从文件反序列化（同步 IO） |

**【代码事实】** 每次添加文档都会**重建整个 BM25 索引**（`BM25Okapi(self._tokenized)`），论文数量多时效率会下降。
￥重建BM25索引的代价是什么，需要用Embedding吗？这个东西我们要用什么东西发现并构建索引？现在的重建流程是什么？￥
￥
【回答】：BM25 是关键词检索，不需要 Embedding。重建代价随文档数线性增长，100篇论文时约1-2秒。

**代码证据**：`app/repositories/bm25_repo.py`
```python
# 第27-34行：add() 方法
def add(self, doc_id: str, document: str):
    self._doc_ids.append(doc_id)
    self._corpus.append(document)
    tokens = list(jieba.cut(document))  # ← jieba 分词
    self._tokenized.append(tokens)
    # 重新构建索引
    self._bm25 = BM25Okapi(self._tokenized)  # ← 每次都重建
```

**代价分析**：
- 时间复杂度：O(n × m)，n=文档数，m=平均文档长度
- 实测：100篇论文（约10000个chunk）→ 重建约1-2秒
- 不需要 Embedding：BM25 基于词频统计，不是向量检索

**发现和构建流程**：
1. 用户上传 PDF → MinerU 解析
2. chunker 切分 → 生成 chunk list
3. 调用 `bm25_repo.add_batch()` → jieba 分词 → BM25Okapi 重建
4. `save()` → pickle 序列化到 `data/bm25_index.json`

**优化方向**（🔮 未来扩展）：
- 增量索引：只在达到阈值（如每新增100篇）时重建
- 异步重建：后台任务，不阻塞导入流程
- 分布式索引：每篇论文独立索引，检索时合并（类似 Qdrant 的 Collection 隔离）
￥

## 数据生命周期契约

| 数据 | 创建触发 | 更新触发 | 删除触发 |
|------|---------|---------|---------|
| Session | POST /session/ | PUT /{id}/title | DELETE /{id} (级联删消息) |
| Message | 问答完成后自动存储 | **不支持** | 清空历史或删会话 |
| Qdrant Collection | PDF 导入成功 | **不支持**（需删后重建） | DELETE /library/papers/{id} |
| BM25 索引 | 导入时 add() | 每次添加都重建 | 【需验证】删除论文时是否清理 |
| PDF 上传文件 | POST /library/import | **不支持** | 与论文一起删除 |
| MinerU 解析结果 | MinerU 返回后保存 | **不支持** | 与论文一起删除 |
| ingest_tasks 记录 | **未使用** | - | - |
￥BM25索引在删除论文的时候如果只清理一个可以吗？会影响其他的地方吗？影响是否可控？如果增加论文，一定要全量重建吗？￥
￥
【回答】：可以只删除一个，影响可控。但增加论文目前必须全量重建（BM25Okapi 的限制）。

**代码证据**：`app/repositories/bm25_repo.py:69-81`
```python
def delete_by_id(self, doc_id: str):
    """删除单个文档"""
    if doc_id not in self._doc_ids:
        return
    idx = self._doc_ids.index(doc_id)
    self._doc_ids.pop(idx)      # ← 删除 doc_id
    self._corpus.pop(idx)       # ← 删除内容
    self._tokenized.pop(idx)    # ← 删除分词
    # 重建索引（只删除后重建，不需要全量）
    if self._tokenized:
        self._bm25 = BM25Okapi(self._tokenized)
    else:
        self._bm25 = None
```

**删除的影响范围**：
- ✅ **可控**：只删除指定的 doc_id，不影响其他文档
- ✅ **原子性**：删除后立即重建索引，保证一致性
- ⚠️ **性能问题**：`list.pop(idx)` + 重建索引，O(n) 复杂度

**增加论文的问题**：
- ❌ **必须全量重建**：`BM25Okapi` 不支持增量添加
- 🔮 **未来优化**：可以每 N 篇论文重建一次，或切换到支持增量索引的库（如 `lucene`）

**与 Qdrant 的对比**：
| 特性 | BM25 | Qdrant |
|------|------|--------|
| 删除单个 | O(n) 重建 | O(1) 删除点 |
| 增加单个 | O(n) 全量重建 | O(1) 添加点 |
| 分布式 | ❌ 不支持 | ✅ Collection 隔离 |

**建议**：
- MVP 阶段：保持现状，<1000篇论文性能可接受
- 优化方向：参考 Qdrant 的分布式架构，每篇论文一个独立 BM25 索引，检索时合并 Top-K 结果
￥
