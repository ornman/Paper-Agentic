## 内存占用估算

### Qdrant 内存模型

| 组件 | 内存占用公式 | 100 篇估算 | 500 篇估算 |
|------|------------|-----------|-----------|
| HNSW 索引 | 向量数 × 1536 × 4 × 1.5（HNSW 开销） | ~1.5 GB | ~7.5 GB |
| Payload 缓存 | 文本内容大小 × 1.0 | ~0.5 GB | ~2.5 GB |
| **Qdrant 总计** | | **~2.0 GB** | **~10.0 GB** |

**【模型推断】** 假设每篇论文产生 ~200 个 chunks（每个 1536 维 × 4 字节 = 6KB 向量）：
- 100 篇 = 20,000 chunks = 120MB 原始向量 × 1.5 HNSW 开销 = ~180MB
- 500 篇 = 100,000 chunks = 600MB × 1.5 = ~900MB
- 2000 篇 = 400,000 chunks = 2.4GB × 1.5 = ~3.6GB

实际内存取决于 Qdrant 的加载策略和 HNSW 参数。

### Python 进程内存

| 组件 | 估算 | 说明 |
|------|------|------|
| FastAPI + 依赖 | ~100 MB | 基础框架 |
| BM25 索引 | ~10-200 MB | 取决于论文数量，全量内存 |
| jieba 词典 | ~100 MB | 中文分词词典 |
| 临时数据（导入/检索） | ~50-500 MB | 取决于并发操作 |

### 内存占用总结

| 论文数量 | Qdrant + Python | 建议最低内存 |
|---------|----------------|-------------|
| 100 篇 | ~2.5 GB | 4 GB |
| 500 篇 | ~12 GB | 16 GB |
| 2000 篇 | ~25+ GB | **32 GB** |
￥这个内存肯定不行，我只接收存储这么大，但不接受运行时这么大。我宁愿降低效率也要全面￥

【问题】：Qdrant 的 HNSW 索引是**内存常驻**的，论文数量多时内存占用很高。

**代码证据**：`qdrant_store.py`
```python
# Qdrant 默认配置（未显式指定 HNSW 参数）
client.create_collection(
    collection_name=collection_name,
    vectors_config=VectorParams(size=1536, distance=Distance.COSINE),
    # ← HNSW 索引常驻内存
)
```

**内存占用分析**：
- 100 篇论文 → 约 10000 个 chunk × 1536 维 × 4 字节 = ~60 MB（向量）
- HNSW 索引 → 约 2-3 GB（图结构）
- Python 进程 → 约 500 MB
- **总计**：~2.5-3 GB

**降低内存的方案**：

### 方案 A：降低 HNSW 参数（简单）

**改动位置**：`qdrant_store.py`
```python
from qdrant_client.models import HnswConfigDiff

client.create_collection(
    collection_name=collection_name,
    vectors_config=VectorParams(size=1536, distance=Distance.COSINE),
    hnsw_config=HnswConfigDiff(
        m=16,          # 默认 16，降低到 8（牺牲精度）
        ef_construct=100,  # 默认 100，降低到 50（构建速度）
    ),
)
```

**影响**：
- ✅ 内存降低 ~30%
- ❌ 检索精度下降

### 方案 B：磁盘索引（Qdrant on-disk）

**改动位置**：`config.py` + `qdrant_store.py`
```python
# 使用 Qdrant 的磁盘存储模式
client = QdrantClient(path=settings.qdrant_path, storage_type="on_disk")
```

**影响**：
- ✅ 内存占用降低 ~90%
- ❌ 检索速度降低 ~50%（磁盘 IO）

### 方案 C：分布式架构（复杂）

**改动位置**：重构 `qdrant_store.py`
- 每篇论文一个 Collection
- 检索时并行查询多个 Collection，然后 RRF 融合
- 单个 Collection 的内存占用可控

**影响**：
- ✅ 内存可控（每篇论文独立）
- ✅ 支持增量扩展
- ❌ 架构复杂度增加

---

**建议**：
- **短期**：方案 B（磁盘索引），牺牲速度换取内存
- **长期**：方案 C（分布式架构），已在 `retrieval_service.py` 中实现 RRF 融合

**优先级**：P1（影响用户硬件需求）

## 临时文件清理策略

| 临时文件 | 清理策略 | 风险 |
|---------|---------|------|
| `uploads/*.pdf` | **无自动清理** | 磁盘持续增长 |
| `papers/{id}/` | 删论文时 `shutil.rmtree` | 仅手动触发 |
| MinerU 下载的临时 zip | 取决于 `parse_pdf` 实现 | 【需验证】是否解压后删除 zip |
| BM25 重建时的临时数据 | 内存中处理，无临时文件 | 低风险 |

## 网络带宽估算

### 导入单篇论文

| 操作 | 数据量 | 方向 |
|------|--------|------|
| 上传 PDF 到 MinerU | 5-20 MB | 上行 |
| 下载解析结果 | 10-30 MB | 下行 |
| VLM 图片描述请求 | 10-30 次 × 图片数据 | 双向 |
| Embedding API 请求 | ~200 次 × 文本 | 双向 |
| **总计** | **50-150 MB** | 混合 |

### 检索单次问答

| 操作 | 数据量 | 方向 |
|------|--------|------|
| Query Embedding | < 1 KB | 上行 |
| Qdrant 检索 | 内存操作 | 本地 |
| Rerank 请求 | Top-K 文本 (~50KB) | 上行 |
| DeepSeek/Kimi 问答 | ~2-5 KB | 双向 |
| **总计** | **~55 KB** | 混合 |
