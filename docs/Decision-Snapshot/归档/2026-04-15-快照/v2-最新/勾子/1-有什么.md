**关键架构特征**：

1.  **三层模块化结构**：`modules/` 下按业务域组织（ingestion/library/qa/retrieval/session），每个模块包含 models/repository/service，形成领域边界。
2.  **双客户端系统**：`app/clients/` 对外服务，`app/modules/ingestion/mineru_client.py` 是模块内副本。两份 `mineru_client` 同时存在。
3.  **`services_archive/` 目录**：6 个已废弃文件仍留在代码树中，未清理。
4.  **路由层直接实例化服务**（无 DI 容器）：每个路由函数体内 `_get_xxx_service()` 手动构造，例如 `query.py:13`、`library.py:25`。

## 数据流

### 导入链路（6 阶段）

`library.py:import_pdf` → `LibraryService.import_pdf()` → `IngestionService.ingest_document()`

```
pending → parsing(MinerU) → cleaning(清洗) → indexing(Embedding) → Qdrant存储 → completed
                                                          ↑ 任何阶段失败 → failed
```

证据：`backend/app/modules/ingestion/service.py:50-126`

### 问答链路（SSE 流式）

`query.py:/ask` → `QAService.ask_stream_with_rag()`

```
Query改写(LLM) → RAG检索(向量+BM25+Rerank) → SSE流式生成(Kimi LLM) → 保存消息(SQLite)
```

证据：`backend/app/modules/qa/service.py:79-187`

### 前端-后端通信

- **JSON API**：`api-client.ts` → `fetch()` → 后端 `/api/v1/library/*`
- **SSE 流**：`sse-client.ts` → `fetch()` → 后端 `/api/v1/query/ask`
- **WPS 宿主**：`wps-host.ts` → `globalThis.wps.WpsApplication()` → 5s 轮询

## 风险点

1.  **`.env` 包含明文 API Key 并已提交到仓库**。文件 `backend/.env` 含 3 组真实 API Key（DeepSeek `sk-302c...`、SiliconFlow `sk-yjpb...`、MinerU JWT），但 `.gitignore:1` 虽有 `.env` 条目，该文件在 git 历史中是否曾被提交需进一步确认。如果已提交，密钥已泄露。

2.  **向量库配置不一致**：`.env` 仍含 `CHROMA_PERSIST_DIR` / `CHROMA_COLLECTION_NAME`（`backend/.env:31-32`），代码已全部迁移到 Qdrant。`config.py` 无 ChromaDB 字段。这意味着 `.env` 中有两行配置永远不被读取，但会给维护者造成"还在用 ChromaDB"的错觉。

3.  **`services_archive/` 废弃代码未清理**（`backend/app/services_archive/`，6 个文件）。如果其他模块有 import 指向这里，不会被静态分析发现。

4.  **MinerU 客户端双份存在**：`app/clients/mineru_client.py` 和 `app/modules/ingestion/mineru_client.py` 同时存在。导入服务使用的是后者（`backend/app/modules/ingestion/service.py:19`），前者可能已废弃。

5.  **CORS 全开**：`app/main.py:22` 设置 `allow_origins=["*"]`。开发阶段可接受，但与 `.env` 中 `APP_HOST=127.0.0.1` 的本地部署意图矛盾——任何网页都能跨域调用后端 API。

6.  **`pytest` 在生产依赖中**：`pyproject.toml:17` 把 `pytest` 和 `pytest-asyncio` 放在 `[project.dependencies]` 而非开发依赖组。Docker 镜像会包含测试框架。

7.  **Kimi API 需伪装 User-Agent**：`kimi_client.py:28` 硬编码 `User-Agent: claude-code`，如果 Kimi 方面封禁此 UA，所有 VLM 图片描述 + 备用 LLM 问答链路会同时失效。

8.  **QAService 使用 KimiLLMClient 而非 DeepSeek LLMClient**：`qa/service.py:75` 默认 `self.llm_client = KimiLLMClient()`，但健康检查 `health.py:17` 检测的是 `settings.llm_api_key`（DeepSeek）。健康检查说"LLM 可用"并不意味着 QA 链路真正可用——QA 用的是 Kimi Key。

9.  **KimiLLMClient.chat_stream 是伪流式**：`kimi_client.py:219-224` 的 `chat_stream` 先完整等待响应，再按 50 字符切块 yield。SSE 的低延迟优势完全丧失。

10. **前端 SSE 安全校验逻辑重复**：`api-client.ts:76-109` 和 `sse-client.ts:67-98` 各自实现了完全相同的 `normalizeConfiguredApiBaseUrl` + `LOCAL_API_HOSTNAME_ALLOWLIST` 逻辑，且 `DEFAULT_API_BASE_URL` 也各写了一份。如果后端地址变更，需要改两处。
