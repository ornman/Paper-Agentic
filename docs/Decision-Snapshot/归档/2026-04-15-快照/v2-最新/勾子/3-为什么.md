# 3-为什么这样？——设计意图

## 架构决策

### 为什么选 FastAPI？

**[事实]** 设计文档 `docs/plans/2026-03-25-mvp-backend-design.md` 明确说明：

> 后端采用模块化单体（Modular Monolith），运行上是一个本地 FastAPI 服务。

**[事实]** `app/main.py` 使用 FastAPI + CORSMiddleware + 全局异常处理，启动时自动初始化数据库。

**[推断] 选择 FastAPI 的原因：**
1. **异步原生**：SSE 流式输出是核心需求（问答场景），FastAPI 原生支持 `StreamingResponse` + `async/await`，无需额外适配。
2. **本地单机 MVP**：设计文档明确"部署形态为纯本地单机 MVP"，FastAPI 轻量启动快，适合单进程本地服务。
3. **自动 OpenAPI 文档**：`CLAUDE.md` 中提到"API 文档见 `http://localhost:8000/docs`"，零配置即可获得调试界面。
4. **Pydantic 集成**：项目大量使用 Pydantic 模型（`DocumentRecord`, `CleanedBlock`, `AskRequest` 等），FastAPI 与 Pydantic 深度整合，自动做请求校验。

### 为什么选 Qdrant（而不是设计文档中提到的 ChromaDB）？

**[事实]** 设计文档 `2026-03-25-mvp-backend-design.md` 第 8.1 节明确写"使用 ChromaDB"。但实际代码中：

- `stores/qdrant_store.py` 使用 Qdrant
- `config.py` 配置了 `qdrant_path`、`qdrant_url`、`qdrant_api_key`、`qdrant_hnsw_m` 等参数
- ChromaDB 仅在 `INGEST_DESIGN.md`（旧文档）中被提及

**[推断] 从 ChromaDB 迁移到 Qdrant 的原因：**
1. **分布式 Collection 支持**：`qdrant_store.py` 的文档注释"每篇论文独立 collection，支持分布式隔离管理"，Qdrant 原生支持动态创建/删除 Collection，而 ChromaDB 的 Collection 管理较粗糙。
2. **内存优化配置**：代码中显式配置了 HNSW 参数（`m=12`, `ef_construct=64`, `full_scan_threshold=5000`），注释说"降低内存占用的策略"，说明遇到了 ChromaDB 的内存瓶颈。
3. **分布式检索架构**：`retrieval_service.py` 注释说"分布式架构优势：每种资源类型独立 Collection...删除/更新原子性"，Qdrant 更适合这种每篇论文独立 Collection 的模式。

**[风险] ⚠️ 设计文档与实际代码不一致**：设计文档仍写 ChromaDB，但代码已切换到 Qdrant，文档未同步更新。

### 为什么选分布式 Collection（每篇论文独立 Collection）？

**[事实]** `qdrant_store.py` 第 4 行注释：

> 每篇论文独立 collection，支持分布式隔离管理。

**[事实]** `retrieval_service.py`（archive 版本）注释：

> 分布式架构优势：
> - 每种资源类型独立 Collection（paper_xxx, video_xxx, note_xxx...）
> - 删除/更新原子性：直接删除整个 Collection
> - 可扩展到知识图谱：Collection 作为图节点

**[事实]** `config.py` 中 `DOCUMENT_INDEX_MODES` 包含 `"distributed"` 模式。

**[推断] 原因：**
1. **删除原子性**：`delete_paper()` 直接 `delete_collection`，无需逐条清理向量，避免残留脏数据。
2. **错误隔离**：单篇论文索引失败不影响其他论文的 Collection。
3. **扩展性预留**：注释中多次提到"未来扩展到视频、笔记、网页剪藏等"，Collection 命名规范 `paper_{uuid}`, `video_{uuid}` 已经预定义。
4. **检索灵活性**：`search_all()` 方法遍历所有 Collection 并行搜索，支持跨论文全局检索；`search()` 方法在单篇论文内搜索。

### 为什么 Embedding 模型和维度是硬约束？

**[事实]** `config.py` 第 8-12 行定义了固定常量：

```python
PINNED_EMBEDDING_MODEL = "Qwen/Qwen3-Embedding-8B"
PINNED_EMBEDDING_DIMENSIONS = 1536
PINNED_RERANK_MODEL = "Qwen/Qwen3-Reranker-8B"
```

**[事实]** `config.py` 第 318-329 行有 `validate_pinned_model_contract` 校验器，如果值不匹配会抛出 `ValueError`。

**[事实]** `config.py` 注释明确：

> 这里不是"推荐值"，而是"系统硬约束"。一旦模型名或维度漂移，历史向量、检索结果和索引结构都会失去一致性。

**[事实]** `INGEST_DESIGN.md` 记录了一个实际踩坑：

> 错误 1：Embedding 维度错误。测试时误将 EMBEDDING_DIMENSION 从 1536 改为 4096，后果是向量空间稀疏，检索质量下降。

**[推断] 这是经验教训的直接结果**：团队已经因维度漂移吃过亏，因此加入硬约束校验防止再犯。

### 为什么用 Kimi Coding API 做 VLM？

**[事实]** `kimi_client.py` 第 4 行注释：

> 必须伪装 User-Agent: claude-code 才能正常使用。

**[事实]** `kimi_client.py` 第 28 行请求头设置 `"User-Agent": "claude-code"`。

**[推断] 选择 Kimi 的原因：**
1. **成本**：Kimi Coding API 可能是免费或低成本的（需要伪装 UA 才能用），适合 MVP 阶段。
2. **图片理解能力**：用于论文图表理解，VLM Prompt（第 34-105 行）非常详尽，覆盖流程图、架构图、UML 图、ER 图等学术常见图表类型。

### 为什么采用三阶段导入流程？

**[事实]** `INGEST_DESIGN.md` 完整记录了设计缺陷和重新设计过程：

> 错误 3：没有中间缓存和断点续传。清洗完直接入库，失败了就全丢。

**[事实]** 新设计为：清洗 -> JSON 缓存 -> 向量化 -> 入库，每阶段有独立状态标记。

**[推断] 这是典型的"踩坑后重构"**：旧方案没有断点续传，大批量导入中途失败要重头开始，因此引入了三阶段 + JSON 中间缓存。

### 为什么使用状态机管理文档生命周期？

**[事实]** `models.py` 第 62-71 行定义了完整的 `ALLOWED_STATUS_TRANSITIONS`：

```python
"pending": {"parsing", "failed", "deleting"},
"parsing": {"cleaning", "failed", "deleting"},
...
"deleted": set(),
```

**[事实]** `LibraryService` 的 `reindex_document()` 方法注释：

> Task 3 还没有真正的索引模块，因此这里必须避免把健康的 completed 文档推进到 indexing，否则会制造一个当前任务无法自愈的卡死状态。

**[推断] 状态机的目的是防止非法状态转移**：团队意识到"状态卡死"是实际痛点（如文档永远停在 `indexing`），因此用白名单严格控制状态转换路径。

### 为什么分 `services_archive/` 和 `modules/` 两套代码？

**[事实]** `services_archive/` 包含 5 个文件：
- `qa_service.py` — 旧问答服务（纯 LLM，无 RAG）
- `qa_service_rag.py` — 旧 RAG 问答（初步集成）
- `embedding_service.py` — 旧向量化服务
- `ingest_service.py` — 旧导入服务
- `retrieval_service.py` — 旧检索服务

**[事实]** `modules/` 目录按业务域组织：`session/`, `library/`, `ingestion/`, `retrieval/`, `qa/`。

**[事实]** 设计文档第 15 节明确说"保留 FastAPI 壳，重写偏离需求的业务层"。

**[推断] 这是一次架构重构的遗迹**：
1. 旧代码（`services_archive/`）是早期"所有 service 堆在一起"的风格。
2. 新代码（`modules/`）是设计文档要求的"按业务领域拆模块"的模块化单体。
3. 旧文件保留而非删除，可能是因为部分路由或其他模块仍在引用旧代码中的某些函数（如 `qa_service_rag.py` 引用了 `rewrite_query`），需要渐进迁移。

### 为什么前端使用 SSE 而非 WebSocket？

**[事实]** `sse-client.ts` 实现了完整的 SSE 客户端，包含：
- 帧解析器 `parseSseFrame()`
- 流式读取器 `readSseStream()`
- 重试机制 `postAskInspirationStreamWithRetry()`

**[事实]** 后端 `query.py` 使用 `StreamingResponse(media_type="text/event-stream")`。

**[推断] 选择 SSE 的原因：**
1. **单向推送足够**：问答场景只需要服务器向客户端推送，不需要双向通信。
2. **HTTP 原生**：SSE 基于 HTTP，无需额外协议升级，WPS 插件环境可能对 WebSocket 有限制。
3. **自动重连**：SSE 协议内置重连机制，配合前端的 `RetryConfig` 实现了指数退避重试。
4. **防火墙友好**：SSE 走标准 HTTP，不存在被代理/防火墙阻断的问题。

### 为什么前端有严格的本机回环地址校验？

**[事实]** `sse-client.ts` 第 36-97 行实现了多层 URL 校验：
- `LOCAL_API_HOSTNAME_ALLOWLIST` 只允许 `127.0.0.1`, `localhost`, `[::1]`, `::1`
- `normalizeConfiguredApiBaseUrl()` 校验协议、主机名、纯 origin
- `normalizeCurrentRuntimeOrigin()` 检查当前页面是否本机

**[事实]** 注释说明：

> ask 接口虽然不发送本地路径，但会发送当前论文正文，因此仍然应该只允许本机回环地址，避免误把正文发到外部主机。

**[推断] 安全考量**：这是一个本地部署的 WPS 插件，会处理用户的论文正文（敏感数据），必须防止数据泄露到外部服务器。

### 为什么 Embedding 和 Rerank 统一走 SiliconFlow？

**[事实]** `config.py` 第 78 行注释：

> Embedding 与 Rerank 统一走 SiliconFlow，避免不同服务商混用导致契约漂移。

**[推断] 原因：**
1. **契约一致性**：同一服务商的 Embedding 和 Rerank 模型可能共享底层 tokenizer 或语义空间，混合检索 + 重排序的效果更好。
2. **简化配置**：只需一个 API Key 和 Base URL，减少配置出错的可能。
3. **网络延迟**：同服务商内部调用延迟更低。

### 为什么使用 SQLite 而非 PostgreSQL？

**[事实]** `sqlite_repo.py` 使用 SQLAlchemy + SQLite，配置了 WAL 模式和 StaticPool。

**[事实]** `config.py` 默认 `sqlite_db_path = "./data/app.db"`。

**[事实]** 设计文档明确"部署形态为纯本地单机 MVP"。

**[推断] 原因：**
1. **零运维**：SQLite 是文件数据库，无需安装和启动数据库服务。
2. **MVP 足够**：单用户本地使用场景下，SQLite 的并发限制不是问题。
3. **WAL 模式**：代码中启用了 WAL 模式和 64MB 缓存，解决了一般 SQLite 的读写并发问题。
