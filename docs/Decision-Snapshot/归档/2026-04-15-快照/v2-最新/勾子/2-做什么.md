## 核心流程

### 流程一：PDF 导入流程（6 阶段管线）

这是系统最长的同步链路，从用户上传到可检索。

```
用户上传/指定路径
      |
      v
[LibraryService.import_pdf()]       library/service.py:63
      |
      |-- 校验路径安全性（防遍历/URL/符号链接）  library/service.py:138-183
      |-- import_document() 创建 pending 记录    library/service.py:84
      v
[IngestionService.ingest_document()]  ingestion/service.py:50
      |
      |-- 阶段1: pending → parsing
      |   MinerU 解析 PDF                  ingestion/service.py:58-61
      |   校验返回结构                       ingestion/service.py:308-329
      v
      |-- 阶段2: parsing → cleaning
      |   clean_mineru_payload()           ingestion/cleaning.py:38
      |   - 提取块（文本+图片）               cleaning.py:111-136
      |   - 过滤噪音（页眉页脚/页码/短块/重复） cleaning.py:273-298
      |   - 图片生成 VLM 描述                 cleaning.py:301-395
      v
      |-- 阶段3: cleaning → VLM图片描述
      |   describe_chunks_async()          processing/describer.py:45
      |   - 加载/更新 JSON 缓存              describer.py:24-42
      |   - 并发调用 VLM（信号量控制）         describer.py:78-108
      v
      |-- 阶段4: 混合语义切分
      |   chunk_by_semantic_units()        processing/chunker.py:205
      |   - 两个小块合并（合计 ≤ 32k）        chunker.py:84-135
      |   - 超长块拆分（带重叠）              chunker.py:142-202
      v
      |-- 阶段5: Embedding（带重试）
      |   EmbeddingClient.embed()          clients/embedding_client.py:64
      |   - 分批并发 + 速率限制              embedding_client.py:100-132
      |   - 维度校验                         embedding_client.py:200-211
      v
      |-- 阶段6: indexing → Qdrant 存储
      |   QdrantStore.add_chunks()         stores/qdrant_store.py:113
      |   - 幂等写入（先删旧点再插入新点）      qdrant_store.py:142-147
      |   - 每篇论文独立 collection           qdrant_store.py:66-111
      v
  completed（成功）或 failed（标记错误阶段）
```

**文档状态机** (`library/models.py:62-71`)：

```
pending → parsing → cleaning → indexing → completed
   |         |          |          |
   +--→ failed ←--------+----------+   (任意阶段失败)
            |
            +--→ pending (resume 断点续传)

任意状态 → deleting → deleted
```

**关键风险点**：

| 风险 | 级别 | 说明 | 证据 |
|------|------|------|------|
| MinerU API 不可用 | 🔴 | 解析阶段完全依赖外部服务，不可重试 | `ingestion/service.py:60`，MinerUClient 同步调用 |
| VLM 描述阻塞导入 | ⚠️ | 图片描述使用 Kimi API，超时 120s，并发受信号量控制 | `kimi_client.py:297`，`describer.py:79` |
| Embedding 速率限制 | ⚠️ | 硅基流动 API 限流，有重试但可能全部失败 | `embedding_client.py:189-193` |
| 清洗后无有效块 | ⚠️ | 短论文或全图片论文可能清洗后为空，直接失败 | `ingestion/service.py:69-74` |

### 流程二：RAG 问答流程

这是系统的核心用户交互流程。

```
前端 POST /api/v1/query/ask
  payload: { session_id, query, context?, resource_types?, selected_papers? }
      |
      v
[query.py ask()] → StreamingResponse(SSE)     query.py:23-112
      |
      v
[QAService.ask_stream_with_rag()]             qa/service.py:79
      |
      |-- 1. Query 改写                         qa/service.py:122-126
      |   QueryRewriteService.rewrite()          services/query_rewrite_service.py:97
      |   - 4 种场景模板                         query_rewrite_service.py:22-88
      |   - LLM 非流式调用                       query_rewrite_service.py:122
      |   - 降级：失败时用原始查询                query_rewrite_service.py:126
      |   → yield { type: "rewrite" }           qa/service.py:129-136
      |
      |-- 2. RAG 检索                           qa/service.py:139-151
      |   RetrievalService.retrieve()           retrieval/service.py:36
      |   - 向量检索：Embedding → Qdrant search_all  retrieval/service.py:64-69
      |   - Rerank：硅基流动 Reranker            retrieval/service.py:72-78
      |   - 降级：Embedding 失败 → BM25          retrieval/service.py:98-122
      |   → yield { type: "sources" }           qa/service.py:153-157
      |
      |-- 3. 构建消息                            qa/service.py:160
      |   _build_messages()                      qa/service.py:29
      |   - system prompt + 最近 10 轮历史        qa/service.py:43-44
      |   - RAG 上下文注入到 user message         qa/service.py:49-59
      |
      |-- 4. 流式 LLM 生成                       qa/service.py:163-175
      |   KimiLLMClient.chat_stream()            clients/kimi_client.py:203
      |   - ⚠️ 注意：KimiLLMClient 是伪流式       kimi_client.py:218-224
      |     实际先完整获取再分块 yield
      |   - retry_async 包装                     qa/service.py:167
      |   → yield { type: "chunk" } N 次         qa/service.py:169
      |
      |-- 5. 保存消息                            qa/service.py:178-185
      |   add_message(user) + add_message(assistant, sources)
      |
      → yield { type: "done" }                  qa/service.py:187
```

**Query 改写的 4 种场景** (`query_rewrite_service.py:130-177`)：

| 场景 | 输入 | 权重 | 模板 |
|------|------|------|------|
| 1 | 仅有 prompt | 100% prompt | `_REWRITE_PROMPT_ONLY` |
| 2 | 仅已写内容 | 推断续写意图 | `_REWRITE_WRITTEN_ONLY` |
| 3 | 已写 + 圈选 | 30% 已写 + 70% 圈选 | `_REWRITE_WRITTEN_SELECTED` |
| 4 | 已写 + 圈选 + prompt | 20% + 30% + 50% | `_REWRITE_FULL` |

**关键风险点**：

| 风险 | 级别 | 说明 | 证据 |
|------|------|------|------|
| KimiLLMClient 伪流式 | 🔴 | `chat_stream` 实际是非流式调用后分块返回，用户感知不到真正逐字流式 | `kimi_client.py:218-224`：先 `await self.chat()` 再按 50 字符切块 |
| RAG 检索失败静默降级 | ⚠️ | 检索失败时 `sources = []`，用户不知道发生了什么 | `qa/service.py:148-150` |
| Query 改写失败静默降级 | ⚠️ | 改写失败回退原始查询，无用户提示 | `qa/service.py:124-126` |
| 历史消息无截断保护 | ⚠️ | 取最近 20 条（10 轮），超长历史可能导致 token 溢出 | `qa/service.py:44` |
| `resource_types`/`selected_papers` 未实现 | ⚠️ | 预留接口，检索时实际忽略 | `retrieval/service.py:56-60` 注释掉 |

### 流程三：SSE 流式输出协议

后端 SSE 事件格式 (`query.py:80-84`)：

```
event: {event_type}
data: {json_payload}

```

事件类型定义：

| 事件类型 | 方向 | 数据结构 | 证据 |
|----------|------|----------|------|
| `rewrite` | 后端→前端 | `{ original, rewritten: [str] }` | `qa/service.py:129-136` |
| `sources` | 后端→前端 | `{ sources: [...] }` | `qa/service.py:153-157` |
| `chunk` | 后端→前端 | `{ content, index }` | `qa/service.py:169` |
| `done` | 后端→前端 | `{ total_tokens }` | `qa/service.py:187` |
| `error` | 后端→前端 | `{ code, message, request_id }` | `query.py:88-102` |

前端 SSE 解析器 (`sse-client.ts:167-195`)：

- 逐帧解析 `event:` + `data:` 字段
- `done` 和 `error` 为终止事件，触发 `reader.cancel()`
- 缓冲区大小限制 1MB，单帧 100KB (`sse-client.ts:297-298`)
- 带重试的请求封装：指数退避 1s → 2s → 4s，最多 3 次 (`sse-client.ts:445-489`)

**前端状态机** (`stores/conversation.ts:8`)：

```
idle → requesting → streaming → done
                       |            |
                       +→ error ←---+  (可重试)
```

关键设计：`pendingAssistantSources` 暂存来源，等首个 `chunk` 到来后才创建助手消息壳，避免空壳消息（`conversation.ts:130-137`）。

## 业务逻辑层

### 服务层架构

系统采用分层架构，职责清晰：

```
路由层 (routes/*.py)
  ├── 仅做 HTTP 协议适配、参数提取、错误格式化
  └── 委托给服务层
       |
服务层 (modules/*/service.py)
  ├── 业务编排、状态管理、降级策略
  └── 委托给客户端/仓储/处理层
       |
客户端层 (clients/*.py)
  ├── 外部 API 调用封装
  └── 抽象接口 + 具体实现（可替换）
       |
仓储/存储层 (repositories/, stores/)
  ├── SQLite 持久化
  ├── Qdrant 向量存储
  └── BM25 关键词索引
```

### 服务实例化模式

⚠️ **每次请求新建服务实例**：路由层的 `_get_*_service()` 工厂函数在每次请求时 `return XxxService()`，不缓存实例。这意味着：

- `EmbeddingClient` 每次重建 `httpx.AsyncClient`，未复用 TCP 连接（`embedding_client.py:44-52` 有复用逻辑，但 `EmbeddingClient` 对象本身被反复创建）
- `QdrantStore` 每次重新初始化 Qdrant 客户端（`qdrant_store.py:48-64`）
- `BM25Repo` 每次从磁盘重新加载 pickle 索引（`bm25_repo.py:32-33`）

### 检索服务：混合检索 + 降级 (`retrieval/service.py`)

```
主路径：Embedding → Qdrant search_all → Rerank → Top-K
  失败降级：BM25 → 简化 payload
```

- 向量检索：对所有 `paper_*` collection 逐一搜索，合并排序（`qdrant_store.py:226-249`）
- Rerank：仅当 `vector_results > top_k` 时才调用 Reranker（`retrieval/service.py:72-78`）
- BM25 降级：使用 jieba 分词 + BM25Okapi，返回简化 payload（`retrieval/service.py:103-122`）

### 双 LLM 客户端体系

系统使用两套 LLM 客户端，用途不同：

| 客户端 | 用途 | API 协议 | 流式支持 |
|--------|------|----------|----------|
| `LLMClient` (OpenAI 兼容) | Query 改写、Config 测试 | OpenAI API (AsyncOpenAI) | 真流式 (`llm_client.py:45-65`) |
| `KimiLLMClient` | QA 问答主力 LLM | Anthropic 兼容 (伪装 Claude Code) | 伪流式 (`kimi_client.py:218-224`) |

`QAService` 使用 `KimiLLMClient` 作为主力 LLM（`qa/service.py:75`），意味着问答流程是伪流式。`QueryRewriteService` 使用 `LLMClient`（`query_rewrite_service.py:95`），仅做非流式改写。

### 前端行为

前端（Vue 3 + Pinia）核心状态在 `conversation.ts` store 中：

- **单会话模式**：`sessionId` 在 `reset()` 时重新生成（`conversation.ts:61-69, 476`）
- **两类用户动作**：`askInspiration`（基于草稿获取灵感）和 `sendPrompt`（直接提问）（`conversation.ts:330, 405`）
- **WPS 集成**：通过 `wps-host.ts` 和 `wps-selection.ts` 获取当前文档正文和圈选文本
- **安全边界**：API 基地址只允许本机回环地址（`api-client.py:44`, `sse-client.ts:36`）

### 前端-后端交互时序（问答场景）

```
[前端]                          [后端]
  |                               |
  |-- POST /query/ask ----------->|
  |   {session_id, query,         |
  |    context, selected_papers}  |
  |                               |-- QAService.ask_stream_with_rag()
  |                               |   1. QueryRewriteService.rewrite()
  |<-- event: rewrite ------------|   (yield 改写结果)
  |                               |   2. RetrievalService.retrieve()
  |<-- event: sources ------------|   (yield 引用来源)
  |   {sources: [...]}            |   3. _build_messages() + RAG context
  |                               |   4. KimiLLMClient.chat_stream()
  |<-- event: chunk --------------|   (yield 第一个文本块)
  |   {content, index: 0}        |   → 前端此时才创建 assistant message
  |<-- event: chunk --------------|   (yield 后续块...)
  |   {content, index: N}        |
  |                               |   5. add_message() 保存到 SQLite
  |<-- event: done ---------------|
  |   {total_tokens}             |
  |                               |
  |-- 页面渲染完整回复 + 来源卡片 --|
```
