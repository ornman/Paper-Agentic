## 三、架构瓶颈清单

### 3.1 当前瓶颈

| 瓶颈 | 影响 | 触发条件 | 解决方案 |
|------|------|---------|---------|
| **Qdrant串行检索** | 延迟>1秒 | 论文>500篇 | 改用单一collection+payload过滤 |
| **SQLite并发** | 响应超时 | QPS>100 | 迁移到PostgreSQL |
| **BM25全量重建** | 导入变慢 | 论文>100篇 | 改用增量索引 |
| **无连接池** | 内存泄露 | 长时间运行 | 引入连接池 |
| **单点故障** | 服务不可用 | 任意外部API失效 | 多服务商备份 |

### 3.2 扩展性分析

**【代码事实】基于当前架构**:

| 维度 | 当前限制 | 理论上限 | 瓶颈点 |
|------|---------|---------|--------|
| **论文数量** | ~500篇 | 10000篇 | Qdrant检索延迟 |
| **并发用户** | ~10人 | 100人 | SQLite锁 |
| **QPS** | ~20 | 100 | Python GIL |
| **存储容量** | ~100GB | 1TB | 磁盘I/O |

### 3.3 扩展性改进方案

**1. Qdrant优化**:
```python
# 当前: 每篇论文一个collection
# 问题: 检索需要遍历所有collection
# 方案: 改为单一collection，用payload过滤

collection_name = "all_papers"
qdrant.search(
    collection_name=collection_name,
    query_vector=vector,
    query_filter=Filter(
        must=[FieldCondition(key="paper_id", match=MatchValue(value=paper_id))]
    )
)
```

**2. 数据库迁移**:
```python
# 当前: SQLite
# 问题: 并发性能差
# 方案: 迁移到PostgreSQL

# 优势:
# - 连接池
# - 更好的并发性能
# - 全文搜索支持（可替代BM25）
# - JSON字段支持（灵活schema）
```

**3. 缓存层**:
```python
# 引入Redis缓存:
# - Query改写结果缓存（24小时）
# - Embedding向量缓存（永久）
# - 检索结果缓存（1小时）

from redis import Redis
cache = Redis(host='localhost', port=6379)

async def retrieve_with_cache(query):
    cache_key = f"retrieve:{hash(query)}"
    cached = cache.get(cache_key)
    if cached:
        return json.loads(cached)
    results = await self.retrieve(query)
    cache.setex(cache_key, 3600, json.dumps(results))
    return results
```
