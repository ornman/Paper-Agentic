## 2. 模块边界与职责

### 2.1 模块划分

| 模块 | 职责 | 对外接口 | 依赖 |
|------|------|----------|------|
| **WPS 插件** | UI 渲染、轮询监听、用户交互 | HTTP/SSE → 后端 API | 后端 API |
| **API 路由层** | 请求路由、参数校验、响应格式化 | REST API (对外) | 业务逻辑层 |
| **数据清洗管道** | PDF 导入、解析、清洗、切块、向量化 | `/start` 触发 | MinerU, Zvec, Embedding |
| **检索问答管道** | 查询理解、多路召回、融合、生成 | `/query` 触发 | Zvec, BM25, Redis, LLM |
| **对话管理** | 对话历史 CRUD、滑动窗口管理 | 内部服务 | Redis |
| **存储层** | 向量存储（Zvec）、元数据（SQLite）、对话缓存（Redis） | 内部接口 | Zvec, SQLite, Redis |

### 2.2 依赖方向

```
WPS 插件 → API 路由层 → 业务逻辑层 → 存储层
                           ↕
                      外部服务（MinerU, LLM(DeepSeek等), Kimi VLM, Embedding）
```

**原则**: 单向依赖，上层不直接访问存储层，通过业务逻辑层中转。

## 3. 数据清洗管道

### 3.1 完整流程

```
用户输入: /start 'D:\papers\xxx.pdf'
  ↓
[1] MinerU API 解析
  ├─ 输出: Markdown（文本块来源）+ JSON（元数据索引）
  └─ 超时/失败 → 自动降级处理
  ↓
[2] 噪音过滤（正则）
  ├─ 过滤: 页码、页眉页脚、乱码、重复空行
  └─ JSON 元数据: 提取标题、作者、章节结构、图表引用
  ↓
[3] 图片语义理解
  ├─ OCR / VLM / 多模态模型 → 图片描述
  └─ 描述以标签形式附加到 Markdown 对应位置
  ↓
[4] 语义切块
  ├─ 默认: 按章节/段落边界切分（保留语义完整性）
  ├─ 超大块 (>32k): 均匀切分为 24k + 首尾重叠
  └─ 用户可选颗粒度（暴力/父子块/递归，见 3.2）
  ↓
[5] Embedding
  ├─ 硅基流动 Qwen3-Embedding-8B (1536维)
  └─ 批量向量化，并发控制
  ↓
[6] 入库
  ├─ Zvec: 单 Collection，payload 附带 paper_id 等元数据
  ├─ SQLite: 论文元数据（标题、作者、导入时间等）
  └─ 去重检查: 是否已存在相同论文（基于文件 hash）
```

### 3.2 多索引策略

| 策略 | 切分方式 | 优势 | 劣势 | 阶段 |
|------|----------|------|------|------|
| **暴力切分** | 每 chunk 500~1000 token | 速度快 | 效果一般，前端需提示用户 | MVP |
| **父子块（级联）** | 父块(大) + 子块(小) 关联 | 精准定位 + 保留上下文 | 实现复杂度中等 | v1.1 |
| **递归检索** | 多级跳转 | 最精确 | 计算量较大 | v1.2 |

### 3.3 可复用的旧代码

- **语义切块逻辑**: 按章节级切分，>32k 均分 + 首尾重叠缓冲（逻辑可直接迁移）
- **MinerU API 调用**: 请求/响应处理可复用
- **噪音过滤正则**: 基础规则可复用，需扩展 Markdown 场景

### 3.4 关键变更（vs 旧方案）

| 维度 | 旧方案 | 新方案 |
|------|--------|--------|
| 文本格式 | 只留 JSON，抛弃 Markdown | **Markdown 为主 + JSON 元数据** |
| 设计理由 | 默认 LLM 对 JSON 接收度更高 | LLM 训练偏好 Markdown，JSON 反而训练少 |
| 向量库 | Qdrant（服务式，多 Collection） | Zvec（嵌入式，单 Collection + Payload） |
| 图片描述 | VLM 描述后存入 JSON | VLM/OCR 描述以标签形式附加到 Markdown |

## 4. 检索问答管道

### 4.1 三层架构

```
┌─────────────────────────────────────────────────────┐
│                  Layer 1: Source Understanding       │
│  理解每个输入源的本质，生成不同粒度的表示             │
├─────────────────────────────────────────────────────┤
│                  Layer 2: Channel-specific Query     │
│  为 Dense/BM25 生成各自的查询表示                     │
├─────────────────────────────────────────────────────┤
│                  Layer 3: Fusion                     │
│  先 source 内融合，再跨 source 融合                  │
└─────────────────────────────────────────────────────┘
```

### 4.2 Layer 1: Source Understanding

**核心原则**: 不同输入源的本质不同，不能一概而论。

| 输入源 | 本质 | 处理方式 | 错误做法 |
|--------|------|----------|----------|
| `user_prompt` | 用户意图 | 提取主题、实体、约束、任务类型 | 直接当 query |
| `selection` (圈选) | 焦点/证据锚点 | 保留原文短语，提取概念，带邻近上下文 | 当普通 query |
| `draft` (已写内容) | 上下文约束/缺口检测 | 提取当前主题和**未覆盖点** | 直接检索（只会召回已写过的东西） |

**Intent 分类**（决定后续检索策略）:

```json
{
  "task": "ask | explain | continue | revise | compare",
  "need_external_retrieval": true
}
```

| task | 含义 | 检索侧重 |
|------|------|----------|
| `ask` | 用户提问 | 精准匹配答案 |
| `explain` | 解释某概念 | 概念定义、背景知识 |
| `continue` | 续写 | 找用户**还没写到的**相关内容 |
| `revise` | 修改/润色 | 找替代表述、更准确的引用 |
| `compare` | 对比分析 | 找不同观点、对立立场 |

**已写内容的特殊处理**（续写场景）:
- 不能直接拿已写内容去检索——只会召回重复内容
- 应该: 提取当前主题 → 检测已覆盖的子话题 → 生成"缺口查询"去找未覆盖的相关内容
- 例: 用户已写了 attention mechanism 的定义和优势 → 缺口查询应为 "attention mechanism 的局限性/挑战/最新改进"

### 4.3 Layer 2: Channel-specific Query Generation

**核心原则**: 不要做"一份改写，服务所有检索通道"；每个 source 生成多种 query 表示。

```
                    user_prompt: "Transformer在NLP中的应用"
                              ↓
                    ┌─────────┴─────────┐
                    │                   │
              Dense Query          BM25 Query
        "Transformer architecture     ["Transformer",
         applications in natural       "NLP",
         language processing           "text classification",
         text classification           "machine translation",
         machine translation"          "language model"]
                    │                   │
                    ↓                   ↓
              向量语义检索          关键词精确匹配
```

| 通道 | 输入偏好 | 生成方式 |
|------|----------|----------|
| **Dense** (Embedding) | 原始自然语言或轻量规范化 | 保留语义完整性，不要压成关键词 |
| **Sparse** (BM25) | 关键词、实体、精确短语 | 提取关键词 + 同义扩展 |
| **Filter** | 结构化条件 | Zvec payload 过滤（paper_id 等） |

**为什么不能共用**: 把自然语言压成关键词再做 Embedding，会丢失语义；把关键词喂给 BM25 做精确匹配，又不如直接提取的短语准确。

### 4.4 Layer 3: Fusion

**MVP（无 Reranker）的融合策略**:

```
每个 source 内:
  dense_results + sparse_results → source 内融合（RRF）
                                    ↓
                               per_source_cap (每个 source 最多 N 条)
                                    ↓
跨 source 融合:
  source_prompt (cap=10) + source_selection (cap=10) + source_draft (cap=5)
                                    ↓
                          去重（按 doc_id）
                                    ↓
                          per_doc_cap (每篇论文最多 M 条 chunk)
                                    ↓
                          source-aware fusion（按 weight_hint 加权）
                                    ↓
                          最终结果 Top-K
```

**关键参数**:

| 参数 | 含义 | 默认值 |
|------|------|--------|
| `per_source_cap` | 每个 source 最多召回条数 | 10 |
| `per_doc_cap` | 每篇论文最多出现条数 | 2 |
| `topk` | 最终返回条数 | 8~15 |
| `weight_hint` | source 权重（动态，非固定） | 由 intent + source 质量决定 |

**为什么不用固定 50/30/20**:
- 固定权重不稳定——一个高质量的 prompt 可能只需要 30% 就够，一个模糊的 prompt 可能需要 70%
- 应根据信号质量动态调整（如 prompt 长度、具体程度、包含的实体数等）

**v1.1 上 Reranker 后**:
- 融合后加一步 Reranker（硅基流动 Qwen3-Reranker-8B）
- Reranker 接收 fused results → 重排序 → 最终 Top-K

### 4.5 内部 Query Schema

```json
{
  "intent": {
    "task": "ask|explain|continue|revise|compare",
    "need_external_retrieval": true
  },
  "sources": [
    {
      "type": "prompt",
      "raw": "Transformer在NLP中的应用",
      "dense_query": "Transformer architecture applications in natural language processing",
      "sparse_query": ["Transformer", "NLP", "text classification", "machine translation", "language model"],
      "must_terms": ["Transformer"],
      "weight_hint": 0.6
    },
    {
      "type": "selection",
      "raw": "cross-attention mechanism in machine translation",
      "exact_span": "cross-attention mechanism",
      "dense_query": "cross-attention mechanism and its role in machine translation sequence models",
      "sparse_query": ["cross-attention", "machine translation", "sequence-to-sequence", "decoder attention"],
      "weight_hint": 0.3
    },
    {
      "type": "draft",
      "raw": "[用户已写的800字关于attention mechanism的综述]",
      "gap_query": "attention mechanism 的局限性和最新改进方法",
      "covered_topics": ["self-attention", "multi-head attention", "positional encoding"],
      "dense_query": "attention mechanism recent improvements challenges limitations beyond standard transformer",
      "sparse_query": ["attention mechanism", "improvement", "limitation", "efficient attention"],
      "weight_hint": 0.1
    }
  ],
  "fusion": {
    "method": "weighted_rrf",
    "per_source_cap": 10,
    "per_doc_cap": 2
  }
}
```

### 4.6 端到端流程

```
用户触发（提问/圈选/灵感按钮/发送）
  ↓
Layer 1: Source Understanding
  ├─ 识别 intent（task 类型）
  ├─ 分类输入源（prompt / selection / draft）
  └─ 每个源生成 dense_query + sparse_query
  ↓
Layer 2: Channel-specific Retrieval
  ├─ Dense: 每个 dense_query → Embedding → Zvec 向量检索
  ├─ Sparse: 每个 sparse_query → jieba 分词 → BM25 检索
  └─ Filter: payload 条件过滤（如 paper_id）
  ↓
Layer 3: Fusion
  ├─ Source 内融合: dense + sparse → RRF → per_source_cap
  ├─ 跨 source 融合: 按 weight_hint 加权 → 去重 → per_doc_cap
  └─ 输出: 最终 Top-K 结果（附带来源位置）
  ↓
拼装上下文:
  ├─ 召回的文档片段（带来源标注）
  ├─ 对话历史（Redis 滑动窗口）
  └─ 系统提示词（按 intent 路由）
  ↓
LLM（DeepSeek 等）→ 流式返回
  ↓
回答 + 来源标注（段落级，可点击跳转本地 PDF）
```

## 5. 对话管理

### 5.1 Redis 数据模型

**Key 设计**: `conversation:{session_id}`

**Value 格式**: Redis List，每个元素是一条消息

```json
{
  "role": "user | assistant | system",
  "content": "消息内容",
  "timestamp": 1713440000,
  "sources": [
    {
      "paper_id": "xxx",
      "title": "论文标题",
      "page": 12,
      "chunk_index": 3
    }
  ]
}
```

### 5.2 滑动窗口策略

| 参数 | 值 | 说明 |
|------|----|------|
| 窗口大小 | 模型上下文上限 | 当前 LLM 的上限（如 DeepSeek） |
| 保留策略 | 保留所有消息，直到接近上限 | 接近极限时强制截断最早的消息 |
| TTL | ~1 小时 | 自动清理过期对话 |
| system prompt | 始终保留，不纳入窗口计算 | 确保系统指令始终生效 |

### 5.3 上下文拼装

```
最终发给 LLM 的上下文:
┌────────────────────────┐
│ System Prompt (固定)    │ ← 按 intent 路由
├────────────────────────┤
│ 召回的文档片段          │ ← RAG 检索结果
├────────────────────────┤
│ 对话历史 (滑动窗口)     │ ← Redis
├────────────────────────┤
│ 当前用户输入            │ ← 本次提问
└────────────────────────┘
```

## 7. 存储层设计

### 7.1 Zvec Schema（单 Collection）

```python
schema = CollectionSchema(
    name="papers",
    fields=[
        FieldSchema("paper_id", DataType.STRING),       # 论文唯一 ID
        FieldSchema("file_hash", DataType.STRING),      # 文件 MD5（去重用）
        FieldSchema("chunk_type", DataType.STRING),     # chunk 类型: section/paragraph/sentence/image
        FieldSchema("chunk_index", DataType.INT32),     # 块序号
        FieldSchema("content", DataType.STRING),        # 文本内容（Markdown）
        FieldSchema("source_page", DataType.INT32),     # 来源页码
        FieldSchema("section_title", DataType.STRING),  # 所属章节标题
        FieldSchema("has_image", DataType.STRING),      # 是否包含图片标签: "true"/"false"
    ],
    vectors=VectorSchema("embedding", DataType.VECTOR_FP32, dimension=1536),
)
```

**索引策略**:
- `embedding` 字段: FLAT 索引（MVP，简单可靠）
- `paper_id` 字段: 倒排索引（加速 payload 过滤）
- v1.1 可切换 HNSW 提升大规模性能

**CRUD 通过 payload 过滤**:
```python
# 删除某篇论文的所有 chunk
collection.delete_by_filter("paper_id == 'xxx'")

# 只在某篇论文内检索
collection.query(
    vectors=VectorQuery("embedding", vector=query_vec),
    topk=10,
    filter="paper_id == 'xxx'"
)
```

### 7.2 SQLite（元数据）

```sql
CREATE TABLE papers (
    paper_id   TEXT PRIMARY KEY,
    title      TEXT,
    authors    TEXT,
    file_path  TEXT,
    file_hash  TEXT UNIQUE,     -- 去重
    chunk_count INTEGER,
    import_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    metadata_json TEXT           -- MinerU 输出的 JSON 元数据
);

CREATE TABLE import_logs (
    id         INTEGER PRIMARY KEY,
    paper_id   TEXT,
    status     TEXT,            -- pending/processing/completed/failed
    error_msg  TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```
