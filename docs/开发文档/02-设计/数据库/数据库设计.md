## 1. 存储架构

本地单用户场景，三个存储各司其职：

| 存储 | 技术 | 用途 | 生命周期 |
|------|------|------|----------|
| 向量数据 | Zvec（单 Collection） | 文档向量 + payload 元数据 | 持久化（磁盘） |
| 结构化元数据 | SQLite | 论文信息、导入日志、配置 | 持久化（磁盘） |
| 对话缓存 | Redis | 对话历史、轮询缓存 | 临时（TTL ~1h） |

**并发模型**: 单用户，几乎不会同时读写。要么只读（问答），要么只写（导入）。不需要连接池、事务隔离等复杂设计。

## 2. SQLite 表结构

### 2.1 papers — 论文元数据

```sql
CREATE TABLE papers (
    paper_id    TEXT PRIMARY KEY,          -- UUID
    title       TEXT NOT NULL DEFAULT '',   -- 论文标题（从 MinerU JSON 提取）
    authors     TEXT NOT NULL DEFAULT '',   -- 作者
    file_path   TEXT NOT NULL,              -- 本地 PDF 路径
    file_hash   TEXT NOT NULL UNIQUE,       -- 文件 MD5（去重）
    file_size   INTEGER,                    -- 文件大小（字节）
    chunk_count INTEGER NOT NULL DEFAULT 0, -- 向量块数量
    total_pages INTEGER,                    -- 总页数
    import_time TEXT NOT NULL,              -- 导入时间 ISO8601
    status      TEXT NOT NULL DEFAULT 'completed', -- completed / deleted
    metadata_json TEXT                       -- MinerU 输出的原始 JSON 元数据
);

CREATE INDEX idx_papers_file_hash ON papers(file_hash);
CREATE INDEX idx_papers_import_time ON papers(import_time);
```

### 2.2 import_logs — 导入日志

```sql
CREATE TABLE import_logs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id     TEXT NOT NULL UNIQUE,       -- 导入任务 ID
    paper_id    TEXT,                        -- 关联论文（完成后填入）
    file_path   TEXT NOT NULL,
    status      TEXT NOT NULL DEFAULT 'pending', -- pending/processing/completed/failed
    current_step TEXT,                       -- 当前步骤: parse/clean/chunk/embedding/store
    error_msg   TEXT,                        -- 失败原因
    created_at  TEXT NOT NULL,
    updated_at  TEXT NOT NULL
);

CREATE INDEX idx_import_logs_task_id ON import_logs(task_id);
CREATE INDEX idx_import_logs_status ON import_logs(status);
```

### 2.3 chunks — 向量块索引（可选，SQLite 侧的冗余索引）

> 是否需要这张表取决于查询需求。如果只需要通过 Zvec payload 过滤就能满足，可以不建。

```sql
CREATE TABLE chunks (
    chunk_id     TEXT PRIMARY KEY,           -- {paper_id}_{chunk_index}
    paper_id     TEXT NOT NULL,
    chunk_index  INTEGER NOT NULL,           -- 块序号
    chunk_type   TEXT NOT NULL DEFAULT 'paragraph', -- section/paragraph/sentence/image
    section_title TEXT,                       -- 所属章节
    source_page  INTEGER,                    -- 来源页码
    content_hash TEXT,                        -- 内容 MD5（去重）
    FOREIGN KEY (paper_id) REFERENCES papers(paper_id) ON DELETE CASCADE
);

CREATE INDEX idx_chunks_paper_id ON chunks(paper_id);
```

**说明**: Zvec 的 payload 已经包含这些字段。SQLite 侧建表是为了支持"查看某篇论文的所有块"、"全文搜索"等 Zvec 不擅长的操作。如果 MVP 阶段用不到，可以先不建。

### 2.4 config — 用户配置

```sql
CREATE TABLE config (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL,
    updated_at TEXT NOT NULL
);

-- 默认配置
INSERT INTO config VALUES ('dense_topk', '15', datetime('now'));
INSERT INTO config VALUES ('sparse_topk', '15', datetime('now'));
INSERT INTO config VALUES ('fusion_topk', '15', datetime('now'));
INSERT INTO config VALUES ('per_source_cap', '10', datetime('now'));
INSERT INTO config VALUES ('per_doc_cap', '2', datetime('now'));
INSERT INTO config VALUES ('chunk_max_context', '32000', datetime('now'));
INSERT INTO config VALUES ('chunk_target_size', '24000', datetime('now'));
INSERT INTO config VALUES ('chunk_overlap_buffer', '8000', datetime('now'));
```

**设计理由**: 检索/切分参数可动态调整，不需要改 .env 重启。.env 只放 Key 和连接信息，业务参数放 SQLite。

## 3. Zvec Collection Schema

单 Collection，所有论文的向量存在一起，通过 payload 区分。

```python
schema = CollectionSchema(
    name="papers",
    fields=[
        FieldSchema("paper_id", DataType.STRING),       # 关联 papers 表
        FieldSchema("file_hash", DataType.STRING),      # 关联去重
        FieldSchema("chunk_type", DataType.STRING),     # section / paragraph / sentence / image
        FieldSchema("chunk_index", DataType.INT32),     # 块序号
        FieldSchema("content", DataType.STRING),        # 文本内容（Markdown）
        FieldSchema("source_page", DataType.INT32),     # 来源页码
        FieldSchema("section_title", DataType.STRING),  # 所属章节
        FieldSchema("has_image", DataType.STRING),      # "true" / "false"
    ],
    vectors=VectorSchema("embedding", DataType.VECTOR_FP32, dimension=1536),
)

# 索引
# embedding: FLAT（MVP，简单可靠）
# paper_id: 倒排索引（加速过滤）
```

### 3.1 CRUD 操作

```python
# 插入
collection.insert([
    Doc(id=f"{paper_id}_{i}",
        vectors={"embedding": vec},
        fields={
            "paper_id": paper_id,
            "chunk_type": "paragraph",
            "chunk_index": i,
            "content": markdown_text,
            "source_page": 5,
            "section_title": "Introduction",
            "has_image": "false"
        })
    for i, (vec, markdown_text) in enumerate(chunks)
])

# 按论文检索
results = collection.query(
    vectors=VectorQuery("embedding", vector=query_vec),
    topk=15,
    filter=f"paper_id == '{paper_id}'"
)

# 全库检索（不过滤）
results = collection.query(
    vectors=VectorQuery("embedding", vector=query_vec),
    topk=15
)

# 删除某篇论文的所有向量
collection.delete_by_filter(f"paper_id == '{paper_id}'")

# 单块删除
collection.delete(f"{paper_id}_{chunk_index}")
```

### 3.2 数据量估算

| 论文数 | 平均块数/篇 | 总块数 | 向量存储 | payload 存储 |

| 100 | 30 | 3,000 | ~18MB | ~5MB |
| 500 | 30 | 15,000 | ~90MB | ~25MB |
| 2,000 | 30 | 60,000 | ~360MB | ~100MB |
| 5,000 | 30 | 150,000 | ~900MB | ~250MB |

## 4. Redis 数据结构

### 4.1 对话历史

```
Key:    conversation:{session_id}
Type:   LIST
Value:  JSON 序列化的消息对象
TTL:    3600s（1 小时）
```

```json
// 单条消息
{
  "role": "user",
  "content": "Transformer在NLP中的应用",
  "timestamp": 1713440000,
  "sources": null
}

// assistant 消息附带来源
{
  "role": "assistant",
  "content": "Transformer在自然语言处理...",
  "timestamp": 1713440010,
  "sources": [
    {"paper_id": "xxx", "title": "...", "page": 3, "section": "3.2"}
  ]
}
```

### 4.2 轮询缓存

```
Key:    poll:{session_id}
Type:   STRING
Value:  用户当前 WPS 文档的文本内容
TTL:    3600s（WPS 关闭后自动清理）
```

### 4.3 滑动窗口管理

```python
# 写入消息
redis.rpush(f"conversation:{session_id}", json.dumps(message))
redis.expire(f"conversation:{session_id}", 3600)

# 读取全部历史（滑动窗口）
messages = redis.lrange(f"conversation:{session_id}", 0, -1)

# 接近模型上下文上限时，截断最早的消息
total_tokens = estimate_tokens(messages)
if total_tokens > MODEL_CONTEXT_LIMIT * 0.9:
    # 删除最早的消息直到安全
    while total_tokens > MODEL_CONTEXT_LIMIT * 0.8:
        redis.lpop(f"conversation:{session_id}")
        messages = messages[1:]
        total_tokens = estimate_tokens(messages)
```

## 5. 数据一致性

### 5.1 导入流程的原子性

```
SQLite BEGIN:
  INSERT papers → paper_id
  INSERT import_logs → task_id
SQLite COMMIT

→ 异步处理（MinerU → 切块 → Embedding → Zvec insert）

SQLite BEGIN:
  UPDATE papers SET chunk_count = N, status = 'completed'
  UPDATE import_logs SET status = 'completed'
SQLite COMMIT
```

**失败处理**: 如果异步处理中途失败，import_logs 记录失败状态和 error_msg。papers 表中该论文 status 可能为空或 processing。下次启动时可清理孤儿数据。

### 5.2 删除流程

```
1. Zvec: delete_by_filter("paper_id == 'xxx'")
2. SQLite: DELETE FROM papers WHERE paper_id = 'xxx'（CASCADE 删 chunks）
3. SQLite: DELETE FROM import_logs WHERE paper_id = 'xxx'
```

单用户无并发，不需要事务包裹 Zvec + SQLite。顺序执行即可。

### 5.3 去重

导入时计算 file_hash（MD5），插入前查询 `papers.file_hash`。已存在则拒绝导入并返回已有 paper_id。

## 6. 备份与恢复

单用户本地项目，简单粗暴：

- **SQLite**: 直接复制 `.db` 文件
- **Zvec**: 直接复制 Collection 目录
- **Redis**: 不需要备份（临时数据，TTL 自动清理）

恢复时把文件放回原位即可。
