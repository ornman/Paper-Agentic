# 踩坑日志 - Chromadb 重构与容灾降级

## 1. zvec → Chromadb 迁移

### 问题
- **Windows RocksDB 锁文件问题**：zvec 使用 RocksDB 作为底层存储，在 Windows 上进程异常退出时锁文件无法释放，导致数据无法访问，需要手动删除 `.lock` 文件
- **频繁数据丢失**：每次进程崩溃都可能导致索引损坏，需要重建

### 解决方案
- **Chromadb 替换 zvec**：Chromadb 使用纯 Python 实现，基于 SQLite 持久化，无 C++ 依赖，锁机制更稳定
- **API 兼容层**：`get_zvec = get_chroma` 别名，确保现有代码无需修改
- **数据迁移**：366 篇论文重新导入，生成 7812 个 chunks

### 经验
- 选择向量库时，优先考虑**锁机制稳定**的实现
- Windows 环境下，避免使用依赖 C++ 库（如 RocksDB）的组件
- 持久化使用 **SQLite** 而非专用存储引擎，更可靠

---

## 2. Redis 移除

### 问题
- Redis 作为额外依赖，增加部署复杂度
- 对话历史存储无需高性能缓存，SQLite 足够

### 解决方案
- **对话历史迁移到 SQLite**：`conversations` 表存储所有对话
- **轮询缓存改为 in-memory dict**：`_poll_cache` 字典存储短期任务状态
- **删除 Redis 依赖**：pyproject.toml 移除 `redis`，`health_routes.py` 移除 Redis 健康检查

### 经验
- **够用原则**：不是所有场景都需要 Redis
- SQLite 的事务和 ACID 特性对小规模数据完全够用
- 减少外部依赖 = 减少故障点

---

## 3. API 速率限制与并发优化

### 问题
- 原配置过于保守：MinerU 5 req/s（远超 50/min 限制），Embedding 并发 5（远低于 RPM 2000）
- 导入 376 篇论文耗时过长

### 解决方案
- **MinerU 双限速器**：
  - 提交限速：50/min ≈ 0.83/s（`RateLimiter(rate=0.83, burst=5)`）
  - 轮询限速：1000/min ≈ 16.7/s（`RateLimiter(rate=16.7, burst=50)`）
- **Embedding 并发提升**：从 5 提升到 20，充分利用 RPM 2000
- **批量导入并发**：从 3 提升到 10

### 经验
- **针对 API 限制优化**：查阅官方文档，获取精确的速率限制
- **分离不同操作的限速器**：提交和轮询是不同的操作，应有不同限速
- **并发与速率的平衡**：太高会触发 429，太低浪费资源

## 4. 错误处理与重试策略

### 问题
- 所有失败统一重试 3 次，无区分可恢复/不可恢复错误
- 重试退避固定 `2^attempt` 秒，未考虑 `Retry-After` 头
- VLM 失败导致整篇论文导入失败

### 解决方案
- **统一错误处理模块**（`error_handler.py`）：
  - `is_retryable(error)`：判断是否可重试（429、502、503、504、超时）
  - `get_backoff(attempt, error)`：优先读取 `Retry-After` 头，否则指数退避+随机抖动
- **VLM 降级**：单图描述失败降级为占位符，不影响整篇论文
- **阶段降级**：VLM 阶段失败标记为 `skipped`，继续清洗、切块

### 经验
- **区分错误类型**：网络错误可重试，业务错误（如 400）不应重试
- **读取 Retry-After**：服务器返回的 `Retry-After` 头优先级最高
- **降级优于失败**：非关键阶段失败应降级，而非终止整条流程

## 5. 备份管理与断点续传

### 问题
- MinerU 解析后无统一备份，后续流程断掉需从头开始
- 大文件解析耗时长，中断后重跑浪费时间

### 解决方案
- **备份管理器**（`backup_manager.py`）：
  - 备份目录：`./data/backups/{file_hash}/`
  - 阶段数据持久化：`full.md`、`full_with_desc.md`、`chunks.json`、`vectors.json`
  - 元数据：`backup.json` 记录每阶段状态
- **断点续传**：检查 `stages.{stage} != "completed"` 再执行
- **PDF 复制备份**：原始 PDF 复制到备份目录，确保引用路径始终可访问

### 经验
- **每阶段持久化**：长流程必须分段保存中间结果
- **幂等性设计**：重复执行同一阶段应产生相同结果
- **备份即文档**：`backup.json` 本身就是最佳的状态文档

## 6. PDF 引用跳转

### 问题
- 浏览器无法直接打开本地文件路径（`file://` 协议被阻止）
- WPS 插件环境下，需要调用 WPS JS API 打开 PDF

### 解决方案
- **PDF 打开 API**：`GET /{paper_id}/open` 返回 `FileResponse`
- **前端 WPS 集成**：
  ```javascript
  const wps = globalThis.wps || globalThis.Window?.wps
  if (wps) {
    wps.WpsApplication().Documents.Open(url)
  }
  ```
- **备份目录作为 PDF 源**：确保文件始终可访问

### 经验
- **Web 环境限制**：浏览器禁止 `file://` 协议，必须通过 HTTP API
- **WPS API 文档缺失**：需要通过调试工具探索可用的 API
- **引用路径持久化**：备份目录比原始路径更可靠

## 7. SSE 流式响应解析

### 问题
- 测试脚本最初无法正确解析 SSE 流，导致答案和来源都为空
- SSE 格式：`event: xxx\ndata: {...}\n\n`

### 解决方案
- **正确解析 SSE**：
  - 按 `\n\n` 分割事件
  - 提取 `event:` 和 `data:` 行
  - 使用 `json.loads` 解析数据
  - 累加 `chunk` 事件构建完整答案
  - 从 `metadata` 事件获取来源列表

### 经验
- **SSE 是流式协议**：需要处理缓冲、分割、重组
- **事件驱动**：不同 `event` 类型有不同处理逻辑
- **测试先行**：用 `curl` 先验证 SSE 格式，再写解析代码

## 8. Embedding 模型切换

### 问题
- 原使用 `Qwen/Qwen3-Embedding-8B`，需要切换到 `Qwen/Qwen3-Embedding-4B`
- 两种模型向量维度相同（1536），但语义空间不同

### 解决方案
- **清空旧数据**：删除 ChromaDB 和 BM25 索引
- **更新配置**：`.env` 中 `EMBEDDING_MODEL=Qwen/Qwen3-Embedding-4B`
- **全量重导**：366 篇论文重新生成向量

### 经验
- **向量索引与模型强绑定**：更换模型必须重建索引
- **8B vs 4B**：4B 模型更快，可能精度略低，但对大多数场景够用
- **批量导入前验证**：先用 2 篇论文测试全流程，再批量处理

## 性能对比结果

### 测试条件
- 10 个问题，同时测试 RAG 和纯 LLM
- 366 篇论文，7812 个 chunks
- Embedding：Qwen/Qwen3-Embedding-4B（1536 维）

### 结果
| 指标 | RAG | 纯 LLM | 提升 |
|------|-----|--------|------|
| 平均响应时间 | 25.1s | 35.0s | +28% |
| 来源数量 | 10/问 | 0 | — |
| 答案质量 | 详细，有引用 | 泛化，无引用 | — |

### 结论
- **Chromadb 性能良好**：RAG 检索不影响响应速度
- **向量质量可靠**：检索结果相关性高
- **4B 模型够用**：在学术文献问答场景下表现良好
