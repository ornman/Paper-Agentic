## 2026-04-23: 三阶段容灾重构

### 背景

旧版批量导入 `batch_import.py` 是单线程管道，存在严重容灾问题：
1. **无中间状态持久化**：所有处理在内存中，进程崩溃 → 全部丢失
2. **Zvec LOCK 导致数据全删**：`shutil.rmtree` 删除整个向量库，7000+ 向量全丢
3. **单点失败阻断全部**：1 篇论文失败 → 异常传播到 `asyncio.gather`
4. **无法断点续传**：每次都从零开始，浪费 MinerU API 调用

### 完成的工作

#### 三阶段架构

```
阶段1: 解析 (MinerU + VLM + 清洗) → data/parsed/{paper_id}.json
阶段2: 切块 (chunker)               → data/chunks/{paper_id}.json
阶段3: 向量化 (Embedding + 存储)    → Zvec + BM25 + SQLite
```

每阶段输出独立 JSON 文件，下游失败不影响上游结果。

#### 新建文件

## 2026-04-23: 清洗器兼容性修复

### 背景

三阶段容灾测试发现 3 篇论文（6%）因"清洗后为空"失败。经过分析，真正原因是**清洗器的格式兼容性问题**，而非 MinerU 解析失败。

### 问题分析

**失败论文特征**：
- 期刊论文（非学位论文）
- 摘要格式：`【摘要】：`（中文括号，而非 `# 摘要`）
- 无标准章节标题（直接标题+段落）
- 内容较短（2-3 页，2740-4587 字符）

**清洗器缺陷**（`app/pipelines/ingestion/cleaner.py`）：
1. 无法识别 `【摘要】：` 格式（只支持 `# 摘要` 或 `摘要：`）
2. 依赖严格章节标题（`第1章`、`Chapter 1`），期刊论文不用
3. 无标准格式时，整个文档被标记为 `cover_toc`，不提取内容

### 修复内容

**文件**: `app/pipelines/ingestion/cleaner.py`

**1. 扩展摘要检测（Line 178-191）**
```python
# 新增中文括号格式支持
if re.match(r"^【摘\s*要】", stripped):
    body_start = idx
    break
```

**2. 添加智能降级（Line 193-198）**
```python
# 找不到标准格式时，把整个文档视为 body
if body_start == n and len(lines) > 5:
    logger.debug("清洗器降级: 无法识别标准格式，全文档处理")
```

#### 反直觉的发现：为什么 RAG 更快？

**传统认知**：RAG = 检索时间 + 生成时间，应该比 Pure LLM 慢

**实际真相**：
```
Pure LLM = 生成大量"幻觉内容"（编造数据、政策文件）
         = 更多 tokens (2568 vs 955, +169%)
         = 更长生成时间 (114s vs 34.8s, +228%)
         = 更慢

RAG = 只基于检索到的真实文献回答
     = 更少 tokens（精准回答，不编造）
     = 更快（尽管有检索时间）
     = 更好
```

**核心结论**：
1. **速度优势**: Pure LLM 产生幻觉 → 生成更多内容 → 更慢
2. **准确性**: RAG 基于真实文献，Pure LLM 包含幻觉
3. **可溯源**: RAG 提供来源引用，Pure LLM 无法验证
4. **效率**: RAG Token 使用少 41-169%

| 内容 | 3 种模式（基于文献） | 9 种模式（含幻觉） |

**RAG**：承认"具体案例和数据的引用较为有限"，诚实基于文献回答
**Pure LLM**：编造 5 种城市模式 + 4 种农村模式，包含大量无法验证的内容
