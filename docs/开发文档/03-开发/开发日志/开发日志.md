## 2026-04-19: 批量导入 + 清洗器增强

### 完成的工作

#### 清洗器格式匹配增强

初始版清洗器只匹配 `# 摘要`、`第1章`、`# 1. Introduction` 三种格式，导致大部分中文论文被判定为"空内容"。

新增匹配:
- **行内摘要**: `摘 要：与西欧城乡分立的传统不同...`（非标题格式）
- **中文数字章节**: `# 一、被遗忘的"治理"`、`# （二）城乡共同体的经济偏向`
- **数字编号章节**: `# 1 公共数字文化服务营销研究进展`

提取 `_find_first_chapter()` 为独立函数，集中管理章节检测正则。

#### MinerU data_id 字节截断

MinerU API `data_id` 字段限制 128 **字节**（非字符数），中文字符 UTF-8 编码每字 3 字节。
45 个中文字 ≈ 135 字节，超过限制。改为按字节截断到 120 字节。

#### 批量导入脚本优化

`scripts/batch_import.py` 改进:
- **始终去重**: 不管 `--skip-existing` 参数，都先检查 file_hash 避免浪费 API
- **自动重试**: 失败后按 `attempt × 3s` 间隔重试（默认 3 次），仅对可重试错误（超时/连接/429/5xx）
- **代码结构**: 提取 `_do_import()` 为独立函数，`import_one()` 负责重试编排
- **默认并发 10**: 匹配 MinerU 50/min 限流

| 指标 | 数值 |
|------|------|
| 总 PDF | 376 篇 |
| 成功导入 | **347 篇** |
| 总 chunks | **7,779** |
| 平均 chunks/篇 | **21.8** |
| 空内容（清洗后无有效文本） | 19 篇 |
| MinerU 服务端失败 | 3 篇 |
| 耗时 | ~30 分钟 |

### 踩过的坑

| 坑 | 原因 | 解决 |
|----|------|------|
| 清洗器返回 0 chunks | 只匹配 `# 摘要`/`第1章` 格式，漏掉 `一、`/`摘 要：` | 扩展正则匹配 |
| MinerU data_id 超限 | 按 Unicode 字符数限制，但 API 按 UTF-8 字节数限制 | 改为字节截断 |
| SQLite UNIQUE 冲突 | 去重检查只在 `--skip-existing` 时执行 | 始终检查 |
| Zvec LOCK 文件残留 | 前次进程未正常关闭 | `rm -f LOCK` 后重启 |

## 2026-04-22: RAG 端到端验证 + 混合检索

### 完成的工作

#### P3: RAG 问答验证（完成）

**修复 Token 估算错误**:
- `retrieval/service.py` 第 82 行 `len(content)` 对中文低估约 1.5 倍
- 替换为 `chunker.py` 的 `estimate_tokens()`（中文 1.5，其他 0.75）
- 修复前：中文内容实际 token 可能远超 30000 上限

**实现 Dense + BM25 混合检索 + RRF 融合**:
- 新增 `_rrf_fuse()` 函数，融合 Zvec Dense 检索 + BM25 关键词检索
- RRF 公式：`score(d) = Σ 1/(k + rank_i(d))`，k=60
- Dense topk=20 + BM25 topk=20 → 融合后取 topk=10
- BM25 失败时自动降级为纯 Dense 检索

**增强错误处理**:
- Embedding 失败 → 返回"向量化服务不可用"
- BM25 失败 → 降级为纯 Dense（日志 warning）
- LLM 流式失败 → 返回"LLM 服务不可用"
- 对话历史保存失败 → 静默跳过
- 关键节点添加 info/warning/error 级别日志

## 2026-04-22: LLM 从 DeepSeek 换成 Kimi K2.6

### 完成的工作

#### LLM 客户端替换

**从 DeepSeek 换成 Kimi Coding API**:
- 移除 DeepSeek 配置（`llm_base_url`, `llm_api_key`, `llm_model`）
- 改用 Kimi K2.6-code-preview（256K 上下文窗口）
- 必须带的 headers：
  - `Authorization: Bearer {KIMI_API_KEY}`
  - `anthropic-version: 2023-06-01`
  - `User-Agent: claude-code`

**实现方式**:
- 从 AsyncOpenAI 改为 httpx.AsyncClient（因为需要特殊 headers）
- Kimi SSE 格式：`data: {"content": [{"type": "text", "text": "..."}]}`
- 不返回 token 统计，用估算值替代（中文 1.5 字/token）

#### 配置更新

**`.env` 更新**:
- 移除 `LLM_API_KEY`, `LLM_BASE_URL`, `LLM_MODEL`
- 只保留 `KIMI_API_KEY`

**`.env.example` 更新**:
- 清理 LLM 配置部分
- 添加 Kimi 必需 headers 注释

**测试脚本更新**:
- `compare_rag_llm.py` 改用 LLMClient
- 移除 AsyncOpenAI 依赖
- Token 统计改为估算值

### 踩过的坑

| 坑 | 原因 | 解决 |
|----|------|------|
| OpenAI SDK 不支持自定义 headers | Kimi 需要 `anthropic-version` 等特殊 headers | 改用 httpx.AsyncClient |
| Kimi 不返回 token 统计 | API 响应格式不包含 usage | 用估算值（中文 1.5 字/token） |

### 上下文窗口对比

| 模型 | 上下文窗口 | 变化 |
|------|----------|------|
| DeepSeek | 1M tokens | → |
| Kimi K2.6 | 256K tokens | 减少 75% |
| 当前代码限制 | 30K tokens | 不变（远低于两者限制） |

## 2026-04-22: 批量导入管道优化

### 完成的工作

#### 官方限流策略验证

查阅 [MinerU API 官方限流文档](https://mineru.net/apiManage/limit)：

| 接口类型 | 限制 |
|---------|------|
| 提交任务接口 | **300 请求/分钟** ≈ 5 请求/秒 |
| 获取任务结果 | 1000 请求/分钟 |
| 每日上传上限 | 10,000 个文件/天 |

之前代码注释中的 "50/min" 是过时信息。

#### 令牌桶速率限制器

新建 `app/utils/rate_limiter.py`：
- 令牌桶算法，支持 `async with` 上下文
- 可配置速率和突发上限

#### MinerU 客户端增强

`mineru_client.py` 改进：
- `submit_task()` 添加 429 错误处理
- 读取 `Retry-After` 响应头，等待后重试
- 全局速率限制器 `RateLimiter(5.0)` 匹配官方 300/分钟限制
- 提取 `_do_submit()` 为独立函数

#### batch_import.py 优化

- 默认并发从 10 降到 5（匹配 MinerU 限流）
- 新增 `--vlm-concurrency` 参数（默认 2）
- 失败日志按错误类型分类（429限流/超时/连接错误/服务端错误）

#### image_handler.py 并发调整

- 默认并发从 10 降到 2

### 踩过的坑

| 坑 | 原因 | 解决 |
|----|------|------|
| 并发 10 导致 429 | MinerU 限制 300/min，瞬时请求过多 | 添加 RateLimiter + 降低并发到 5 |
| MinerU 无 429 处理 | 依赖上层重试 | 客户端内置 429 重试 + Retry-After |
| UNIQUE constraint 失败 | force-rebuild 模式 INSERT 与已有 file_hash 冲突 | 改用 INSERT OR REPLACE |
