# PRD - 论文助手（学术写作 RAG 助手）

> 版本: v2.0（重构版）
> 更新日期: 2026-04-18

---

## 1. 产品定位

**一句话描述**: 面向学术研究者的 RAG + LLM 知识问答助手，以 WPS/Word 插件形式嵌入主流写作场景，提供实时、基于个人文献库的智能辅助。

**产品形态**: WPS Office 插件（Copilot 风格侧边栏 / 悬浮窗）

**核心公式**: 用户私有文献库 + RAG 检索增强 + LLM 生成 = 写作场景内的即时知识问答

---

## 2. 目标用户

### 2.1 用户画像

| 画像 | 场景特征 | 核心需求 |
|------|----------|----------|
| **硕博研究生 / 本科毕业生** | 高频进行学术论文写作，积累大量细分领域参考文献，需要频繁引用文献支撑论点，关注写作效率与引用规范性 | 快速匹配对应论点的文献内容、一键生成规范引用、降低文献反复查阅成本 |
| **高校 / 科研院所科研工作者** | 长期从事学术研究与论文撰写，有完善的个人文献积累，对内容严谨性、溯源准确性要求极高 | 私有知识库高效管理、写作中实时论据检索、观点精准溯源、内容相似度比对 |
| **学术撰稿人** | 需基于大量文献完成内容创作，关注内容产出效率与来源可追溯性 | 批量文献管理、快速语义匹配、多文献内容整合检索 |

### 2.2 共同需求

- **论文参考利用**: 已有大量 PDF 参考文献但利用率低，需要高效检索和利用
- **知识沉淀积累**: 读过的文献形成体系化沉淀，而非散落各处
- **工作场景**: 主要工作环境是 WPS Office / Microsoft Word
- **数据隐私**: 论文未发表内容不应泄露，文献数据需本地存储

---

## 3. 核心痛点与解决方案

### 3.1 痛点一：已有参考资料，却无法充分利用

**现状**: 用户收藏了数十甚至数百篇 PDF，但写作时只能凭记忆引用，搜索效率极低。

**解决方案**:
- PDF 一键导入，自动解析为结构化文本 + 元数据
- 基于向量检索的语义搜索，支持跨论文检索
- 选中文字即可"查找相关文献片段"
- 检索结果附带来源标注（哪篇论文、哪个章节）

### 3.2 痛点二：实时创作需求的即时反馈闭环

**现状**: 写作过程中遇到疑问需要切换到浏览器搜索，打断思路。

**解决方案**:
- 侧边栏常驻，不离开写作环境
- 支持选中文字直接提问（"这段引用的出处？"）
- 支持实时读取用户当前写作内容，主动提供建议
- 流式返回，减少等待感

### 3.3 痛点三：碰撞为主，思想风暴

**现状**: 文献之间可能存在观点碰撞或互补关系，但靠人工难以发现。

**解决方案**:
- 跨论文对比提问（"A 论文和 B 论文在 X 观点上有什么分歧？"）
- "探索模式"：基于当前话题自动推荐相关文献和观点
- 对话式头脑风暴，LLM 基于文献库内容参与讨论
- 知识图谱可视化（v2.0）：概念关系、引用链条

### 3.4 痛点四：知识沉淀累积

**现状**: 读过的文献时间久了就忘，笔记散落各处，无法形成体系。

**解决方案**:
- 所有问答记录自动关联到对应文献
- 支持"文献卡片"：每篇论文自动生成核心观点摘要
- 知识图谱随使用自动生长（v2.0）
- 跨会话记忆（v2.0）：上次讨论过的话题可以延续

```
│  │  Query → 检索 → 拼上下文 → LLM → 流式返回    │ │
```

### 5.2 技术选型

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| 前端框架 | Vue 3 + TypeScript | 轻量、WPS 插件兼容 |
| 前端架构 | MVC 模式 | 从简，功能优先 |
| 后端框架 | FastAPI (Python 3.13) | 异步、类型安全 |
| LLM 框架 | MVP: 直线 RAG / v1.1+: LangGraph | 先跑通再引入 Agent 模式 |
| LLM 对话 | DeepSeek（用户自填 URL + Key） | 通用对话问答，可随时切换任意 LLM 厂商 |
| VLM 图片理解 | Kimi Coding API (K2.6) | PDF 图片语义理解、描述打标签 |
| 向量库 | Zvec (阿里巴巴) | 嵌入式、零配置、单 Collection + Payload 过滤 |
| 关键词检索 | BM25 + jieba | 轻量高效，中文分词优秀，Hybrid 检索 |
| 对话缓存 | Redis (redis-py) | 滑动窗口式记忆，保留对话历史 |
| 知识图谱 | SQLite + NetworkX → FalkorDB → Neo4j | v2.0，MVP 不实现 |
| Embedding | 硅基流动 Qwen3-Embedding-8B (1536维) | 沿用，成熟稳定 |
| PDF 解析 | MinerU API | 沿用，输出 Markdown + JSON 双格式 |

### 5.3 关键技术约束

**LLM 对话服务（DeepSeek 等，用户自配）**:
- 用户自行填写 LLM 的 URL 和 API Key
- 通过 LangChain 抽象层，支持随时切换任意 OpenAI 兼容 API
- MVP: DeepSeek → 后续可换 ChatGLM、Qwen、OpenAI 等

**Kimi Coding API 调用要求（仅用于 VLM 图片理解）**:
- Base URL: `https://api.kimi.com/coding/v1/messages`
- 必须携带 3 个 Header:
  - 自定义认证头 + API Key
  - `anthropic-version: 2023-06-01`
  - `User-Agent: claude-code`
- 缺少任何一个都会被服务端拒绝
- 模型: K2.6-code-preview
- 用途: PDF 图片语义理解 → 自然语言描述 → 打标签

**Zvec 向量库特性**:
- 嵌入式架构（类 SQLite），无需独立进程
- **单 Collection + Payload 元数据过滤**，替代之前"每篇论文一个 Collection"的分布式方案
- 通过 payload 字段（如 paper_id、chunk_type 等）实现 CRUD 和精准过滤
- 优势：最大化性能，即使在低配电脑上也能流畅运行；管理开销极低
- 2000 篇论文（~5万向量）仅需 1GB 内存
- 8GB 内存可支撑 2万篇论文

**图谱演进路线**（v2.0，MVP 不实现）:
```
中期: SQLite + NetworkX（本地、轻量）
  ↓
远期: LadybugDB / FalkorDB（更专业的图能力）
  ↓
商业化: Neo4j（企业级）
```

**PDF 解析与数据流**:
```
PDF → MinerU API 解析 → 输出双格式:
  ├─ JSON → 元数据索引（标题、作者、章节结构、图表引用等）
  └─ Markdown → 文本块来源
       ↓
  噪音过滤（页码、页眉页脚、乱码等）
       ↓
  图片处理: OCR / VLM / 多模态模型 → 语义描述 → 以标签形式附加到对应位置
       ↓
  按语义切分（保留段落/章节边界）
       ↓
  硅基流动 Embedding (Qwen3-Embedding-8B, 1536维)
       ↓
  Zvec 单 Collection 存储（payload 附带 paper_id、chunk_type 等元数据）
```

**Markdown + JSON 双格式设计决策**:
- **旧方案**：只清洗 JSON，完全抛弃 Markdown — 过于激进
- **决策依据**：LLM 训练语料中 Markdown 占比远高于 JSON，模型对 Markdown 的理解和生成能力更强
- **新方案**：Markdown 用于文本块切分和 LLM 输入，JSON 用于结构化元数据索引
- **各自所长**：Markdown 擅长语义表达，JSON 擅长结构化检索

**图片语义理解**:
- 使用 OCR / VLM / 多模态模型对 PDF 中的图片进行语义理解
- 生成描述后以标签形式附加到对应文本位置
- 确保图表、公式等视觉内容的语义不会在向量化过程中丢失

## 6. 非功能需求

### 6.1 性能

| 指标 | 目标 |
|------|------|
| PDF 解析（单篇） | < 30s（取决于 MinerU / 解析服务） |
| 向量化（单篇） | < 10s |
| 问答首字延迟 | < 2s |
| 流式输出 | 打字机效果，无明显卡顿 |
| 检索延迟 | < 500ms |
| 内存占用（空闲） | < 500MB |
| 内存占用（2000篇） | < 2GB |

- 所有文献数据本地存储（向量库 + 元数据均在本地磁盘）
- LLM 调用走云端 API，仅传输检索结果片段（非全文）
- 不上传原始 PDF 文件到云端
- 用户可随时删除任何文献数据

### 6.3 兼容性

| 环境 | 要求 |
|------|------|
| WPS Office | 最新桌面版 |
| 操作系统 | Windows 10/11（MVP），后续扩展 macOS |
| Python | 3.13+ |
| 内存 | ≥ 8GB（推荐 16GB） |
| 磁盘 | ≥ 2GB 可用空间（不含文献存储） |

### 6.4 可扩展性

- LLM 提供商可替换（通过 LangChain 抽象层）
- 向量库可替换（Zvec → 其他）
- 图谱存储可升级（NetworkX → FalkorDB → Neo4j）
- Embedding 模型可替换（需重建索引）

### 7.1 交互形态

```
│  │                    │  │  论文助手      │  │
│  │   文档编辑区        │  │ (浅绿，半透明)  │  │
│  │                    │  │                │  │
│  │   用户选中的文字     │  │ [对话区域]       │  │
│  │   ═══════════      │  │                │  │
│  │                    │  │ 用户可输入指令    │  │
│  │                    │  │ 或拖入 PDF      │  │
│  │                    │  │                │  │
│  │                    │  │ [输入框]  [发送] │  │
│  └────────────────────┘  └────────────────┘  │
└──────────────────────────────────────────────┘
```

### 7.2 文献导入流程

**方式一：指令导入（MVP）**
```
用户在对话框输入: /start 'D:\papers\xxx.pdf'
  ↓
Agent 按预设清洗工作流执行:
  MinerU 解析 → 噪音过滤 → 语义切块 → 向量化 → 入库
  ↓
遇到报错 → 自动降级处理或修复
  ↓
完成，用户可开始提问
```

**方式二：拖拽导入（v1.1+）**
```
用户直接将 PDF 拖入对话框
  ↓
自动触发 /start 流程
  ↓
向量化前优先检索：是否已存在完全相同的论文？
  ├─ 已存在 → 提示用户"该论文已在库中"
  └─ 不存在 → 正常向量化入库
```

### 7.3 三级权重与四种检索模式

**三级输入源**（参考性从高到低）：

| 级别 | 输入源 | 变量 | 说明 |
|------|--------|------|------|
| 一级 | 用户提示词 | `{user_prompt}` | 最具参考性，用户明确意图 |
| 二级 | 用户圈选文本 | `{user_text}` | 用户主动圈出的重点内容 |
| 三级 | 已写内容（轮询） | `{text}` | WPS 文档自动轮询缓存 |

WPS 插件后台每 5 秒轮询一次用户当前文档内容（缓存，关闭 WPS 即删）。基于三种输入源的组合，有四种检索模式：

**模式 1：纯提示词**
- 条件：用户什么都没写，只有向量化的知识库 + prompt
- 权重：`{user_prompt}` 100%
- 流程：prompt → Query 标准化 schema → Embedding → 语义匹配 → 召回 Top-K → 重排序 → LLM 生成

**模式 2：纯续写（轮询触发）**
- 条件：用户已写了内容，没有圈选，没有 prompt，点击发送
- 权重：`{text}` 100%
- 流程：轮询已写内容 → Query 改写 → 推断接下来可能的句子（在对话框内输出，结合知识库参考）
- 系统提示词专为续写场景设计

**模式 3：圈选辅助（轮询 + 圈选）**
- 条件：用户已写了 + 圈选了，没有 prompt，点击发送
- 权重：`{text}` 30% + `{user_text}` 70%
- 流程：两者分别 Query 改写 → 各自召回 Top-K → 按权重选取 → 重排序 → LLM 生成

**模式 4：全量（轮询 + 圈选 + 提示词）**
- 条件：用户已写了 + 圈选了 + 有 prompt，点击发送
- 权重：`{text}` 20% + `{user_text}` 30% + `{user_prompt}` 50%
- 流程：三者组合 Query 改写 → 召回 → 按权重选取 → 重排序 → LLM 生成

**所有模式的共同要求**：
- 生成的回答必须标注来源（具体到段落级）
- 来源可点击跳转：直接打开本地存储的 PDF 并定位到具体位置
- 圈选范围内的文本权重更高，未圈选的也向量化但权重降低
- 整个对话上下文为临时性质（滑动窗口，不持久化到检索层面）

## 8. Agent 架构

### 8.1 设计原则

**先跑通，再扩展。** MVP 回归最简线路：能对话 + 能结合 PDF 回答。

### 8.2 MVP 核心链路

MVP 只做两件事：

1. **能对话** — LLM 基础聊天能力，保留对话历史（Redis），支持随时续聊
2. **能结合 PDF 回答** — RAG 检索增强生成，基于用户上传的文献库回答问题

```
用户提问 /start '路径'
  ↓
Query 标准化（用户问题 → 标准 schema 格式）→ Embedding 化
  ↓
Zvec 向量检索（单 Collection，payload 过滤）+ BM25 混合检索
  ↓
召回相关片段 → BM25 + Reranker 重排序（从高到低）
  ↓
召回结果（附带来源位置，可点击跳转本地 PDF）+ 对话历史（Redis 滑动窗口）
  ↓
拼装 Prompt → Kimi Coding API → 流式返回
  ↓
回答 + 来源标注（段落级，可点击跳转）
```

### 8.3 三大管道

#### 管道一：数据清洗管道

```
用户输入路径 /start '路径'
  ↓
MinerU API: PDF → JSON + Markdown 双格式输出
  ↓
正则过滤噪音（页码、页眉页脚、乱码等）
  ↓
图片处理: OCR / VLM / 多模态 → 语义描述 → 标签形式附加
  ↓
语义级切块:
  ├─ 正常块（≤ 32k 上下文）: 直接保留
  └─ 超大块（> 32k）: 均匀切分为 24k + 首尾重叠
  ↓
硅基流动 Embedding (Qwen3-Embedding-8B, 1536维)
  ↓
Zvec 单 Collection 存储（payload: paper_id, chunk_type 等元数据）
```

**多索引策略**（用户可选颗粒度）:

| 索引策略 | 说明 | 优缺点 | 阶段 |
|----------|------|--------|------|
| 暴力切分 | 每 chunk 500~1000 token | 速度快，效果一般；前端提示用户 | MVP |
| 父子块检索（级联） | 父块 + 子块关联 | 精准定位 + 保留上下文 | v1.1 |
| 递归检索 | 多级跳转 | 最精确，计算量较大 | v1.2 |

**用户可随时新增或删除 PDF**，通过 payload 过滤实现 CRUD。

#### 管道二：RAG 召回管道

```
用户输入（按模式组合三级输入源）
  ↓
Query 标准化（自然语言 → 标准 schema 格式）
  ↓
Embedding 化 → 语义匹配
  ↓
混合检索 (Hybrid):
  ├─ BM25 关键词检索 (jieba 分词，MVP 轻量高效，中文友好)
  └─ 向量语义检索 (flat 算法)
  ↓
召回所有相关 Top-K → BM25 + Reranker 重排序（从高到低）
  ↓
来源附带位置信息（点击可直接打开本地存储的 PDF 并定位）
  ↓
拼装上下文 + 对话历史（Redis 滑动窗口）
  ↓
系统提示词（根据模式路由） → Kimi Coding API → 流式返回
```

**记忆机制**: 滑动窗口式记忆，上下文长度 = 模型上下文上限（暂不做长期记忆系统）

#### 管道三：WPS 轮询监听（双循环架构）

> 已有验证原型：`D:\同步\.tools\wps-debug`（轮询成功）

**常驻循环（后台轮询）**:
```
WPS 插件启动 → 每 5 秒轮询当前文档内容
  ↓
缓存到临时文件（关闭 WPS 即清空）
  ↓
轮询为独立进程，不阻塞对话
（轮询的复制体进入改写时，轮询本体继续监听）
  ↓
关闭 WPS → 临时文件清空
重新打开 → 优先读取文档已有内容作为初始缓存
```

**对话级循环（用户触发）**:
```
用户点击发送（触发四种模式之一）
  ↓
输入组合（三级输入源按权重） → LLM
  ↓
调用 Query 标准化工具 → 调用 RAG 工具
  ↓
召回（混合检索 + 重排序）
  ↓
组织自然语言 → 输出结果（带上所有参考来源）
  ↓
整个上下文为临时性质（滑动窗口内）
```

## 9. 约束与假设

### 约束
- 必须以 WPS 插件形式交付（非独立应用）
- 向量库必须嵌入式（Zvec），不依赖外部服务
- LLM 调用需要伪造 UA 头（Kimi Coding API 限制）
- 更换 Embedding 模型会导致向量库索引失效

### 假设
- 用户网络环境可以访问 Kimi API（或配置代理）
- 用户机器有足够磁盘空间存储文献数据
- MVP 阶段不需要多用户/协作功能
