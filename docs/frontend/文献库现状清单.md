## 二、后端 API 接口文档（前后端对接依据）

> 以下接口均已实现，前端必须严格按照此文档对接。

### 2.1 Pydantic 响应模型

后端 `LibraryItemOut` 有 `model_validator` 会自动填充别名：

```
item_id        → 同时填充 library_item_id, paper_id（三个值相同）
page_count     → 同时填充 total_pages（两个值相同）
file_type      → 同时填充 kind（两个值相同）
```

**`LibraryItemOut`**（GET /library/items 和 GET /library/trash 的返回元素）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `item_id` | `str` | 主键 |
| `library_item_id` | `str` | = item_id（别名） |
| `paper_id` | `str` | = item_id（别名） |
| `title` | `str` | 论文标题 |
| `authors` | `str` | 作者，逗号分隔字符串 |
| `year` | `int \| null` | 发表年份 |
| `file_path` | `str` | 文件路径 |
| `file_hash` | `str` | SHA-256 前 16 hex |
| `file_type` | `str` | 文件扩展名，如 `".pdf"` |
| `kind` | `str` | = file_type（别名） |
| `import_time` | `str` | ISO 时间戳 |
| `page_count` | `int` | 页数 |
| `total_pages` | `int` | = page_count（别名） |
| `chunk_count` | `int` | 切分块数 |
| `status` | `str` | `"ready"` / `"failed"` |
| `keywords` | `string[]` | 关键词列表 |
| `file_size` | `int \| null` | 文件大小（字节） |

**`ImportStartResponse`**（POST /import/start 的返回）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `task_id` | `str` | 任务 ID（重复时为空字符串） |
| `status` | `str` | `"queued"` 或 `"duplicate"` |

**`ImportStatusResponse`**（GET /import/status/{taskId} 的返回）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `task_id` | `str` | |
| `paper_id` | `str \| null` | 完成后才有 |
| `status` | `str` | `"queued"` / `"running"` / `"completed"` / `"failed"` |
| `current_step` | `str \| null` | 与 status 相同 |
| `error_msg` | `str \| null` | 失败时的错误信息 |
| `file_name` | `str \| null` | 文件名（从 file_path 提取） |
| `percent` | `float \| null` | ⚠️ **永远是 null**（见下方说明） |

> ⚠️ **重要**：`percent` 字段在轮询接口中 **永远是 null**。实时百分比只通过 SSE 流获取。
> 如果不用 SSE，前端需要自行根据 `status` 估算进度，或改用 SSE。

### 2.3 SSE 事件格式（接口 #9）

`event: progress`
`data: JSON`

SSE 事件字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `status` | `str` | `"running"` / `"stage_done"` / `"stage_failed"` / `"completed"` / `"failed"` / `"done"` |
| `step` | `str` | 阶段标识：`transforming` / `vlm_enriching` / `cleaning` / `chunking` / `embedding` / `indexing` |
| `percent` | `int` | 0-100，基于阶段权重计算 |
| `stage_name` | `str` | 中文标签：PDF 解析 / 图片理解 / 内容清洗 / 语义切分 / 向量化 / 建立索引 |
| `paper_id` | `str \| null` | 完成时有值 |
| `error_msg` | `str` | 失败时有值 |

**阶段权重**（用于进度计算）：

| 阶段 | step | 权重 |
|------|------|------|
| PDF 解析 | `transforming` | 25% |
| 图片理解 | `vlm_enriching` | 10% |
| 内容清洗 | `cleaning` | 10% |
| 语义切分 | `chunking` | 15% |
| 向量化 | `embedding` | 20% |
| 建立索引 | `indexing` | 20% |

### 2.4 关键行为细节

#### 导入去重

`POST /import/start` 会自动检查文件 hash（SHA-256 前 16 位）。如果库中已有相同文件（且未被删除），返回 `{"task_id": "", "status": "duplicate"}`，不启动任务。

#### 软删除流程

删除分两步：
1. 向量索引/关键词索引中标记为已删除（RAG 检索排除）
2. SQLite `library_items` 表设置 `deleted_at` 时间戳

恢复时两步都逆转。

#### 永久删除注意

`DELETE /library/items/{id}/permanent` 只删除 SQLite 记录，**不会**删除磁盘文件和向量索引数据。后端的 `DocumentManager.hard_delete()` 存在但未被任何路由调用。

#### ⚠️ 没有重试接口

后端 **没有** `POST /library/items/{id}/retry` 端点。当前前端调用了这个不存在的接口。

**解决方案**（前端重构时二选一）：
- **方案 A**：重试 = 重新上传文件（调用 `POST /import/start`）
- **方案 B**：后端新增 retry 接口（需后端配合）

#### `GET /import/status/{task_id}` 的 status 值

| status | 含义 |
|--------|------|
| `"queued"` | 已排队 |
| `"running"` | 正在处理 |
| `"completed"` | 完成 |
| `"failed"` | 失败 |

#### 错误码汇总

| 接口 | 状态码 | 含义 |
|------|--------|------|
| POST /import/start | 413 | 文件超过 100MB |
| POST /import/start | 400 | 非 PDF 格式 |
| GET /import/status/{id} | 404 | 任务不存在 |
| DELETE /library/items/{id} | 404 | 文献不存在 |
| POST /library/items/{id}/restore | 404 | 文献不存在 |
| POST /library/items/{id}/restore | 400 | 该文献不在回收站中 |
| DELETE /library/items/{id}/permanent | 404 | 文献不存在 |
