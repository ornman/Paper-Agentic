# PDF 清洗重构方案

## 1. 这份文档解决什么问题

这一轮不再讨论“谁的效果理论上最好”，而是收敛 `PDF 清洗模块` 当前阶段的默认落地方式。

目标只有三个：

1. 尽可能保留 PDF 的语义结构
2. 让 LLM 更容易阅读和复用
3. 让默认链路能在普通 Windows 机器上稳定跑起来，而不是强依赖本地 GPU

这份文档只聚焦 `PDF`，不扩散到 `DOCX`。

---

## 2. 本轮已经确认的事实

### 2.1 不能再把 Docling 当当前默认主链

这不是“Docling 效果不好”，而是“当前阶段不适合作为默认本地方案”。

本轮实际验证到的事实：

- 项目 [backend/pyproject.toml](D:\真项目\论文助手\backend\pyproject.toml) 声明了 `docling>=2.0.0`
- 但当前 `backend/.venv` 并没有同步出可直接运行的 `docling`
- 用 `uv` 临时隔离环境做过一次最小验证，`Docling` 本体可以装起来
- 但它在标准 PDF pipeline 启动时，会继续拉取 OCR 与布局模型
- `RapidOCR` 的 CPU 模型可以下载
- `layout` 模型下载卡在 Hugging Face，当前环境下超时

因此，本轮对 Docling 的结论不是“彻底废弃”，而是：

- `Docling` 不适合当前版本的默认清洗主链
- 它更适合作为未来高资源模式、服务器模式或付费增强模式
- 当前 PDF 默认链路必须转向轻量 / API 优先

### 2.2 Marker 不适合当前本地默认链路

你给出的判断是对的：

- `Marker` 对学术 PDF、公式、技术文档更强
- 但它更吃算力，更适合 GPU 或专用服务器
- 老师电脑跑本地重模型，不该是当前默认方案

所以 `Marker` 也不应作为当前本地默认链路，而应放到：

- 远程 API / 服务化能力
- 未来高配机器
- 未来付费增强路径

### 2.3 当前阶段更适合的方向

当前默认方案应改成：

- 本地默认：轻量探针 + 轻量提取 + 必要时调用远程 API
- 重模型能力：下沉到可选远程服务
- 所有路由都产出统一的 `Markdown + JSON + 证据元数据`

## 3. 本轮决策

### 3.1 总决策

`PDF 清洗链路重构为“探针路由 + 多策略清洗 + 统一产物契约”`

不再采用“单一解析器包打天下”的思路。

### 3.2 当前默认优先级

当前优先级调整为：

1. `MarkItDown`：轻量文本主链
2. `pypdf / pdfplumber`：补充结构探测、文本校验、表单探测
3. `convert_pdf_to_images.py`：页面转图，给 VLM / OCR /表格复核使用
4. `extract_form_field_info.py`：表单探针
5. `VLM API / OCR API`：处理图片、扫描件、复杂表格、复杂页面
6. `Marker`：远程增强链路，不是当前本地默认
7. `Docling`：未来高资源可选链路，不是当前默认

### 3.3 对外口径

当前版本不宣称“最强 PDF 解析”。

当前版本宣称的是：

- 有轻量、稳定、可落地的默认链路
- 能按文档特征自动分流
- 对复杂 PDF 提供 API 增强路径
- 后续可接入更强的重模型模式

## 4. 默认架构：先探针，再路由

### 4.1 为什么必须先探针

不同 PDF 差异太大：

- 有的几乎纯文字
- 有的有大图
- 有的有表单
- 有的公式密集
- 有的是扫描件

如果不先探针，直接把所有 PDF 扔给同一条链：

- 轻文档会被过度处理
- 重文档会处理失败或成本失控
- 后续清洗规则会越来越乱

### 4.2 探针要检测什么

导入 PDF 时先做一轮轻量探针，至少输出以下字段：

- `has_text_layer`
- `text_density`
- `is_scan_like`
- `has_images`
- `image_density`
- `has_form_fields`
- `has_formula_signals`
- `has_table_signals`
- `doc_complexity_level`
- `recommended_route`

### 4.3 探针实现优先使用哪些手段

#### 文字层检测

优先用：

- `pypdf`
- `pdfplumber`

判断：

- 能不能提取出稳定文本
- 每页文本量是否足够
- 文本是否明显乱码或断裂

#### 表单检测

优先用：

- `anthropics/skills/pdf` 里的 `extract_form_field_info.py`

判断：

- 是否存在 AcroForm 字段
- 表单字段类型和数量

#### 图片 / 扫描检测

优先用：

- `pdfplumber`
- 页面尺寸与图片对象占比统计
- `convert_pdf_to_images.py` 作为后续视觉链输入

判断：

- 页面是否接近整页图片
- 文本层是否很弱但图片占比很高

#### 公式检测

先只做轻量启发式，不上重模型：

- 数学符号密度
- 连续单行短段落 + 编号 `(1)(2)(3)` 样式
- `∑ ∫ √ ≤ ≥ × ÷ α β λ` 等符号出现频率
- `LaTeX-like` 痕迹，例如 `_ ^ \frac \sum`

#### 表格检测

先做“疑似表格”探针，不要求一次性结构化完美：

- 文本对齐异常密集
- 连续短字段
- 页内大量数字列
- `表 1 / Table 1` 等显式提示
- 页面图像中存在明显表格轮廓时，交给 VLM / 远程增强

## 5. 推荐路由矩阵

## 5.1 Route A：纯文本优先

适用条件：

- 有稳定文字层
- 无明显图片
- 无表单
- 无明显公式
- 无明显复杂表格

默认链路：

- `MarkItDown` 主提取
- `pypdf / pdfplumber` 做抽样校验
- 输出 `normalized markdown + probe report`

为什么这样做：

- 成本最低
- CPU 友好
- 足够覆盖大量结构清晰的普通文档

## 5.2 Route B：文字 + 图片

适用条件：

- 有稳定文字层
- 有图片
- 无表单
- 无明显公式

默认链路：

- `MarkItDown` 提取主文本
- `convert_pdf_to_images.py` 导出相关页图片
- `VLM API` 只负责图片描述和必要的图表说明
- 最终把图片描述回挂到对应页或对应段落锚点

注意：

- 图片描述是增强信息，不应该覆盖原始文本
- 图片增强失败时，不阻断主文本链

## 5.3 Route C：文字 + 表单 / 表格 / 公式

适用条件：

- 有稳定文字层
- 存在表单字段，或明显表格，或明显公式

当前默认链路：

- 文本主干仍先用 `MarkItDown`
- `extract_form_field_info.py` 解析表单字段
- 公式与复杂表格不走本地重模型默认链
- 改为调用远程增强：
  - 公式：`Marker API` 或等价远程公式识别能力
  - 表格：`VLM API` / 专用表格 API 做复核

双保险原则：

- 文本结果有一份
- 表单 / 表格 / 公式结果各自有独立增强结果
- 最终在归一化阶段做字段级合并，而不是让单一工具覆盖一切

## 5.4 Route D：文字 + 表单 + 公式 + 图片

适用条件：

- 结构复杂
- 信息模态混合

当前默认链路：

- `MarkItDown`：拿正文主骨架
- `extract_form_field_info.py`：拿表单字段
- `Marker API`：专门处理公式，必要时顺带处理复杂表格
- `VLM API`：处理图片语义、图表说明、表格复核

这一路由里不要再追求“一个工具搞定全部”。

应该拆成多来源结果，再统一归并。

## 5.5 Route E：扫描件 / 几乎无文字层

适用条件：

- 文字层极弱或没有
- 整页大图特征明显

默认链路：

- 不走 `MarkItDown` 主提取
- 页面转图
- 走 `OCR API + VLM API`
- 若是高价值文档，再进入远程 `Marker` / 重模型增强

结论：

扫描件不是本地轻量链路的强项，必须明确承认并转 API。

## 6. 关于你提出的“80% 相似度放行”

这个思路可以保留，但不要做成模糊口号。

建议具体化成：

- 比对对象：主文本链结果 vs 增强链结果
- 比对粒度：页级或块级，不做整篇粗糙比较
- 指标：
  - 文本重合度
  - 公式块数量是否一致
  - 表格块数量是否一致
  - 关键字段是否缺失

建议用法：

- 不是“低于 80% 就失败”
- 而是“低于阈值则标记 `needs_review` 或走降级合并”

当前更稳的策略：

- `>= 0.80`：自动放行
- `0.60 ~ 0.80`：降级放行，但标记质量风险
- `< 0.60`：进入人工复核 / 重试增强链

## 7. 统一产物契约

无论走哪条路，最终都不要只吐一份裸 markdown。

至少保留四层产物：

### 7.1 原始证据层

- 原始 PDF 路径
- 探针报告
- 中间工具原始返回
- 页图 / 表单字段 / OCR 原文 / API 原文

### 7.2 标准文本层

- `normalized.md`

要求：

- 标题层级尽量保留
- 段落尽量完整
- 不因为清洗而抹掉关键术语

### 7.3 结构化增强层

- `structured.json`

至少包含：

- 页码
- 段落锚点
- 图片描述
- 表格块
- 公式块
- 表单字段
- 来源工具
- 置信度 / 质量标记

### 7.4 路由与质量层

- `extraction_report.json`

至少包含：

- 探针结果
- 命中的 route
- 用了哪些工具
- 哪些模块失败但降级放行
- 是否需要人工复核

## 8. 当前推荐的实际默认流程

可以先按下面这条最务实的链路落地：

1. 导入 PDF
2. 先做轻量探针
3. 若纯文本，直接 `MarkItDown`
4. 若有图片，补 `VLM API`
5. 若有表单，补 `extract_form_field_info.py`
6. 若疑似公式 / 复杂表格 / 扫描件：
   - 默认不在本机硬跑重模型
   - 直接走远程增强 API
7. 最终统一归一化为 `markdown + json + extraction_report`

这比“本地先接一堆重模型，再想办法跑通”更适合当前阶段。

## 9. 对 Docling、Marker、MarkItDown 的定位

## 9.1 MarkItDown

定位：

- 当前默认轻量文本主链

优势：

- 轻量
- CPU 友好
- 启动快
- 适合结构清晰、文本为主的 PDF

限制：

- 不适合复杂公式
- 不适合高质量复杂表格恢复
- 不适合扫描件主链

## 9.2 Marker

定位：

- 远程增强链
- 高价值学术 PDF 增强链

优势：

- 学术文献、公式、技术文档能力强

限制：

- 本地成本高
- 更吃 GPU
- 不适合作为老师电脑上的默认链

## 9.3 Docling

定位：

- 未来高资源增强链
- 服务器侧高级模式

优势：

- 布局理解强
- 表格和结构化表达好
- JSON 索引价值高

限制：

- 运行环境更重
- 首次模型准备成本高
- 当前环境下布局模型拉取不稳定
- 不适合作为当前默认本地方案

## 10. 对 `anthropics/skills/pdf` 的使用边界

这一套不当主解析链，只借工具能力：

- `extract_form_field_info.py`
- `convert_pdf_to_images.py`

它的角色是：

- 作为轻量探针和辅助脚本
- 作为 OCR / VLM 前处理
- 作为复杂文档的旁路能力

它不应该承担：

- 学术 PDF 公式主解析
- 全量统一结构恢复
- 全链路 PDF 清洗主引擎

## 11. 当前阶段的明确结论

### 11.1 做

- 做轻量探针
- 做多路由 PDF 清洗
- 做统一产物契约
- 做 API 增强通道
- 让默认链路能在普通机器稳定运行

### 11.2 不做

- 不把 Docling 当当前默认主链
- 不把 Marker 当当前本地默认主链
- 不为了追求“最强解析”把老师电脑变成重模型实验场
- 不再坚持单一工具全包
