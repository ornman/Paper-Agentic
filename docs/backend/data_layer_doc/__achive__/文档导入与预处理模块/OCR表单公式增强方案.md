# OCR、表单与公式增强方案

> 当前说明：这份文档保留为下一阶段增强设计参考。当前默认链路先按 `VLM-first` 落地，`OCR` 暂不作为这一版的默认启用能力。

## 1. 这份文档讨论什么

这份文档聚焦三个高风险块：

- 表单
- 公式
- 高文字密度图片

它们的共同特点是：

- 不能只靠普通文本抽取
- 不能只靠一句 VLM 描述
- 需要 `OCR + VLM` 协同

这里的目标不是追求学术级公式排版还原，而是：

- 让 LLM 知道这里到底写了什么
- 让表单字段、公式关系、图表文字尽可能进入可搜索文本
- 保留原图和原始 OCR 证据

---

## 2. 总体原则

### 2.1 表单、公式都视为“视觉块”

当前阶段不再把表单和公式强行拆成完全不同的技术栈。

在预处理模块里，它们都先视为：

- 一种图片或局部裁切块
- 先拿视觉证据
- 再做 OCR
- 再交给 VLM 解释

### 2.2 OCR 负责“看清字”

OCR 负责：

- 文字转录
- 字段标签提取
- 数字、单位、符号的尽量保真

### 2.3 VLM 负责“理解它在表达什么”

VLM 负责：

- 字段之间的关系
- 勾选状态、分区结构
- 公式的大意、变量关系
- 表格或表单的整体语义

### 2.4 标准 markdown 不追求完美排版复刻

当前模块目标不是恢复：

- 原论文公式排版
- 原表单视觉布局

而是生成：

- LLM 可读
- 可检索
- 可引用
- 可回看原图

---

## 3. 表单处理策略

## 3.1 什么叫表单

这里的表单包含两类：

### 原生 PDF 表单

例如：

- AcroForm 字段
- 可填写文本框
- 复选框
- 单选框

这类首先走字段探针。

### 视觉表单

例如：

- 扫描申请表
- 调查问卷
- 信息登记页
- 带框线、标签和值的纸质表格

这类不能只靠字段探针，必须走图像增强链。

## 3.2 处理顺序

建议顺序：

1. 先做原生表单字段探针
2. 若字段不足或根本没有字段对象，再把该区域视为视觉表单
3. 对视觉表单先 OCR，再 VLM 理解

## 3.3 表单输出目标

表单增强的目标不是“描述这是一张表单”。

而是尽量输出：

- 字段名
- 字段值
- 勾选项
- 空字段
- 分区结构

建议标准化为：

```json
{
  "form_fields": [
    {"name": "姓名", "value": "张三", "confidence": 0.93},
    {"name": "性别", "value": "男", "confidence": 0.88},
    {"name": "是否同意", "value": "已勾选", "confidence": 0.81}
  ]
}
```

## 3.4 表单回嵌策略

表单不建议只生成一句泛化描述。

建议回嵌为“摘要 + 原图链接”：

```md
[表单包含姓名、性别、联系方式等字段；其中“是否同意”项已勾选。](images/form_001.jpg)
```

更详细字段进入 `structured json`，不全部塞进正文。

原因：

- 否则正文会被表单字段淹没
- 但完全不进正文，LLM 又不知道这里有关键字段

## 4. 公式处理策略

## 4.1 当前阶段对公式的现实目标

当前阶段不追求：

- 高保真 LaTeX 重建
- 学术出版级公式复刻

当前阶段追求：

- 识别公式是否存在
- 识别公式大致内容
- 尽量保留关键符号和变量
- 让 LLM 知道该公式表达什么关系

## 4.2 公式为什么不能只靠 VLM

只靠 VLM 的问题：

- 容易把符号说糊
- 容易丢上下标
- 容易把相似字符混掉

因此公式块默认流程应该是：

1. OCR 先拿文本或符号序列
2. VLM 再基于图像 + OCR 文本解释公式关系

## 4.3 公式输出分两层

### 第一层：近似转录

例如：

- `E = mc^2`
- `y = a + bx`
- `P(A|B) = P(A,B)/P(B)`

即便不完美，也比完全没有强。

### 第二层：语义解释

例如：

- 该式表示能量与质量平方成正比
- 该式表示线性回归关系
- 该式表示条件概率公式

## 4.4 公式回嵌策略

建议默认回嵌成：

```md
[公式近似为 E = mc^2，表达质量与能量的对应关系。](images/formula_001.jpg)
```

如果 OCR 转录质量太差：

```md
[此处包含一条数学公式，表示变量之间的定量关系，建议结合原图查看。](images/formula_001.jpg)
```

规则：

- 有较可靠公式串时，写进描述
- 没把握时，不编造精确公式
- 原图链接始终保留

## 5. 高文字密度图片处理策略

包括：

- 图表截图
- 表格截图
- 软件界面截图
- 扫描页局部块

这一类最容易出现的问题是：

- OCR 字很多，但不知道重点
- VLM 会概括，但可能漏关键字段

因此建议统一采用：

- OCR 先全量抽字
- VLM 再摘要重点
- 结构化层保留 OCR 全文
- markdown 只保留摘要版

---

## 6. OCR 与 VLM 的协同协议

## 6.1 OCR 输出给 VLM 的不是原样堆砌

不建议把大段 OCR 原文原封不动喂给 VLM。

建议先做轻量整理：

- 去除明显噪声
- 合并断裂短行
- 保留候选字段名 / 数值 / 符号

让 VLM 接收到的是“可用 OCR 文本”，不是垃圾堆。

## 6.2 VLM 提示词应该携带块类型

对于不同块，prompt 重点不同：

### 表单

要求 VLM：

- 识别字段区域
- 判断已填写值
- 判断勾选状态

### 公式

要求 VLM：

- 解释变量关系
- 尽量校正 OCR 中明显错误的符号理解

### 表格 / 图表

要求 VLM：

- 总结主题
- 提取关键趋势、关键值、列关系

## 6.3 VLM 不能覆盖 OCR 原文

必须分开存：

- `ocr_text`
- `vlm_description`
- `merged_text`

不能只留最终描述。

原因：

- 以后需要复核
- 以后可能换模型重算描述
- 有时检索直接用 OCR 原文更有效

---

## 7. 何时需要人工复核

以下情况建议直接打 `needs_review`：

- 公式块 OCR 完全乱码
- 表单关键字段缺失率高
- 复选框 / 单选框状态不确定
- 表格数值错位明显
- VLM 和 OCR 结论明显冲突

这里的关键思想是：

不是所有失败都阻断；
但关键块不能静默胡说。

---

## 8. 与 Markdown 的关系

表单、公式、高文字密度图片，最终都应遵守同一回嵌协议：

```md
[……](images/xxx.jpg)
```

如果该块存在原文图注，则采用：

```md
这里是上一段正文。

图1 ……
[……](images/xxx.jpg)
```

但正文只放“摘要后的可读版本”。

没有原始图注时，不额外伪造图注，顺其自然只保留描述行即可。

详细字段、OCR 全文、置信度、失败原因，都放进 `structured json`。

这能保证：

- Markdown 干净
- LLM 有语义
- 系统不丢证据

---

## 9. 当前默认决策

- 表单、公式、高文字密度图片统一按“视觉块”处理
- 原生表单字段探针优先于视觉表单 OCR
- 视觉表单默认 `OCR -> VLM`
- 公式默认 `OCR -> VLM`
- 结果分成：
  - `ocr_text`
  - `vlm_description`
  - `merged_text`
- 标准 markdown 只嵌摘要版描述
- 详细结构进入 `structured json`
- 关键信息缺失或冲突时标记 `needs_review`
