# 统一产物契约

## 1. 这份文档定什么

这份文档只定“文档导入与预处理模块”的统一产物字段，尤其聚焦本轮已经讨论清楚的视觉块部分：

- `anchors`
- `structured json`
- `extraction report`

这里不讨论数据库表结构，也不讨论 API 路由。

---

## 2. 设计原则

本契约借鉴参考项目里“输入校验、任务路由、标准化结果 envelope”的思路，但不照搬它的工具形态。

本模块当前坚持三条原则：

1. 不把所有信息揉成一段字符串
2. 不把运行状态和语义内容混在一起
3. 不让图片回嵌结果失去原文定位关系

因此产物分三层：

- `anchors`：正文锚点
- `structured json`：语义与定位主产物
- `extraction report`：执行过程与降级记录

---

## 3. 顶层产物集合

每次导入至少产出四份东西：

- `markdown`
- `structured json`
- `extraction report`
- `anchors`

说明：

- `markdown` 给 `LLM` 直接读
- `structured json` 给系统二次消费、复核、重建、重跑
- `extraction report` 给运行观测、排障、降级解释
- `anchors` 虽然也会出现在 `structured json` 中，但逻辑上是一个单独的一等产物

---

## 4. `anchors` 字段定稿

每个锚点至少包含：

- `anchor_id`
- `anchor_type`
- `page`
- `block_id`
- `heading_path`
- `paragraph_index`
- `char_start`
- `char_end`
- `bbox`
- `source_text_hash`
- `anchor_text`
- `visual_child_ids`

字段说明：

### `anchor_id`

稳定锚点 ID。

要求：

- 文档内唯一
- 可被 `parent_anchor_id` 直接引用

### `anchor_type`

枚举建议固定为：

- `paragraph`
- `heading`
- `page_stub`

其中 `page_stub` 是图片页或无正文页的兜底锚点。

### `page`

原文页码，`1-based`。

### `block_id`

上一步基础解析器的块 ID，用于回查原始块。

### `heading_path`

该锚点所属标题路径，例如：

```json
["3 实验设计", "3.2 数据来源"]
```

### `paragraph_index`

页内或分节内的稳定顺序号，具体以解析器实现为准，但必须可排序。

### `char_start` / `char_end`

该锚点对应正文在标准 Markdown 主体中的字符区间。

如果是 `page_stub`，允许为空。

### `bbox`

原文中的大致版面框。

### `source_text_hash`

原文文本的 hash，用于：

- 判断锚点是否漂移
- 发现重复或重跑差异

### `anchor_text`

锚点正文的简要文本。

建议：

- 保留完整段落，或
- 至少保留前 `100~200` 字摘录

### `visual_child_ids`

当前锚点下挂载的视觉块 ID 列表，顺序与 `order_in_parent` 一致。

---

## 5. `visual_blocks` 字段定稿

当前阶段在 `structured json` 中统一采用 `visual_blocks`，不再分别造：

- `images`
- `formula_images`
- `form_images`

这些差异统一收敛到 `visual_type`。

每个 `visual_block` 至少包含：

- `visual_block_id`
- `visual_type`
- `source_block_id`
- `page`
- `bbox`
- `image_rel_path`
- `image_hash`
- `parent_anchor_id`
- `order_in_parent`
- `heading_path`
- `nearby_text_excerpt`
- `caption_original`
- `caption_normalized`
- `analysis_route`
- `analysis_provider`
- `analysis_model`
- `analysis_status`
- `attempt_count`
- `final_description`
- `markdown_line`
- `markdown_block`
- `needs_review`
- `review_reason`
- `error_code`
- `error_message`

字段说明：

### 定位层

- `visual_block_id`
- `source_block_id`
- `page`
- `bbox`
- `image_rel_path`
- `image_hash`

作用：

- 找回原图
- 定位原文位置
- 做去重与重试

### 挂载层

- `parent_anchor_id`
- `order_in_parent`
- `heading_path`
- `nearby_text_excerpt`

作用：

- 决定图片应挂在谁后面
- 决定多图顺序
- 给 VLM 与人工复核提供上下文

### 内容层

- `caption_original`
- `caption_normalized`
- `final_description`
- `markdown_line`
- `markdown_block`

作用：

- 保留原文图注
- 保存增强后的回嵌文本

### 运行层

- `analysis_route`
- `analysis_provider`
- `analysis_model`
- `analysis_status`
- `attempt_count`
- `needs_review`
- `review_reason`
- `error_code`
- `error_message`

作用：

- 说明这张图走了哪种分析模式
- 说明成功、降级还是失败
- 记录是否需要人工复核

---

## 6. 状态枚举定稿

### `analysis_status`

建议固定为：

- `pending`
- `ok`
- `degraded`
- `failed`
- `skipped`

语义如下：

- `pending`：已入队，尚未完成
- `ok`：成功拿到可用描述并完成回嵌
- `degraded`：描述不完整或走了兜底文本，但主链继续
- `failed`：分析失败，且没有得到可接受的增强结果
- `skipped`：本次策略明确跳过

### `needs_review`

布尔值。

判为 `true` 的典型情况：

- 公式图描述明显空泛
- 表单图关键字段疑似缺失
- VLM 连续失败后只插入了兜底文本
- 同页大面积视觉块都进入 `degraded`

---

## 7. `structured json` 顶层结构定稿

建议顶层结构至少为：

```json
{
  "document_id": "pdf_xxx",
  "doc_type": "pdf",
  "source_file_path": "papers/demo.pdf",
  "pipeline_version": "preprocess/v1",
  "markdown_path": "full.md",
  "images_dir": "images/",
  "anchors": [],
  "visual_blocks": [],
  "stats": {
    "anchor_count": 0,
    "visual_block_count": 0,
    "visual_block_ok_count": 0,
    "visual_block_degraded_count": 0,
    "visual_block_failed_count": 0
  }
}
```

当前不强制把所有正文块都塞进 `structured json`。

这一版先确保：

- 锚点稳定
- 视觉块稳定
- Markdown 可逆向定位

后续如果要把全文块结构完全纳入，再往这个顶层结构扩。

---

## 8. `extraction report` 字段定稿

`extraction report` 不保存正文语义，主要保存执行事实。

建议至少包含：

- `document_id`
- `doc_type`
- `source_file_path`
- `pipeline_version`
- `status`
- `started_at`
- `finished_at`
- `degraded`
- `stages`
- `counters`
- `vlm_runtime`
- `warnings`
- `errors`
- `needs_review_items`

### `status`

建议固定为：

- `ok`
- `degraded`
- `failed`

### `stages`

建议至少记录：

- `probe`
- `parse`
- `anchor_build`
- `visual_extract`
- `vlm_enrich`
- `markdown_embed`
- `artifact_emit`

每个 stage 建议包含：

- `stage`
- `status`
- `duration_ms`
- `notes`

### `counters`

建议至少包含：

- `anchor_count`
- `visual_block_count`
- `visual_block_enqueued_count`
- `visual_block_ok_count`
- `visual_block_degraded_count`
- `visual_block_failed_count`
- `visual_block_embedded_count`
- `visual_block_embed_failed_count`

### `vlm_runtime`

当前阶段建议记录：

- `provider`
- `base_url`
- `model`
- `api_key_env`
- `request_mode`
- `max_concurrency`
- `max_retries`
- `backoff_strategy`

### `warnings`

轻度问题，例如：

- 图注规范化后有排版噪声
- 某张图描述过短但仍可接受

### `errors`

强错误，建议至少包含：

- `stage`
- `visual_block_id`
- `code`
- `message`
- `attempt`

### `needs_review_items`

建议至少包含：

- `visual_block_id`
- `page`
- `reason`

## 10. 当前明确不定的内容

这一版刻意不定：

- 数据库存储表结构
- `OCR` 字段细节
- 全文正文块的完整 JSON schema
- 向量化后的 chunk 落库字段

先把视觉块回嵌主链定稳，再继续扩。
