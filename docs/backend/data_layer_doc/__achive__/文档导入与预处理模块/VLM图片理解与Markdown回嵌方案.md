# VLM 图片理解与 Markdown 回嵌方案

## 1. 这份文档解决什么问题

这份文档只定三件事：

1. `parent_anchor` 到底是什么，如何稳定绑定图片块
2. 图片块在 Markdown 里如何回嵌
3. 图片块与图注之间的版式协议是什么

这里不讨论具体 SDK 接法，也不讨论本地重模型。

当前阶段的默认方向已经锁定为：

- 图片理解依赖外部 `VLM API`
- 当前先做 `VLM-first`
- `OCR` 作为下一阶段增强位，暂不作为这一版默认链路

---

## 2. 当前已经锁定的结论

- 图片理解属于“文档导入与预处理模块”的增强阶段，不是独立插件
- 标准 Markdown 面向 `LLM` 阅读，不保留空语义的 `![](...)`
- 图片回嵌格式采用 `[自然语言描述](相对图片路径)`
- 回嵌位置固定为：紧跟对应 `parent_anchor` 之后
- 正文和图片区块之间保留一个空行
- 有图注时：图注单独一行，描述单独一行，图注与描述之间不插空行
- 同一 `parent_anchor` 下连续多图时，只在“正文 -> 第一个图片区块”之间保留一个空行
- 图注允许轻量规范化，但不能改变语义
- 描述不强制加“图片描述：”等前缀，正常中文自然语言即可

---

## 3. 核心术语

### 3.1 `anchor`

`anchor` 是正文中的稳定落点，用于：

- 把 Markdown 与原文位置关联起来
- 给图片、表单、公式等视觉块提供挂载位置
- 给后续检索、回跳、人工复核提供坐标

### 3.2 `parent_anchor`

`parent_anchor` 不是“图片自己的锚点”，而是“图片应该挂在谁后面”的正文锚点。

因此：

- 一个图片块只能有一个 `parent_anchor_id`
- 一个 `parent_anchor_id` 可以挂多张图
- 图片块在 Markdown 中永远是“挂在父锚点后面”，而不是混在父锚点正文内部

### 3.3 `visual_block`

这里的 `visual_block` 是统一概念，当前至少包括：

- `image`
- `chart`
- `table_image`
- `formula_image`
- `form_image`
- `screenshot`
- `scan_crop`
- `unknown`

也就是说，表单和公式在当前阶段也先按“视觉块”处理，再决定是否需要下一阶段的 `OCR` 增强。

---

## 4. `parent_anchor` 契约

## 4.1 基本定义

每个 `visual_block` 都必须带：

- `parent_anchor_id`
- `order_in_parent`

语义如下：

- `parent_anchor_id`：该图片块应该插到哪个正文锚点之后
- `order_in_parent`：当同一个父锚点挂多张图时，该图在父锚点下的稳定顺序

这两个字段一起决定最终回嵌位置。

## 4.2 绑定优先级

图片块选取 `parent_anchor` 的优先级固定为：

1. 同页、同分节下最近的前序正文段落锚点
2. 若没有前序正文段落，则挂到当前分节标题锚点
3. 若该页几乎无正文，则创建页级兜底锚点 `page_stub_anchor`

结论：

- `parent_anchor_id` 不允许为空
- 即使整页几乎是图片，也必须生成可挂载的兜底锚点

## 4.3 回嵌位置规则

最终 Markdown 回嵌规则固定为：

- 图片块紧跟 `parent_anchor` 对应正文之后
- 不允许把图片区块挪到后文其他段落
- 不允许为了凑版式把同一父锚点下的图片拆散

## 4.4 同父锚点多图

如果多个图片块共享同一个 `parent_anchor_id`，则按 `order_in_parent` 升序回嵌。

推荐理解方式：

- `parent_anchor_id` 决定“挂在哪段后面”
- `order_in_parent` 决定“这一段后面的多张图谁先谁后”

---

## 5. 图片块的最小结构化契约

这一层只定与回嵌直接相关的字段。

更完整的 `structured json / extraction report` 字段见：

- [统一产物契约](./统一产物契约.md)

每个 `visual_block` 当前至少应包含：

- `visual_block_id`
- `visual_type`
- `page`
- `bbox`
- `image_rel_path`
- `parent_anchor_id`
- `order_in_parent`
- `caption_original`
- `caption_normalized`
- `nearby_text_excerpt`
- `analysis_route`
- `analysis_status`
- `final_description`
- `markdown_line`
- `markdown_block`

其中最关键的是三层：

1. 布局定位层：`page`、`bbox`
2. 挂载定位层：`parent_anchor_id`、`order_in_parent`
3. 文本产出层：`caption_normalized`、`final_description`、`markdown_block`

---

## 6. Markdown 回嵌协议

## 6.1 无图注

如果没有图注，则采用：

```md
这里是上一段正文。

[这里是图片的自然语言描述](images/xxx.jpg)
```

约束：

- 正文和图片区块之间必须有一个空行
- 描述单独一行
- 链接目标使用相对路径

## 6.2 有图注

如果存在图注，则采用：

```md
这里是上一段正文。

图1 城市社区公共文化空间基本框架图
[这里是图片的自然语言描述](images/xxx.jpg)
```

约束：

- 空行出现在“正文”和“整个图片区块”之间
- 图注单独一行
- 描述单独一行
- 图注和描述之间不插空行
- 图注不和描述合并成一行

## 6.3 多图连续回嵌

如果同一个 `parent_anchor` 下连续挂多张图，则采用：

```md
这里是上一段正文。

图1 ......
[第一张图的描述](images/img_001.jpg)
图2 ......
[第二张图的描述](images/img_002.jpg)
```

约束：

- 只在“正文 -> 第一个图片区块”之间保留一个空行
- 相邻图片块之间不额外强制插空行
- 每张图仍保持自己的小块结构：
  - 有图注：`图注 + 描述`
  - 无图注：`仅描述`

## 6.4 为什么不用 `![](...)`

原因已经明确：

- `![](...)` 偏展示，不偏语义
- `[描述](链接)` 能把图片内容直接放进文本主链
- 当前模块的目标是“让 LLM 可读”，不是“让 Markdown 预览更像网页”

---

## 7. 图注处理规则

图注不要求逐字符原样保留，但只允许轻量清洗：

- 去掉多余空格
- 统一全角 / 半角
- 修正常见排版噪声

不能做的事：

- 改写图注含义
- 省略关键图号
- 把图注总结成另一句话
- 没有图注时伪造图注

因此建议同时保留两个字段：

- `caption_original`
- `caption_normalized`

最终 Markdown 默认使用 `caption_normalized`。

---

## 8. 描述生成规则

描述的目标不是“写得短”，而是“正常、直接、可读、可复用”。

默认要求：

- 中文
- 正常自然语言
- 不寒暄
- 不故意压缩到只剩几个词
- 不强制加“图片描述：”“图中展示了：”等前缀

如果图片本身是表单、公式、表格、流程图，则顺其自然让描述包含对应结构信息，不需要额外回避。

例如：

```md
[表单主要包含姓名、联系方式、单位和签字区域，其中勾选项集中在下半部分。](images/form_001.jpg)
```

```md
[图片中是一条数学公式，主体为条件概率关系，分子为联合概率，分母为条件事件的边缘概率。](images/formula_001.jpg)
```

---

## 9. 当前阶段与 OCR 的边界

当前这份文档只锁定：

- 图片如何挂回 Markdown
- 图片块需要保存哪些最小字段
- `parent_anchor` 如何稳定存在

`OCR` 不是这一版默认主链，但要预留接口位。

因此当前视觉块可以先产出：

- `final_description`
- `markdown_line`
- `markdown_block`

后续如果接入 `OCR`，再补充：

- `ocr_text`
- `merged_text`
- 更细粒度的字段结构

这不会推翻本文件定义的回嵌协议。

---

## 10. 决策记录

- 选择 `[描述](链接)`，而不是 `![](...)`：因为目标是语义回嵌，不是视觉展示
- 选择 `parent_anchor_id + order_in_parent`：因为单独靠页码无法稳定确定插入位置
- 选择“图注单独一行，描述单独一行”：因为图注属于原文结构，描述属于增强信息
- 选择“正文和图片区块之间空一行”：因为这能兼顾 LLM 可读性和后续块切分
- 选择“当前 VLM-first，OCR 后置”：因为这一轮先把轻量化中间件主链定稳
