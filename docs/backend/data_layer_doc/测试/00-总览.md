# data_layer 测试设计总览

## 目标
- 测试不再只验证“函数能不能调”，而是验证 `data_layer` 在真实中英文 PDF 样本上的链路行为。
- 测试设计分为两层：
  1. 模块契约测试：快速定位类型、状态机、异常处理和边界条件。
  2. 真实链路验证：用真实 PDF、真实本地索引、真实外部 API（若配置存在）去跑通设计文档承诺的行为。

## 样本池
- 中文样本：`backend/tests/UnitTesting-Docling&&Anthropic'sPDF&DOCX/Testpaper-PDF-ZH-CN`，43 份。
- 英文样本：`backend/tests/UnitTesting-Docling&&Anthropic'sPDF&DOCX/Testpaper-PDF-EN`，111 份。

## ClaudeCode 执行规则
1. 统一使用 `uv run pytest`，不要混用系统 Python。
2. 先用 `probe_pdf()` 扫描全部 154 份 PDF，生成样本清单，再按 route/复杂度选代表样本。
3. 不允许用 mock 伪装真实链路通过：
   - PDF 输入必须是真实文件。
   - MarkItDown、pdfplumber、pdf2image、ChromaDB、BM25 必须走真实现。
   - VLM / embedding / rerank 若依赖外部 API，配置存在时必须走真 API；配置不存在时要明确报 `blocked`，不能用 mock 替代“通过”。
4. 对本轮审计已经坐实的 P0/P1 缺陷，测试要先复现红灯，再修代码转绿，不要直接 `xfail` 掩盖。
5. 每个模块执行后都要保留 artifacts，建议目录：`backend/tests/_artifacts/data_layer/<run_id>/`。

## 执行阶段
1. Phase A：建立 `probe` 样本清单，完成 route 覆盖。
2. Phase B：实现各模块契约测试，先把 P0/P1 缺陷固化成失败用例。
3. Phase C：在 Route A 上跑通中英文 smoke chain。
4. Phase D：修完 transfer/VLM/retrieval/soft_delete 后，扩展到 B/C/D/E 路由和持久化、检索全链路回归。

- 建议新建 `backend/tests/data_layer/` 作为当前架构下的新测试根目录，再视情况删除或归档旧测试文件。
