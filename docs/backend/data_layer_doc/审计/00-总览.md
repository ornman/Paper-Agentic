# data_layer 审计总览

## 审计范围
- 设计基线：`docs/backend/data_layer层/data_layer架构设计.md`
- 代码范围：`backend/app/data_layer`
- 样本集：
  - `backend/tests/UnitTesting-Docling&&Anthropic'sPDF&DOCX/Testpaper-PDF-ZH-CN`，共 43 份 PDF
  - `backend/tests/UnitTesting-Docling&&Anthropic'sPDF&DOCX/Testpaper-PDF-EN`，共 111 份 PDF

## 已验证事实
1. `uv run` 环境下 `pdfplumber`、`pypdf`、`markitdown`、`pdf2image`、`chromadb`、`jieba`、`rank_bm25` 可导入，说明后续结论主要是代码问题，不是依赖缺失。
2. `from app.data_layer.preprocessing.vlm_understanding import process_images` 直接复现 `ImportError`，说明 Route B/C/D 真实链路当前不可用。
3. `SoftDeleteManager.mark_deleted()` 在 Windows 上复现 `TypeError: unsupported operand type(s) for +: 'WindowsPath' and 'str'`。
4. `KeywordIndex.query()` 在 2 篇、3 篇文档样本下都返回空列表，说明 BM25 当前会把小语料库场景直接打空。
5. 对真实中文 PDF 执行 `convert_pdf(..., route="A")` 可得到 markdown，但 `metadata.page_count == 0`；同一文件 `probe_pdf()` 返回 `page_count == 38`。
6. `convert_pdf(..., route="E")` 会直接返回“扫描件需要 OCR API，当前版本暂不支持”，与设计文档的 Route E 不一致。
7. `rrf_fuse()` 只以 dense 结果集为全集，BM25-only 文档会被静默丢弃；把 `SparseRetriever` 的 `SparseResult` 直接传入时还会触发 `TypeError`。
8. 现有基线测试 `backend/tests/UnitTesting-Docling&&Anthropic'sPDF&DOCX/test_pdf_router.py:26` 仍在导入旧路径 `app.data_layer.conversion.pdf_router`，现状下不能作为有效回归测试。

## 总体结论
- 当前 `data_layer` 更接近“分散的模块草稿”，还不是设计文档里描述的完整三层数据链路。
- Route A 有局部可运行能力，但 metadata、cleaning 后继、chunking 约束和持久化交接都不完整。
- Route B/C/D/E 还不适合做通过性验收，原因不是单一模块，而是 `transfer -> transformation -> vlm_understanding` 的交接链路尚未闭合。
- 持久化层和检索层已经有可审计代码，但检索正确性、软删除可用性、监控接线都存在阻断性缺陷。

## 优先级建议
1. 先修 `transfer`、`vlm_understanding`、`retrieval`、`soft_delete` 的 P0 问题，让真实链路至少能跑通 A/B/C。
2. 再修 `transformation`、`cleaning`、`chunking` 的产物一致性问题，让预处理产物符合设计文档。
3. 最后补齐 `config`、`monitor`、`file_management` 与 service/agent 层的接线，进入全量中英文样本回归。
