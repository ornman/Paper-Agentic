# 后端现状代码映射

这份文档只做一件事：

基于当前 `backend/app` 的代码事实，给出“现有目录/模块 -> 目标三层结构”的映射和去留判断。

它不是目标设计文档，目标设计仍然以 [architecture.md](D:/真项目/论文助手/docs/backend/architecture.md) 为准。

真正开始改代码时，优先配合这张表一起看：

- [迁移任务表](D:/真项目/论文助手/docs/backend/迁移任务表.md)

## 0. 当前真实启动链

当前真实启动链是：

`backend/main.py`
-> `backend/app/main.py`
-> `backend/app/bootstrap/app_factory.py`
-> `backend/app/interfaces/http/router.py`
-> `bootstrap/application/infrastructure/interfaces`

这条链说明两件事：

1. 当前运行态已经不走旧 `api/v1/router.py`
2. 当前运行态仍然只是临时骨架，不是最终三层目录

当前目录状态表：

| 目录/文件组 | 当前身份 | 备注 |
|---|---|---|
| `main.py` + `app/main.py` + `bootstrap/*` + `interfaces/*` | 活动临时入口 | 现在会被启动，但不代表最终结构 |
| `application/*` + `domain/*` + `infrastructure/*` | 活动临时骨架 | 现在承载部分主链，但后续还要按三层重归属 |
| `api/*` + `clients/*` + `core/*` + `models/*` + `pipelines/*` + `stores/*` + `utils/*` | 遗留残留 | 不在当前启动路径上，只能拿来摘逻辑，不能继续补新功能 |
| `services/*` | 空壳目录 | 不应继续扩张 |

评测也不应该在这一状态提前接入，因为现在测到的仍然是混合态实现。

## 1. 先看现状，不讲理想

当前 `backend/app` 里同时存在三套东西：

1. 旧活动实现
2. 临时拉出来的新 Clean 风格骨架
3. 空壳或过渡目录

当前目录：

```text
backend/app/
├─ api/
├─ application/
├─ bootstrap/
├─ clients/
├─ core/
├─ domain/
├─ infrastructure/
├─ interfaces/
├─ models/
├─ pipelines/
├─ services/
├─ stores/
├─ utils/
└─ main.py
```

这不是“多层架构并存”，而是重构过程中留下的混合态。

## 2. 三类代码的定位

### 2.1 旧实现残留

主要是：

- `api/`
- `clients/`
- `core/`
- `models/`
- `pipelines/`
- `stores/`
- `utils/`

它们承载了当前大部分真实业务逻辑，但职责边界是旧的，很多模块把：

- HTTP
- 检索
- LLM
- 会话持久化
- 状态缓存

揉在了一起。

它们当前最准确的身份不是“活动代码”，而是“待迁移的旧逻辑来源”。

### 2.2 临时新骨架

主要是：

- `bootstrap/`
- `application/`
- `domain/`
- `infrastructure/`
- `interfaces/`

这套东西不是最终目标结构，只是前一轮为了把旧逻辑拆出层次临时搭的脚手架。

里面有一些可复用的实现，但它的分层语言已经不再是当前项目的目标架构语言。

### 2.3 空壳或可直接退出的目录

主要是：

- `services/`：当前几乎为空

这个目录不应该继续扩张。

## 3. 按目录做映射

## 3.1 `api/`

### 当前职责

- FastAPI 路由入口
- 旧 API 命名
- 通过 `deps.py` 访问全局单例

代表文件：

- `api/v1/routes/import_routes.py`
- `api/v1/routes/query_routes.py`
- `api/v1/routes/poll_routes.py`
- `api/v1/routes/paper_routes.py`
- `api/v1/routes/conversation_routes.py`

### 当前问题

- `import_routes.py` 仍是上传式 PDF 导入，不符合当前“本地路径 + PDF/DOCX + Docling”主线
- `query_routes.py` 仍然使用 `prompt / selection / draft` 契约，没有切到 `written_context / selection / prompt`
- `poll_routes.py` 直接使用内存 `_poll_cache`，这和现在定下来的常驻轮询 + 冻结副本 + 进程内内存运行态相冲突
- `paper_routes.py`、`conversation_routes.py` 直接碰 SQLite 或 pipeline service，边界很旧
- `deps.py` 依赖全局单例，是明确要淘汰的模式

### 目标归属

- 整体迁入 `service_layer/api/`

但不是平移，而是重写接口契约。

### 去留判断

- `api/v1/routes/health_routes.py`：思路可保留，重写到 `service_layer/api/health`
- `conversation_routes.py`：可保留“查询列表 / 查询会话 / 删除会话”这类功能点，但协议和底层依赖方式要重写
- `paper_routes.py`：可保留“列资源 / 删除资源 / 打开原文”这类能力点，但命名和数据来源要改
- `query_routes.py`：功能要保留，整体逻辑不能直接保留
- `poll_routes.py`：能力要保留，当前实现必须废弃
- `import_routes.py`：当前上传式实现废弃
- `deps.py`：删除

## 3.2 `models/`

### 当前职责

- 旧 HTTP schema
- 路由层和业务层共用的输入输出模型

代表文件：

- `models/query.py`
- `models/paper.py`
- `models/conversation.py`

### 当前问题

- `QueryRequest` 仍然以 `draft` 为核心字段，不符合现在的 `written_context` 设计
- `models` 同时承担了 HTTP schema 和部分业务术语表达
- 与当前三层设计相比，命名仍然偏旧

### 目标归属

- 迁入 `service_layer/schemas/`

### 去留判断

- 大部分字段级定义可以参考
- 但命名、来源语义、返回结构都要重做

## 3.3 `core/`

### 当前职责

- 配置
- 统一错误

代表文件：

- `core/config.py`
- `core/errors.py`

### 当前问题

- `config.py` 的路径和环境变量骨架可以用，但里面仍然带着旧切分参数和旧解析主线假设
- `errors.py` 更像旧 API 错误层，不足以覆盖现在要求的 `retrying / degraded / needs_user_action / self-heal`

### 目标归属

- `config.py` -> `service_layer/config/`
- `errors.py` -> `service_layer/api/error_mapping` + `agent_layer/runtime/error_types` + `data_layer/*/errors`

### 去留判断

- 配置路径约定和环境变量命名有可复用价值
- 错误类型体系必须重构

## 3.4 `clients/`

### 当前职责

- 外部服务客户端

代表文件：

- `clients/mineru_client.py`
- `clients/vlm_client.py`
- `clients/embedding_client.py`
- `clients/llm_client.py`

### 当前问题

- `mineru_client.py` 仍然把 MinerU 当成主 PDF 解析链，而当前主线已经切到 `Docling`
- `embedding_client.py` 和 `llm_client.py` 没有和当前 tokenizer、动态预算、日志观测约束对齐
- 客户端边界本身是对的，但 owner 不清晰

### 目标归属

- `mineru_client.py`：
  - 如果后续还保留，应该降为 `data_layer/preprocessing/reference_or_fallback/`
  - 不再作为主处理链
- `vlm_client.py`：
  - `data_layer/preprocessing/image_semantics/`
- `embedding_client.py`：
  - `data_layer/indexing/embedding/`
- `llm_client.py`：
  - `agent_layer/runtime/model_gateway/`

### 去留判断

- `llm_client.py` 可参考 OpenAI/Kimi 兼容调用方式
- `embedding_client.py` 可参考批量向量化封装方式
- `vlm_client.py` 可参考图片描述调用方式
- `mineru_client.py` 作为主线应退场，只保留可能的补充价值

## 3.5 `stores/`

### 当前职责

- Chroma
- BM25
- SQLite
- 内存缓存

代表文件：

- `stores/chroma_store.py`
- `stores/bm25_store.py`
- `stores/sqlite_repo.py`
- `stores/memory_cache.py`

### 当前问题

- `sqlite_repo.py` 把 papers / import_logs / conversations 混在同一个仓储里，和现在三层职责不一致
- `memory_cache.py` 仍然内建 `_poll_cache` 思路，与正式的进程内会话窗口 / 冻结副本设计冲突
- Chroma/BM25 封装本身有复用价值，但 metadata 设计仍然偏旧

### 目标归属

- `chroma_store.py` -> `data_layer/storage/vector_index/`
- `bm25_store.py` -> `data_layer/storage/keyword_index/`
- `sqlite_repo.py` -> 拆分到：
  - `data_layer/storage/document_repository/`
  - `agent_layer/session/conversation_repository/`
  - `agent_layer/session/summary_repository/`
- `memory_cache.py` -> 不作为正式主实现，最多作为测试替身

### 去留判断

- Chroma/BM25 包装值得保留一部分
- SQLite schema 和仓储边界必须拆
- 内存轮询缓存主实现废弃

## 3.6 `pipelines/ingestion/`

### 当前职责

- 旧导入编排主链
- 清洗、切块、图片注入、备份

代表文件：

- `pipelines/ingestion/service.py`
- `backup_manager.py`
- `cleaner.py`
- `chunker.py`
- `image_handler.py`

### 当前问题

- `service.py` 直接串了：
  - 任务状态
  - MinerU
  - VLM
  - 清洗
  - 切块
  - embedding
  - Chroma/BM25/SQLite
- 这是旧“全链路大编排器”，不能直接搬进新结构
- 导入入口仍然是 PDF 上传，不支持 DOCX
- 主解析链仍然假设 MinerU，不符合现在 Docling 基线
- chunker 没有纳入现在定下来的 tokenizer、retrieval chunk / answer context pack 分离规则

### 目标归属

- `cleaner.py` -> `data_layer/preprocessing/normalize/`
- `image_handler.py` -> `data_layer/preprocessing/image_semantics/`
- `chunker.py` -> `data_layer/indexing/chunking/`
- `backup_manager.py` -> `data_layer/storage/import_artifacts/`
- `service.py` -> 拆散，不保留整体

### 去留判断

- 里面的阶段化思路、file_hash 去重、备份恢复思想可以保留
- 旧导入编排整体必须拆掉重做

## 3.7 `pipelines/retrieval/service.py`

### 当前职责

- 旧问答编排主链
- query 拼接
- dense + BM25 + RRF
- LLM 输出
- 对话持久化

### 当前问题

这是当前旧后端里耦合最重的模块之一。

它同时承担了：

- 输入源组装
- 检索
- 融合
- context pack 裁剪
- LLM 回答
- SQLite 对话保存

而且现在还有几处与目标架构明显冲突：

- 输入仍然是 `prompt / selection / draft`
- 没有 `written_context` 的常驻监听/冻结副本机制
- 固定 `_MAX_CONTEXT_TOKENS` 和固定召回规模
- 没有 reflection 分批取证
- 没有 compact 和上下文剩余比例控制
- 日志和状态语义仍然偏粗

### 目标归属

不能整体迁移。

必须拆到：

- `agent_layer/planning/`
- `agent_layer/runtime/`
- `agent_layer/response/`
- `agent_layer/session/`
- `data_layer/retrieval/`

### 去留判断

- `_rrf_fuse()` 这类融合函数有复用价值
- source label/source snippet 这类结果包装逻辑有复用价值
- QAService 整体必须拆散重写

### 目标归属

- `error_handler.py` -> 更接近 `service_layer/bootstrap/retry_policy` 或外部调用支持模块
- `rate_limiter.py` -> 更接近外部客户端 owner，例如 `data_layer/preprocessing/external/` 或 `agent_layer/runtime/model_gateway/`

### 去留判断

- 两者逻辑都可保留
- 但要归到明确 owner，不继续放在 `utils/`

### 当前职责

这是上一轮临时拉出来的新骨架。

其中有些代码已经具备“新 API 命名”和“本地路径导入”的雏形，但整体仍然围绕错误的分层语言组织。

- `settings.py`
- `logging.py`
- `app_factory.py`
- `lifespan.py`

## 4. 路由级迁移映射

| 当前路由 | 当前问题 | 目标归属 |
|---|---|---|
| `/import/start` | 上传式 PDF 导入 | `service_layer/api/library_items/imports`，改成本地路径 + PDF/DOCX |
| `/import/status/*` | 旧任务状态模型 | `service_layer/api/import status`，改成新状态机 |
| `/query` | 旧输入契约，固定取证 | `service_layer/api/assistant`，改成三源加权 + reflection |
| `/poll/*` | 内存 `_poll_cache` | `service_layer/api/written_context` |

## 5. 可以直接带走的“信号”，不是整块复用

当前代码里真正值得带走的是这些信号：

- file hash 去重
- 导入分阶段记录
- SSE 元数据输出形态
- BM25 + Dense + RRF 的混合检索方向
- 原文来源包装
- backup / resume 思路
- retry / backoff / rate limiter 工具

它们是“信号”，不是“整块模块直接搬家”。

## 6. 明确要停止强化的东西

后续不要再继续强化这些东西：

- 全局 `deps.py` 单例依赖
- 上传式 import
- `prompt / selection / draft` 旧输入主契约
- 内存 `_poll_cache`
- 固定 `topk`
- 把 retrieval 和 answer context 混成一层
- 让 `application / domain / infrastructure / interfaces` 继续长成正式目标结构

## 7. 后续代码重构顺序

基于当前现状，真正合理的顺序是：

1. 先重建数据层主链
   - Docling
   - Markdown + JSON
   - chunk / anchor / vector / bm25
2. 再重建 Agent 层
   - written_context 常驻监听
   - 四种场景加权
   - reflection
   - compact
   - 进程内窗口清理与 compact 监控
3. 最后重建服务层
   - API
   - SSE
   - logging / health / observability
4. 稳定后再删除旧目录和临时骨架

## 8. 结论

当前 `backend/app` 不是“结构已经基本对了，只差微调”，而是：

- 旧实现仍然主导真实行为
- 临时新骨架提供了一部分可复用思路
- 真正目标三层结构还没有落成

所以后续重构必须坚持：

- 先按三层定 owner
- 再摘旧代码里有价值的部分
- 不做目录平移式“翻新”
