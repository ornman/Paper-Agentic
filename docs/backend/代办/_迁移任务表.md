# 后端迁移任务表

这份表是给后续真正改代码用的，不是概念文档。

目标：

- 把当前 `backend/app`、`backend/scripts`、`backend/tests` 里的关键文件逐个归类
- 明确它们进入三层目标结构时该怎么处理

动作只用四类：

| 动作 | 含义 |
|---|---|
| `保留迁移` | 当前文件整体思路对，迁到新位置后继续演进 |
| `部分复用` | 当前文件只能摘一部分逻辑，不能整体保留 |
| `重写` | 功能点要保留，但当前实现不适合作为新结构主实现 |
| `删除` | 不再保留为活动实现 |

## 0. 当前停止线

这张表当前只服务于“先重构后评测”。

在以下条件满足前，不启动正式评测体系：

- 活动链路与遗留目录边界已收敛
- 数据层主链已跑通
- Agent 层主链已跑通
- 服务层主链已跑通

同时，Kimi 不再作为活动默认基线；后续如保留兼容逻辑，也只是兼容分支。

## 1. 先动手的六个批次

### 批次 -1：边界与基线收敛

先做这些：

- 写清当前真实启动链
- 冻结遗留目录
- 修正示例配置，移除 Kimi 默认暗示
- 明确评测后置

原因：

- 不先把边界钉死，后面每一刀都可能继续在混合态上加料

### 批次 0：公共底座

先做这些：

- settings
- structured logging
- lifespan
- request_id / task_id / session_id 贯穿
- 统一错误载荷骨架

原因：

- 先把日志和运行骨架立起来，后面的数据层和 Agent 层切片才有可排障基础

### 批次 1：数据层主链

先做这些：

- Docling 接入
- Markdown + JSON 解析产物
- chunk / anchor / embedding / vector / BM25
- 文件与元数据持久化

### 批次 2：Agent 层主链

再做这些：

- `written_context` 监听
- 三源加权
- reflection
- compact
- 进程内内存运行态与活动窗口清理

### 批次 3：服务层主链

最后再做：

- API
- SSE
- health
- logging / observability

### 批次 4：清理

- 删旧路由
- 删旧 pipeline
- 删全局 deps
- 删临时 clean 骨架

### 批次 5：评测体系后置接入

- 黑盒 runner
- 内部态 runner
- A/B runner
- judge 与报告

更细的切片顺序见：

## 13. 结论

现在已经能明确两件事：

1. 当前代码不是“修修补补就能符合目标架构”
2. 也不是“一切推倒重来”

正确做法是：

- 按三层定 owner
- 只摘旧代码里真正有价值的信号
- 明确哪些必须重写
- 跑通新主链后果断清旧目录

这张任务表就是后面真正开始改代码时的工作底稿。
