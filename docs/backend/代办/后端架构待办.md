## 0. 当前不做的事

在三层主链没有重建完成前，当前明确不做：

- 不先接评测体系
- 不先做 RAG A/B
- 不继续把 Kimi 当活动默认值
- 不继续给旧 `api/clients/core/models/pipelines/stores/utils` 补新功能

这些不是永久放弃，而是明确后置。

## 1. 文档导入与解析链路 ✅

### 默认落地 ✅

- 文档导入采用“探针路由 + 多策略处理链”，不把单一解析器写死为主基线
- `PDF` 与 `DOCX` 在数据契约层面同级支持
- `PDF` 当前默认方向是轻量优先：
  - 轻量探针
  - 轻量文本主链
  - 必要时走图片 / OCR / 公式 / 表格增强 API
- 每次导入统一保存：
  - `markdown`
  - `structured json`
  - `extraction report`
  - 锚点与必要原始证据
- 图片语义增强默认是 best-effort：
  - 有图片就尝试增强
  - 失败进入 `degraded`
  - 不阻断主文本链

相关模块文档：

- [文档导入与预处理模块](../文档导入与预处理模块/README.md)
- [PDF 清洗重构方案](../关于PDF的清洗模块/PDF清洗重构方案.md)

### 仍待验证

- `DOCX` 的模块级实现文档还需要补齐
- PDF 探针字段的阈值还需要真实样本校准
- 远程增强 API 的超时、成本和降级门槛还需要落表

## 2. 图片语义与 chunk 持久化 ✅

### 默认落地 ✅

- 图片语义描述直接写入 markdown，成为 chunk 内容的一部分
- 不需要 parent-child 结构：检索到 chunk 就自然带上了图片语义
- 向量化时：chunk 整体入索引（包含图片描述文本）
- 回答组装时：召回 chunk 则图片语义一并召回，无需额外联动逻辑

## 3. 语义切分与 tokenizer ✅

### 默认落地 ✅

- 默认先走语义切分
- 标题层级参与 chunk 边界
- 表格、公式、脚注不切断
- `PDF` 与 `DOCX` 共用同一套切分原则
- retrieval chunk 与 answer context pack 分离

tokenizer 默认策略：

- embedding 侧使用与 embedding 模型同名 tokenizer
- 远程 embedding API 也必须本地保留对应 tokenizer，用于预算与校验
- rerank 如果模型不同，则维护独立 tokenizer 预算
- 生成模型 token 统计优先信 provider usage，其次才是本地估算

超长处理顺序：

1. 先重切
2. 再强制均分
3. 仍超限则拒绝入索引

overlap 默认策略：

- 只在单个语义块超过 embedding 模型上下文 `1/4` 时启用
- 首次默认 overlap 取块长的 `10%`
- 若样本验证显示冗余过高，再往下调，不往上无节制加

### 仍待验证

- 你的真实论文样本在 `10% overlap` 下的召回收益是否显著
- retrieval chunk 到 answer context pack 的具体拼装规则是否还要再分层

## 4. reflection 驱动的取证预算 ✅

### 默认落地 ✅

- 不使用固定 `topk`
- 每轮都要产出 reflection verdict：
  - `supported`
  - `insufficient`
  - `conflicting`
  - `off_track`
- 默认最多 3 轮 reflection
- 默认最多 2 次方向切换
- 命中 `supported` 就停止继续取证
- 连续两轮 `off_track` 直接停止，返回结构化失败或请求用户补充

### 仍待验证

- 哪类问题更适合第一轮就偏 dense，哪类问题第一轮应偏 sparse

## 5. 输入加权与冻结副本 ✅

### 默认落地 ✅

活动输入固定为：

- `written_context`
- `selection`
- `prompt`

四种起手权重保持当前总纲约定：

- only prompt: `100 / 0 / 0`
- only written_context: `0 / 0 / 100`
- written_context + selection: `0 / 70 / 30`
- written_context + selection + prompt: `50 / 30 / 20`

这里的数字顺序统一写成：

- `prompt / selection / written_context`

冻结副本默认策略：

- 每次用户发送时，以 `request_id` 冻结一份输入快照
- 快照进入当轮 planning / retrieval / generation
- 常驻轮询继续更新 live `written_context`
- 冻结副本默认短 TTL 保存，便于日志排障和失败恢复

### 仍待验证

- 冻结副本是只保留在进程内内存，还是同时落一份 SQLite 调试痕迹

## 6. 上下文 token 监控与 compact ✅

### 默认落地 ✅

- 每轮响应开始时计算：
  - `context_tokens`
  - `remaining_tokens`
  - `remaining_ratio`
- 三个值进入首个 SSE metadata
- `remaining_ratio < 5%` 时触发 compact
- compact 使用成本更低的小模型
- compact 摘要写入长期事实层
- 下一轮 prompt 组装顺序默认是：
  - `summary`
  - 最近窗口消息
  - 当前冻结副本

### 仍待验证

- compact 摘要是单段文本，还是结构化摘要块更适合你的使用习惯

## 7. 进程内内存窗口与恢复策略 ✅

### 默认落地 ✅

监控源：

- 当前活动窗口 token 预算
- 冻结副本数量与过期时间
- `written_context` 最近更新时间
- 最近窗口消息数

窗口策略：

- 活动窗口默认保持在当前上下文预算可承受范围内
- 接近预算上限时优先 compact，再裁剪最旧的窗口内容
- 冻结副本保留短时间，过期后自动失效
- 长期事实和摘要继续走 SQLite，不靠短期内存承担

恢复策略：

- 进程重启后 live window 自然清空
- 冻结副本失效后由最新 editor context + SQLite 摘要重建
- 没有外部服务可重连，只有空窗口回退和重新组装

### 仍待验证

- 是否要把当前窗口使用率、冻结副本数量、compact 触发次数暴露成单独健康指标卡片

## 12. 用户反馈整改项（2026-05-28）

### 12.1 EmbeddingClient 去硬编码

现状：`embedding_client.py` 默认参数硬编码 `base_url="https://api.siliconflow.cn/v1"`、`model="Qwen/Qwen3-Embedding-4B"`、`dimensions=1536`、`max_concurrency=5`。

要求：
- 所有参数从 `BackendSettings` 注入，不在 embedding_client.py 内出现任何供应商名称
- URL 用抽象名称（如 `embedding_base_url`），在 `.env` / `config.py` / yaml 中统一配置，不要写死硅基流动地址
- 模型名称、维度同样自由配置，不绑定特定供应商
- token 上限改为 32000（Qwen3-Embedding 真实上限）
- 最大并发数当前太保守，需要在文档中说明计算公式和安全余量策略
- 并发数动态计算公式：
  ```
  C = min(RPM_limit * L / 60, TPM_limit * L / (60 * T))
  ```
  其中：
  - `RPM_limit = 2000`（每分钟最大请求数）
  - `TPM_limit = 1,000,000`（每分钟最大 Token 数）
  - `L` = 每个请求的平均响应时间（秒），包括网络延迟和模型推理时间
  - `T` = 每个请求平均消耗的 Token 总量（prompt tokens + completion tokens）
  - "最大并发"的实用上限就是 RPM 限制本身（2000），"最优并发"按公式结合业务指标动态计算
  - 向下取整，视情况留出安全余量
- MinerU API 地址也不硬编码，后续走镜像站中转轮询（用户会收集 MinerU 的多种 API 通过镜像站轮询），并发数计算公式也可以自适应

### 12.9 window_store 动态窗口

现状：`max_messages=20` 硬编码，不随模型上下文窗口调整。

问题：如果用户用 128k 上下文的模型，连 5 轮对话都难；如果用户用 10M 上下文的模型，20 轮对话就结束太浪费。窗口大小应该根据模型能力动态决定。

要求：
- 窗口大小根据模型上下文窗口动态计算
- 用户在前端填入模型信息时，后端获取 `context_window` 和 `max_output_tokens`
- 填入时进行检测，得到的信息不只是模型可用性，还有上下文窗口和最大输入输出
- 窗口策略：根据剩余上下文预算决定保留多少轮对话，而非固定轮数
- 公式：`max_messages = f(context_window, avg_message_tokens, reserved_output_tokens)`
