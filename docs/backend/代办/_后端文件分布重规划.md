# 后端文件分布重规划

这份文档只解决一个问题：

当前 `backend/app` 太乱，下一步应该怎么重新摆放文件，才能不再牵一发而动全身。

它不讨论接口细节，也不讨论 RAG 调优，只讨论：

- 哪些目录继续活
- 哪些目录冻结
- 新代码往哪放
- 旧代码怎么迁

## 1. 当前问题不是“层次不够”，而是“混合态太多”

现在的 `backend/app/` 同时挂着三批东西：

```text
backend/app/
├─ api/            # 旧路由
├─ clients/        # 旧外部客户端
├─ core/           # 旧配置/错误
├─ models/         # 旧 schema
├─ pipelines/      # 旧导入/检索编排
├─ stores/         # 旧存储包装
├─ utils/          # 旧杂项
├─ application/    # 临时新骨架
├─ bootstrap/      # 临时新骨架
├─ domain/         # 临时新骨架
├─ infrastructure/ # 临时新骨架
├─ interfaces/     # 临时新骨架
└─ services/       # 空壳
```

这不是正常状态。

真正的问题是：

- 旧实现没退场
- 临时骨架先长起来了
- 两边都不该继续扩张

所以这一步的原则非常简单：

- 不再往旧目录补功能
- 也不再往临时 Clean 骨架里继续堆终局代码
- 直接开始为目标三层目录腾地方

## 2. 目标文件分布

最终目标只保留三层活动代码：

```text
backend/
├─ app/
│  ├─ data_layer/
│  ├─ agent_layer/
│  └─ service_layer/
├─ scripts/
├─ tests/
└─ data/
```

其中：

- `app/` 只放活动实现
- `scripts/` 只放手工运维、导入、修复、检查脚本
- `tests/` 按三层对应拆分
- `data/` 继续是运行态目录，不改

## 4. 哪些旧目录现在就该冻结

从现在开始，这些目录只允许：

- 读取
- 摘逻辑
- 删除

不允许再加新功能：

```text
backend/app/api/
backend/app/clients/
backend/app/core/
backend/app/models/
backend/app/pipelines/
backend/app/stores/
backend/app/utils/
backend/app/application/
backend/app/domain/
backend/app/infrastructure/
backend/app/interfaces/
backend/app/services/
```

注意：

- 前七个是旧实现残留
- 后五个是临时骨架
- `services/` 是空壳

冻结规则只有一条：

- 新增文件一律不进这些目录

## 5. 哪些东西可以先保留原位，作为过渡入口

在完全迁走前，下面这些还可以暂时留着做桥接：

- `backend/main.py`
- `backend/app/main.py`
- `backend/scripts/`
- `backend/tests/`

但它们的职责要很克制：

- `main.py` 只做启动入口
- `scripts/` 只调活动三层代码，不再调旧 pipeline
- `tests/` 开始按三层迁移，不再新增旧目录测试

## 6. PDF / DOCX 处理在文件分布上怎么落

这里把最容易反复打架的地方直接写清楚。

### 6.1 PDF 主链

主链位置：

- `data_layer/conversion/docling_adapter.py`
- `data_layer/conversion/conversion_options.py`
- `data_layer/normalization/*`

不再放在：

- `clients/mineru_client.py`
- `pipelines/ingestion/service.py`

### 6.2 DOCX 主链

主链位置也一样：

- `data_layer/conversion/docling_adapter.py`
- `data_layer/normalization/*`

如果后续需要 DOCX 解包、校验、tracked changes 清洗等旁路工具：

- 放到 `scripts/docx_tools/`
- 或 `data_layer/fallbacks/` 之外的专用 `document_debug/`

不和主转换链混住。

### 6.3 MinerU 去留

当前结论：

- `MinerU` 不再作为主路径保留在活动导入链
- 如果继续留，只能在：
  - `data_layer/fallbacks/mineru_fallback.py`
  - 或 `data_layer/legacy/mineru_compat.py`

也就是说，它从“主 owner”变成“兼容件”

## 7. scripts 怎么重摆

当前 `backend/scripts/` 只有一个 `batch_import.py`，太单一，也没有分组。

目标结构：

```text
backend/scripts/
├─ imports/
│  ├─ batch_import.py
│  ├─ rebuild_one_library_item.py
│  └─ repair_import_artifacts.py
├─ debug/
│  ├─ inspect_docling_output.py
│  ├─ inspect_chunks.py
│  └─ inspect_window_state.py
├─ docx_tools/
│  ├─ unpack_docx.py
│  ├─ pack_docx.py
│  └─ validate_docx.py
└─ evals/
   └─  # 评测体系后置接入后再放
```

这样分的意义是：

- 导入工具和调试工具分开
- DOCX 辅助脚本有明确归属
- 评测脚本不提前混进来

## 8. tests 怎么重摆

目标结构：

```text
backend/tests/
├─ data_layer/
├─ agent_layer/
├─ service_layer/
└─ integration/
```

当前旧的：

- `tests/domain/test_library_entities.py`

后续要迁成：

- `tests/data_layer/contracts/test_library_item.py`
- `tests/data_layer/contracts/test_import_task.py`

不再继续沿用 `domain/` 这类临时骨架命名。

## 9. 建议的实际迁移顺序

这一步不是“一次性大搬家”，而是四刀。

### 第一刀：建立新目录，不搬旧目录

先创建：

- `data_layer/`
- `agent_layer/`
- `service_layer/`

然后新代码只往这里写。

### 第二刀：先迁数据层主链

最先迁：

- 导入入口
- Docling adapter
- normalized JSON / markdown
- chunk / anchor / index / storage

### 第三刀：再迁 Agent 层

再迁：

- written_context
- snapshot freeze
- query rewrite
- reflection
- compact

### 第四刀：最后迁服务层并删旧入口

最后才删：

- `api/v1/deps.py`
- 旧 route
- 旧 pipeline
- 旧 stores

## 10. 这一轮不做什么

这一步只是重规划文件分布和 owner，不做：

- 评测体系接入
- RAG A/B
- 前端联调
- 多 Agent 扩张

## 11. 最终结论

这一步的目标不是“把现在的目录讲通”，而是：

- 让新代码只有一个落点
- 让旧代码只有一个命运：冻结、摘逻辑、删除
- 让 `Docling`、`Anthropic skills`、`MinerU` 在文件分布里各回各位
