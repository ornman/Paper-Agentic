# 后端实施切片

这份文档回答一个更实际的问题：

如果明天开始真正改后端代码，第一刀先切哪里，第二刀切哪里，什么先别碰。

原则：

- 先把主链打通，再做清理
- 先做能建立新真相源的代码，再删旧代码
- 先做日志和可观测骨架，后面的切片才能排障
- 评测、A/B、benchmark 一律后置到服务层主链稳定之后

## 切片 -1：停线与混合态封存

### 目标

先把“什么是活动链，什么是遗留残留”说清楚，停止继续在错误位置加功能。

### 产出

- 活动启动链说明
- 遗留目录冻结规则
- Kimi 退出活动基线的示例配置修正
- 评测后置说明

### 暂时不要做

- 不接评测
- 不给旧 `api/clients/core/models/pipelines/stores/utils` 补新功能
- 不把临时 `application/domain/infrastructure/interfaces` 当最终结构

### 验收

- 后端当前真实启动链可一眼看清
- 遗留目录和临时骨架的身份可一眼看清
- 文档和示例配置不再把 Kimi 当活动默认值
- 文档明确写出“先重构后评测”

## 切片 0：公共底座

### 目标

先把所有后续切片都要依赖的运行底座立起来。

### 产出

- `service_layer/config` 基础 settings
- `service_layer/bootstrap/logging`
- `service_layer/bootstrap/lifespan`
- 统一 `request_id / task_id / session_id` 贯穿机制
- 统一错误载荷骨架

### 暂时不要做

- 不改业务路由命名
- 不碰检索策略
- 不做目录大清理

### 验收

- 后端启动后有正式日志，不靠 `print`
- 任一请求都能带出 `request_id`
- 导入链和对话链都有统一错误载荷骨架

## 切片 1：数据层导入主链

### 目标

用最短主链跑通：

`本地 PDF/DOCX -> Docling -> markdown + raw/normalized json -> 持久化`

### 产出

- 本地路径导入入口
- Docling 适配器
- `parsed/` 目录产物规范
- 导入状态机第一版

### 暂时不要做

- 不接问答
- 不接 WPS 轮询
- 不接 compact

### 验收

- 一篇 PDF、一篇 DOCX 都能导入并落盘解析产物
- 导入日志能清楚显示 `parsing / cleaning / enriching`
- 导入失败能区分 `retrying / degraded / needs_user_action`

## 切片 2：切分、锚点、索引

### 目标

把“解析产物”变成“可检索知识”。

### 产出

- 语义切分器
- 父子块建模
- 原文锚点持久化
- embedding 接入
- Chroma / BM25 写入
- 删除单文档时的三处一致清理

### 暂时不要做

- 不做完整 Agent
- 不做复杂 reflection

### 验收

- retrieval chunk 与 answer context pack 已经分离
- 每个 chunk 都能反查锚点
- 图片语义子块能跟随父块联动
- tokenizer 超限不会静默写入索引

## 切片 3：Agent 输入面与双循环

### 目标

先把 Agent 的输入现场做对。

### 产出

- `written_context` 常驻轮询运行态
- `selection / prompt / written_context` 冻结副本
- 四种起手场景加权
- 进程内活动窗口基础写读

### 暂时不要做

- 不求回答质量完美
- 不先引入复杂工具生态

### 验收

- live `written_context` 和冻结副本严格分离
- 程序重开时优先读取当前文档真实内容
- 进程内活动窗口与 SQLite 长期事实边界清楚

## 切片 4：Agent 取证与回答

### 目标

把“会说话的 RAG 包装器”提升成“会判断的 Agent 主链”。

### 产出

- query rewrite
- dense + sparse recall
- fusion
- rerank
- reflection 判定
- answer context pack 组装
- 引用与锚点返回

### 暂时不要做

- 不引入多 Agent
- 不过早优化复杂工具编排

### 验收

- 检索不是每次必调
- 固定 `topk` 退出主链
- 每轮都有 reflection verdict
- 回答带直接依据和原文锚点

## 切片 5：compact、摘要、运行态治理

### 目标

让长对话能持续，而不是越聊越炸。

### 产出

- token 统计
- SSE metadata 上报上下文占用
- compact hook
- 小模型摘要
- 活动窗口清理与 compact 触发
- 活动窗口预算保护线逻辑

### 暂时不要做

- 不先做花哨前端展示

### 验收

- `remaining_ratio < 5%` 时自动 compact
- compact 日志可观测
- 活动窗口接近上下文预算上限时会触发 compact，并停止继续膨胀

## 切片 6：服务层 API 与清理

### 目标

把服务层稳定暴露出来，并开始删除旧活动实现。

### 产出

- 新 API 路由
- SSE 编码
- 健康检查
- 降级状态暴露
- 旧路由与旧 pipeline 删除计划

### 暂时不要做

- 不再给旧目录补新功能

### 验收

- 新 API 能完整覆盖导入、问答、会话、written_context、health
- 旧 `api/v1/deps.py`、上传式 import、内存 `_poll_cache` 进入删除清单

## 切片 7：评测体系后置接入

### 目标

在三层主链稳定之后，再接黑盒、内部态和 A/B 评测。

### 产出

- blackbox runner
- internal runner
- ab runner
- judge 与报告

### 暂时不要做

- 不在混合态实现上提前下评测结论

### 验收

- 评测对象是重构后的活动主链
- 评测不再依赖旧 Kimi 套餐默认值
- 评测结果可以直接反推当前活动实现

## 第一批实际要改的文件

如果下一轮就进入代码，这一批是最先碰的：

- `backend/.env.example`
- `backend/app/main.py`
- `backend/app/bootstrap/app_factory.py`
- `backend/app/bootstrap/container.py`
- `backend/app/interfaces/http/router.py`
- `backend/app/api/v1/router.py`
- `backend/app/api/v1/deps.py`
- `backend/app/bootstrap/logging.py`
- `backend/app/bootstrap/lifespan.py`

原因很简单：

- 不先收敛活动入口、遗留入口和配置基线，后面的数据层、Agent 层、服务层重构会一直踩双真相源

## 明确后放的东西

这些故意后放：

- 复杂 tool graph
- 多 Agent 分工
- 更换向量库
- 非 Windows 兼容
- 前端联调细节

不是它们不重要，而是现在先动它们会打散主线。
