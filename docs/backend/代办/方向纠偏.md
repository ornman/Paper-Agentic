## 二、Agent 层功能偏差

### 2.1 🔴 TurnRunner 检索未走 RRF 融合（严重）

**架构文档要求**：混合检索 = dense + sparse + RRF 融合。Agent 层调用数据层的混合检索。

**实际代码**（`turn_runner.py` `_retrieve` 方法，第223-241行）：
```python
# 分别拿 dense 和 sparse 结果，直接拼在一起返回
dense_results = self._vector_store.query(...)
sparse_results = self._keyword_search.query(...)
results.extend(dense_results)
results.extend(sparse_results)
```

**偏差**：`TurnRunner._retrieve()` 将 dense 和 sparse 结果简单拼接（`list.extend`），**没有调用** `data_layer/retrieval/fusion/rrf_fusion.py` 中已实现的 `rrf_fuse()` 函数做 RRF 融合。

而 `AnswerGenerator`（`answer_generator.py` 第71-73行）反而正确调用了：
```python
results = fusion.rrf_fuse(dense_results, sparse_results, topk=10, ...)
```

**修正方向**：`TurnRunner._retrieve()` 应调用 `rrf_fuse()` 做融合，而不是简单拼接。`AnswerGenerator` 存在的检索逻辑不应在 response 层重复。

### 2.2 🔴 AnswerGenerator 违反三层职责边界（严重）

**架构文档要求**：response 层只负责"回答生成、引用、context pack"，不负责检索编排。

**实际代码**（`answer_generator.py`）：`AnswerGenerator` 自己做了 embedding → dense 检索 → BM25 检索 → RRF 融合 → 回答生成，完整吃掉了数据层检索 + Agent 层编排的职责。

**偏差**：`AnswerGenerator` 同时承担了：
1. 调用 `embedding_client` 生成查询向量
2. 调用 `vector_store.query()` 做 dense 检索
3. 调用 `keyword_search.query()` 做 sparse 检索
4. 调用 `fusion.rrf_fuse()` 做融合
5. 调用 `chat_model` 生成回答

这违反了架构文档的明确边界：
- 检索编排属于 Agent 层（planning + orchestration）
- 检索执行属于数据层（retrieval）
- response 层只负责"拿到文本后怎么切 block + 怎么插引用"

**修正方向**：`AnswerGenerator` 应被拆解——检索逻辑回归 `TurnRunner`，`AnswerGenerator` 只保留回答生成（不含检索）。

### 2.3 🟡 reflection 未接入主链的检索循环（中等）

**架构文档要求**（§6.1.6）：reflection 是"检索后分析"——每轮拿一小批证据，reflection 判断 `supported/insufficient/conflicting/off_track`，决定是否继续取证或换方向。

**实际代码**（`turn_runner.py` 第166-180行）：reflection 在 LLM 回答**生成完之后**才执行，且只做"回答是否被支持"的审查，不涉及检索方向的调整。

```python
# 先检索 → 生成回答 → 然后才 reflection
if snapshot.reflection_enabled:
    ref_result = await reflect(...)
```

**偏差**：
1. 架构文档的 reflection 是"取证过程中的循环判断"，代码的 reflection 是"回答后审查"
2. 架构文档要求每轮 reflection 决定是否换方向重新检索（`insufficient/conflicting/off_track` → 换 query 重检），代码没有这个循环
3. 架构文档要求"最多 3 轮 reflection"、"最多 2 次方向切换"，代码的 `reflect()` 虽然有这些参数，但不是用在检索循环中

**修正方向**：reflection 应嵌入检索循环——检索 → reflection 判断 → 不够则重检/换方向 → 再检索 → 直到 supported 或用尽预算。

### 2.4 🟡 compact 未接入主链（中等）

**架构文档要求**（§6.1.8）：上下文剩余比例低于 5% 时必须触发 compact，使用小模型总结历史并重注入上下文。

**实际代码**：
- `hooks/compact.py` 实现了 `compact_conversation()` 函数（34行），但 `TurnRunner.run()` 中**没有调用**它
- `token_budget.py` 提供了 `remaining_ratio` 属性，但 TurnRunner 只在 metadata 事件中报告 token 数，不检查 5% 阈值
- 没有任何地方触发 compact

**偏差**：compact 代码存在但未接入主链。

**修正方向**：`TurnRunner.run()` 中，在 `_build_messages()` 之前检查 `remaining_ratio < 0.05`，若低于阈值则触发 compact。

### 2.5 🟡 tool_loop 已实现但未接入（中等）

**架构文档要求**（§6.1.9）：Agent 需要基础 runtime：tool call、loop、tasklist。

**实际代码**：

- `TurnRunner` 构造函数接受 `tool_registry` 参数
- 但 `query_routes.py` 中传入 `tool_registry=None`，`TurnRunner.run()` 中完全没使用 `tool_loop`

**偏差**：tool_loop 基础设施已存在但未被使用。没有任何工具被注册。

**修正方向**：至少注册 `retrieve`（检索）、`read_anchor`（读锚点）、`compact_history`（压缩历史）三个内部工具，在 TurnRunner 主循环中使用。

## 三、数据层功能偏差

### 3.1 🔴 检索与切分职责分散在两个冗余路径（严重）

**问题描述**：
- `TurnRunner._retrieve()` 自己拼 dense + sparse 结果（无融合）
- `AnswerGenerator.generate()` 也自己做 dense + sparse + RRF 融合

存在两条并行的检索路径，且行为不一致。架构文档明确只有一条：Agent 层编排 → 数据层检索 → 结果返回。

**修正方向**：统一走 `TurnRunner` → 调用 `data_layer/retrieval/` 的 dense + sparse + fusion → 结果返回给 response 层。`AnswerGenerator` 不应再持有 `vector_store` 和 `keyword_search`。

### 3.2 🟡 VLM 处理器硬编码了默认 API 地址（中等）

**实际代码**（`vlm_processor.py` 第79行）：
```python
base_url: str = "https://api.coro0.top/v1",
model: str = "qwen3-vl:235b",
```

**架构文档要求**（§1.4）：后端配置保持 provider-neutral，活动示例配置不再默认指向特定提供商。

**修正方向**：默认值应从 `BackendSettings` 读取（`settings.py` 已定义 `vlm_api_key`、`vlm_base_url`、`vlm_model`），不应在 VLMProcessor 构造函数中硬编码。

### 3.3 🟡 语义切分器的 overlap 策略与文档不完全一致（中等）

**架构文档要求**（§5.2.4）：overlap 只在单个语义块超过 embedding 模型上下文 1/4 时启用，取块长 10%。

**实际代码**（`semantic_chunker.py` 第17行）：
```python
OVERLAP_TOKENS = MAX_CHUNK_TOKENS // 4  # 超长块重切时的 overlap
```

**偏差**：
1. `MAX_CHUNK_TOKENS = 512`，overlap = 128，这是固定的 25%（不是 10%）
2. 架构文档说的是"超过 embedding 模型上下文 1/4 时才启用"，代码中任何超长块都会加 overlap，且比例是 25% 不是 10%
3. 首次默认值应为 10%，如果样本验证显示冗余过高再往下调

**修正方向**：将 `OVERLAP_TOKENS` 改为动态计算（块长 * 10%），且只在块超过 embedding 上下文 1/4 时才启用。

### 3.4 🟡 Anchor 结构基本匹配但有字段类型差异（轻微）

**架构文档要求**（§5.2.6）：`heading_path` 是标题层级路径字符串。

**实际代码**（`chunking/semantic_chunker.py` `Anchor` dataclass）：
```python
heading_path: list[str] = field(default_factory=list)  # 用的是 list[str]
```

**偏差**：架构文档未明确 heading_path 是字符串还是列表，但代码用 `list[str]` 存路径层级是更合理的做法。

**结论**：代码实现更优，建议更新架构文档的 heading_path 字段说明为 `list[str]`。

## 四、服务层功能偏差

### 4.1 🟡 health_routes 实现过于简陋（中等）

**API 文档要求**：`GET /health` 返回各组件状态（chroma、bm25、cache、llm_config、embedding_config），status 取值 `ok | degraded | error`。

**实际代码**（`health_routes.py`）：只有 10 行，直接委托给 `container.health()`。

**偏差**：需要验证 `AppContainer.health()` 是否返回了 API 文档要求的完整字段结构。如果 `container.py` 没有实现 `health()` 方法，则 health 端点会直接报错。

### 4.2 🟡 query_routes 中 TurnRunner 构造逻辑重复（中等）

**实际代码**（`query_routes.py`）：`_build_runner()` 在 container 存在和不存在两种情况下，都手动组装 `TurnRunner` 的所有依赖。这是手工 DI，且 `container.py` 已经负责了所有组件的初始化。

**偏差**：`TurnRunner` 应该直接从 `container` 获取已装配好的实例，而不是在 route 层重新手动组装。当前做法导致两个问题：
1. container 已有的装配逻辑被绕过
2. fallback 路径中重新创建 `ChatModel`、`VectorIndex` 等重量级对象

**修正方向**：`container.py` 应直接装配完整的 `TurnRunner` 实例，route 层只做 `container.turn_runner` 的取用。

`router.py` 正确包含所有路由：
