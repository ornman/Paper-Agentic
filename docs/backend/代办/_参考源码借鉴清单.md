# 参考源码借鉴清单

这份文档只回答一个问题：

既然已经把参考源码拉到本地，哪些值得借鉴，哪些不该直接搬进当前后端？

对应本地参考目录：

- `reference/external/docling/`
- `reference/external/anthropic-skills/`

## 1. 当前结论

先把结论说死：

- `Docling` 是当前 `PDF + DOCX` 主转换基线
- `MinerU` 不再是活动主链
- `Anthropic PDF/DOCX skills` 不作为主解析链，只作为工程手法参考

也就是说：

- 文档理解主链看 `Docling`
- PDF 图片渲染、DOCX 解包/验证这类辅助能力可以借鉴 `Anthropic skills`
- 当前旧 `MinerU` 实现只保留“历史兼容 / 兜底适配”的位置，不再影响目标文件分布

## 2. Docling 值得借鉴的点

### 2.1 统一转换入口

参考源码：

- `reference/external/docling/docling/document_converter.py`
- `reference/external/docling/docs/reference/document_converter.md`

能直接确认的事实：

- `DocumentConverter` 是统一入口
- `DOCX` 走 `WordFormatOption -> MsWordDocumentBackend -> SimplePipeline`
- `PDF` 走 `PdfFormatOption -> StandardPdfPipeline`
- 同一个转换器可以按 `InputFormat` 配不同 pipeline / backend

这对我们最重要的启发不是“抄它的类名”，而是：

- 后端数据层不该再按 `pdf_service`、`docx_service` 分裂成两套主链
- 应该先有统一 `document_conversion` owner
- 再在 owner 内按格式配置 backend / pipeline

### 2.2 DOCX 不是“先转 PDF 再处理”

参考源码：

- `reference/external/docling/docling/backend/msword_backend.py`
- `reference/external/docling/docling/pipeline/simple_pipeline.py`

能直接确认的事实：

- `MsWordDocumentBackend` 直接读取 OOXML
- 能处理列表、表格、公式、评论、图片、header/footer 等结构
- `SimplePipeline` 说明 DOCX 这类 declarative backend 可以直接产出结构化文档

这意味着：

- 我们不需要把 `DOCX` 主链设计成“先转 PDF，再复用 PDF 解析”
- `DOCX` 应该在数据层有独立 backend adapter
- 但它和 `PDF` 仍然汇入统一的 normalized document 契约

### 2.3 PDF 主链是 pipeline + options，不是写死的单解析器

参考源码：

- `reference/external/docling/docling/pipeline/standard_pdf_pipeline.py`
- `reference/external/docling/docling/datamodel/pipeline_options.py`
- `reference/external/docling/docs/examples/custom_convert.py`

能直接确认的事实：

- PDF 主链不是一个死函数，而是一条可配置 pipeline
- OCR、表格结构、加速器、VLM 等都由 options 控制
- backend 可以切 `Docling Parse`、`PyPdfium` 等不同组合

对我们的启发：

- 当前后端不该继续把 PDF 处理硬编码在 `MinerUClient -> cleaner -> chunker`
- 应该把“转换配置”从编排逻辑里拆出去
- 文件分布上要留出：
  - `conversion/`
  - `conversion_options/`
  - `normalization/`

### 2.4 产物应该同时保留 markdown 和结构化 JSON

参考源码：

- `reference/external/docling/README.md`
- `reference/external/docling/docs/examples/custom_convert.py`

能直接确认的事实：

- Docling 主打统一结构文档表示
- 标准导出就包含 Markdown 和 lossless JSON

对我们的启发：

- `parsed/` 下不能只留 Markdown
- 应该同时保存：
  - raw conversion result
  - normalized JSON
  - markdown

这也说明当前文件分布里，`parsed/` 应该是数据层正式 owner，而不是随手存临时文件的地方

## 3. Anthropic skills 值得借鉴的点

### 3.1 PDF：借鉴渲染和辅助提取，不借来做主解析链

参考源码：

- `reference/external/anthropic-skills/skills/pdf/SKILL.md`
- `reference/external/anthropic-skills/skills/pdf/reference.md`
- `reference/external/anthropic-skills/skills/pdf/scripts/convert_pdf_to_images.py`

能直接确认的事实：

- 这套 PDF skill 更像通用工具箱
- 强项是：
  - `pypdf`
  - `pdfplumber`
  - `pypdfium2`
  - OCR
  - 表单/注释/图片渲染

对我们的启发：

- 可借鉴为：
  - 原文页渲染
  - 证据页截图
  - bbox 校验
  - 表单/注释类辅助脚本
- 不应把它替代 Docling 当主文档转换器

### 3.2 DOCX：借鉴解包、重打包、验证、redline 清洗

参考源码：

- `reference/external/anthropic-skills/skills/docx/SKILL.md`
- `reference/external/anthropic-skills/skills/docx/scripts/office/unpack.py`
- `reference/external/anthropic-skills/skills/docx/scripts/office/pack.py`
- `reference/external/anthropic-skills/skills/docx/scripts/office/validate.py`
- `reference/external/anthropic-skills/skills/docx/scripts/office/helpers/merge_runs.py`
- `reference/external/anthropic-skills/skills/docx/scripts/office/helpers/simplify_redlines.py`

能直接确认的事实：

- `.docx` 被明确当作 zip + XML 结构体处理
- 提供了解包、pretty-print、重打包、schema 校验
- 还提供 tracked changes 简化、run merge 这类编辑辅助工具

对我们的启发：

- 这非常适合作为后端“编辑器旁路工具”或“调试工具”
- 不适合作为当前主解析链 owner
- 但非常适合单独放进：
  - `data_layer/document_debug/`
  - 或 `scripts/docx_tools/`

## 4. MinerU 现在该怎么定位

当前结论已经可以写清楚，不再留模糊表述：

- `MinerU` 退出主解析链
- 如果保留，只能保留为：
  - 旧数据兼容读取
  - 某些极端 PDF 的补充 fallback
  - 历史导入产物恢复

这意味着文件分布上：

- 不再为 `MinerU` 保留主 owner
- 不再让 `clients/mineru_client.py` 或其新位置主导导入链
- 如果后续保留，实现位置也只能是：
  - `data_layer/fallbacks/mineru_fallback.py`
  - 或 `data_layer/legacy/mineru_compat.py`

## 5. 对后端文件分布的直接影响

基于这些源码，后端文件分布至少要反映以下事实：

### 5.1 数据层必须拆出这几块

- `conversion/`
  - 统一转换入口
  - Docling adapter
  - backend / pipeline options
- `normalization/`
  - raw result -> normalized JSON
  - markdown 导出
  - 图片语义补充
- `chunking/`
  - semantic chunk
  - parent-child image chunk
  - anchor builder
- `indexing/`
  - embedding
  - vector / keyword index
- `storage/`
  - parsed artifacts
  - library items
  - import tasks
  - anchors
  - backups

### 5.2 Agent 层不要接手文档转换细节

Agent 层可以关心：

- 取哪些证据
- 怎么 reflection
- 怎么 compact

但不该直接关心：

- PDF backend
- DOCX XML 解包
- OCR options
- 图片抽取实现

### 5.3 服务层也不要继续碰旧 parser client

服务层只该暴露：

- 导入 API
- 任务状态
- 对话 SSE
- health

不该再直接知道 `MinerUClient`、`VLMClient`、`chunker`

## 6. 最终判定

这轮参考源码阅读后的最终判定是：

1. `Docling` 进入正式参考基线，并指导数据层主目录设计
2. `Anthropic skills/pdf` 作为 PDF 辅助工具参考
3. `Anthropic skills/docx` 作为 DOCX 调试/验证/编辑工具参考
4. `MinerU` 从主链退场，只保留 fallback / legacy 位置
