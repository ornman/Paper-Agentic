# agent_layer 验证计划

## 1. 验证目标

不是"能不能跑通一次 SSE"，而是回答五个问题：

| # | 问题 | 验证手段 |
|---|------|---------|
| Q1 | 每个模块是否按契约工作？ | 单元测试（已有 116 个） |
| Q2 | 模块组装后主链是否闭合？ | 集成测试（mock LLM → 真实 LLM） |
| Q3 | Agent 状态机是否覆盖所有分支？ | 状态机路径测试 |
| Q4 | 出错后能否自愈/降级？ | 故障注入测试 |
| Q5 | 运行时可观测性如何？ | 日志审计 + 指标采集 |

---

## 2. 验证分层

```
Level 0  单元测试（已有）         → 每个函数/类独立验证
Level 1  模块集成测试             → 模块之间真实调用，mock 外部依赖
Level 2  端到端链路测试           → 真实 HTTP → 真实 LLM → 真实 SSE
Level 3  故障注入 + 自愈测试      → 模拟各种故障，验证降级/重试/恢复
Level 4  长时间浸泡测试           → 无人值守跑数小时，采集稳定性数据
```

## 4. Level 1：模块集成测试

目标：验证模块之间真实调用的正确性，不依赖外部 LLM。

### 4.1 snapshot → retrieval_gate 联动

```
测试场景：
1. AskRequest(enable_rag=False) → should_retrieve 返回 False
2. AskRequest(enable_rag=True, paper_ids=["p1"]) → should_retrieve 返回 True
3. AskRequest(enable_rag=True, selection="某段文字") → should_retrieve 返回 True
4. AskRequest(enable_rag=True, paper_ids=None, selection=None) → should_retrieve 返回 False

验证点：
- snapshot_builder 输出的 FrozenTurnSnapshot 字段完整
- retrieval_gate 的输入输出衔接正确
```

### 4.2 retrieval → source_mapper → block_streamer 链

```
测试场景：
1. 模拟 10 条检索结果 → source_mapper 生成 10 个 SourceCard
2. LLM 输出 "研究表明[1]该方法有效[2][3]" → citation_resolver 提取引用
3. 带引用的文本 → block_streamer 生成 BlockParagraph with citations
4. SourceCard.id 和 BlockCitation.sourceId 能对上

验证点：
- source_mapper 的 id 生成唯一性
- citation_resolver 的正则覆盖 [1]~[99]
- block_streamer 的 markdown 解析正确性
- 引用链 source_mapper → citation_resolver → block_streamer 闭合
```

### 4.3 turn_runner + mock LLM 全链

```
测试场景：
1. mock LLM 返回纯文本 → SSE 帧序列正确
2. mock LLM 返回带引用文本 → BlockParagraph 带 citations
3. mock LLM 抛异常 → ErrorEvent 输出
4. 连续 2 轮对话 → window_store 积累 4 条消息
5. 第 3 轮对话 → 只取最近 10 条历史

验证点：
- SSE 事件顺序：thinking → block[] → sources → done
- persistence 写入了 user + assistant 消息
- window_store FIFO 淘汰生效
```

### 4.4 token_budget 上下文裁剪

```
测试场景：
1. 5 条检索结果，总 token < 30000 → 全部保留
2. 20 条检索结果，总 token > 30000 → 截断到预算内
3. 单条结果 > 30000 token → 跳过该条
4. 空检索结果 → context 为空字符串

验证点：
- TokenBudget.remaining 不为负
- 截断后的 context 能正常拼入 system prompt
```

## 5. Level 2：端到端链路测试

目标：真实 HTTP 请求 → 真实 LLM → 真实 SSE 帧解析。

### 5.1 SSE 协议合规性

```
测试方法：发 HTTP POST，解析 SSE 帧流

验证点：
1. 每帧格式正确：`event: xxx\ndata: yyy\n\n`
2. 第一帧是 thinking（如果 thinking_enabled=True）
3. block 帧的 data 是合法 JSON，且包含 type 字段
4. sources 帧的 data 是数组，每个元素有 id/title
5. 最后一帧是 done
6. error 帧的 data 包含 message 字段
7. 前端 TypeScript 能正确 parse 每一帧（用 json-schema 验证）
```

### 5.2 多模型切换

```
测试场景：同一个 prompt，不同 model 参数

| model | 预期行为 |
|-------|---------|
| 默认（不传） | 使用 .env 配置的模型 |
| 指定模型名 | 使用指定模型 |
| 不存在的模型 | ErrorEvent，不是 500 崩溃 |

验证点：
- ChatModel 的 model override 生效
- 错误模型不会导致服务崩溃
```

### 5.3 多轮对话上下文保持

```
测试场景：同一 session_id 连续 5 轮

第 1 轮：问 "什么是 RAG？"
第 2 轮：问 "它有哪些组成部分？"（应该理解 "它" = RAG）
第 3 轮：问 "和传统的搜索有什么区别？"
第 4 轮：问 "给我一个实际应用案例"
第 5 轮：问 "总结一下我们刚才讨论的内容"

验证点：
- 每轮的 window_store 都积累了历史
- 第 2-5 轮的 LLM 输入包含前几轮的对话历史
- 第 5 轮能正确总结前 4 轮内容
```

### 5.4 输入源加权验证

```
测试场景：

| 输入组合 | 预期权重 |
|---------|---------|
| prompt="你好" | prompt=1.0 |
| editor_context(draft="草稿") | written_context=1.0 |
| editor_context(draft="草稿") + selection="选中" | selection=0.7, written=0.3 |
| prompt="改写" + selection="选中" + draft="草稿" | prompt=0.5, selection=0.3, written=0.2 |

验证点：
- FrozenTurnSnapshot.used_weights 字段值正确
- 加权后的 query_text 拼接顺序合理
```

## 6. Level 3：故障注入 + 自愈测试

目标：验证 Agent 在各种异常情况下的行为。

### 6.1 LLM 故障

| 故障类型 | 注入方式 | 预期行为 |
|---------|---------|---------|
| LLM 超时 | mock 30s 无响应 | ErrorEvent，不是无限等待 |
| LLM 返回空 | mock 返回 "" | 正常结束，blocks 为空 |
| LLM 返回非法 JSON | mock 返回 "not json" | 降级为纯文本 BlockParagraph |
| LLM 429 限流 | mock 抛 RateLimitError | ErrorEvent，建议稍后重试 |
| LLM 500 内部错误 | mock 抛 APIError | ErrorEvent，不暴露内部细节 |
| LLM 连接断开 | mock 抛 ConnectionError | ErrorEvent，日志记录 |

### 6.2 检索故障

| 故障类型 | 注入方式 | 预期行为 |
|---------|---------|---------|
| Embedding 服务挂了 | mock embed_single 抛异常 | 跳过 dense 检索，只用 BM25 |
| BM25 索引损坏 | mock keyword_search.query 抛异常 | 跳过 sparse 检索，只用 dense |
| 两者都挂 | 两个都 mock 抛异常 | 跳过 RAG，直接生成（无证据） |
| 检索结果为空 | mock 返回 [] | 正常生成，sources 为空 |
| 检索结果超大 | mock 返回 1000 条 | TokenBudget 截断，不 OOM |

### 6.3 会话故障

| 故障类型 | 注入方式 | 预期行为 |
|---------|---------|---------|
| window_store 满 | 连续发 > max_messages 条 | FIFO 淘汰，不报错 |
| persistence 写入失败 | mock save_message 抛异常 | 日志 warning，不影响 SSE 输出 |
| editor_context 不存在 | get 返回 None | written_context 为空，正常继续 |
| session_id 为空 | 不传 session_id | 自动生成或报错（不能 500） |

### 6.4 并发故障

| 故障类型 | 注入方式 | 预期行为 |
|---------|---------|---------|
| 同 session 并发请求 | 2 个并发 POST 同 session_id | 不崩溃，各自独立响应 |
| 不同 session 并发 | 10 个并发 POST 不同 session_id | 不互相干扰 |

### 6.5 自愈验证

```
场景：LLM 故障恢复后继续使用

1. 第 1 轮：LLM 正常 → 成功
2. 第 2 轮：注入 LLM 故障 → ErrorEvent
3. 第 3 轮：恢复 LLM → 成功
4. 验证：第 3 轮的 window_store 包含第 1 轮的历史（第 2 轮失败不保存）
```

## 7. Level 4：长时间浸泡测试

目标：无人值守运行，采集稳定性数据。

### 7.1 浸泡模式

```
模式 A：连续对话浸泡
- 同一 session，每 30 秒发一轮随机 prompt
- 持续 2 小时
- 监控：内存泄漏、响应延迟漂移、错误率

模式 B：多 session 并发浸泡
- 10 个 session，每个每 60 秒发一轮
- 持续 4 小时
- 监控：内存占用、session 间隔离、FIFO 淘汰正确性

模式 C：故障注入浸泡
- 随机在 10% 的请求中注入故障
- 持续 1 小时
- 监控：错误恢复率、日志完整性、无未捕获异常
```

### 7.2 采集指标

| 指标 | 采集方式 | 告警阈值 |
|------|---------|---------|
| 请求总数 | 计数器 | - |
| 成功率 | 成功/总数 | < 95% |
| 平均响应时间 | 每轮计时 | > 10s |
| P99 响应时间 | 排序取 99% | > 30s |
| 内存占用 | psutil | > 500MB |
| 错误类型分布 | 按 ErrorEvent.message 分组 | - |
| SSE 帧完整性 | 检查每轮都有 done/error | 缺失 = 异常 |
| window_store 大小 | 每轮检查 | > max_messages = bug |
| persistence 消息数 | 每轮检查 | 与请求总数×2 不匹配 = bug |

### 7.3 测试数据集

```
预设 50 个 prompt 模板，随机选取：

简单问答（10 个）：
- "什么是机器学习？"
- "解释一下梯度下降"
- ...

学术相关（15 个）：
- "这篇论文的主要贡献是什么？"
- "帮我改写这段摘要：{selection}"
- ...

边界情况（10 个）：
- 空 prompt
- 超长 prompt（5000 字）
- 纯特殊字符
- 多语言混合
- ...

压力测试（15 个）：
- "详细解释深度学习的各个方面，包括CNN、RNN、Transformer、GAN、VAE..."
- 连续快速发送
- ...
```

## 8. 日志观测性验证

### 8.1 必须存在的日志点

| 位置 | 级别 | 内容 |
|------|------|------|
| turn_runner.run 开始 | INFO | request_id, session_id |
| snapshot freeze 完成 | INFO | used_inputs 权重 |
| retrieval_decision | INFO | need_rag 结果 |
| retrieve 完成 | INFO | 证据数量 |
| LLM 调用开始 | INFO | model 名称 |
| LLM 调用完成 | INFO | 响应 token 数 |
| persist 完成 | INFO | 保存的消息数 |
| 任何异常 | ERROR | 完整 traceback |
| 检索降级 | WARNING | 哪个检索源失败 |
| token budget 超限 | WARNING | 截断了多少 |

### 8.2 验证方法

```
1. 发一轮请求
2. 读取日志文件/输出
3. 验证所有必须的日志点都出现了
4. 验证日志格式一致（时间戳、级别、消息）
5. 验证 ERROR 日志包含完整上下文
```

## 9. 执行顺序

```
Phase 1  Level 1 集成测试（mock LLM，不依赖网络）
  ↓
Phase 2  Level 2 端到端测试（等 LLM API 可用后）
  ↓
Phase 3  Level 3 故障注入测试
  ↓
Phase 4  Level 4 浸泡测试（长时间运行）
  ↓
Phase 5  日志审计 + 指标报告
```

Phase 1 可以立即执行（纯本地，不需要 LLM）。
Phase 2-5 需要一个可用的 LLM API。
