# 正式版 UI/UX 全局重构设计文档

**日期**: 2026-03-27
**状态**: 已确认
**适用范围**: `D:/同步/project/MVP/frontend`

---

## 1. 重构目标

当前正式版前端在功能上已经具备：

- 知识库状态层
- 路径导入
- WPS 宿主桥接
- 正文轮询
- 单会话消息流
- 来源卡片
- 灵感按钮 SSE 主链路

但视觉层面仍偏工程实现感，尚未达到“可卖的办公 AI 产品”状态。

本轮重构的目标不是增加功能，而是：

> 把当前已经跑通的正式版，重构为**像成熟办公产品、具备投资展示说服力**的 UI。

核心要求：

- 简约、大方、克制
- 现代、精致、像真正产品
- 不是系统默认面板
- 不是花哨营销页
- 保持办公插件可信度

---

## 2. 视觉方向

本轮 UI/UX Pro Max 采用：

- **Claude / Copilot 高级工具版** 的整体气质
- **WPS 办公插件形态** 的容器约束
- **模型中立** 的品牌表达
- **产品工作台** 的信息组织方式

### 核心视觉关键词

1. **稳**：像正式办公产品，不像 demo
2. **精**：靠排版、留白、边界、层级建立高级感
3. **轻**：不堆装饰、不堆大阴影、不堆花哨动效
4. **准**：知识库、消息、来源、输入区的层级一眼可懂

---

## 3. 全局视觉语言

### 3.1 字体策略

- 使用系统安全中文无衬线字体
- 不依赖外部字体资源
- 通过字号、字重、行高、字距建立层级，而不是靠花字体

### 3.2 色彩策略

只保留三类颜色系统：

#### 背景层
- 主背景：纯白
- 次背景：极浅灰
- 内容层：更柔和的一层浅灰白

#### 文本层
- 主文本：深灰黑
- 次文本：中灰
- 辅助文本：浅灰

#### 强调层
- 仅保留一个克制蓝
- 用于：
  - 主按钮
  - 当前焦点
  - 状态高亮
  - 来源可点击强调

### 3.3 边框与阴影

- 主要依赖浅灰边框建立结构层级
- 阴影只在必要时轻量使用
- 不用重阴影堆叠层次

### 3.4 圆角体系

统一为三档：
- 小圆角：输入框、轻按钮
- 中圆角：消息卡片、知识库状态条
- 大圆角：主要面板容器

### 3.5 动效策略

只保留三类动效：
- 状态切换淡入
- 轻量按钮反馈
- 抽屉滑入

动效只服务于成熟产品感，不服务于炫技。

## 4. 版式与层级重构

### 4.1 顶部导航栏

目标：从“功能条”变成“产品导航”。

特点：
- 左右操作区对称
- 标题更稳、更像工作上下文
- 图标按钮更轻、更克制
- 与内容区用浅分隔清晰切开

### 4.2 知识库状态层

目标：从“信息块”变成“价值证明条”。

它必须让用户一眼理解：
- 这是私有知识库产品
- 当前回答由知识库支撑
- 它不是普通聊天助手

建议层级：
1. 标题 + 极短副说明
2. 状态标签 + 文献数量 + 导入入口
3. 一句系统提示，说明当前回复与知识库关系

### 4.3 中间内容层

目标：从“聊天区”变成“创作工作流记录区”。

#### 空态
应表现为“系统已准备好工作”，而不是简单占位：
- 小型产品标识
- 创作导向主标题
- 强调文档上下文 + 知识库 + 可溯源
- 主按钮：获取灵感
- 次按钮：查看知识库

#### 消息态
要呈现：
- 用户动作 / 请求
- 助手建议正文
- 结构化辅助
- 来源卡片

### 4.4 底部输入区

目标：从“表单”变成“工作输入台”。

结构：
1. `ContextHintBar`
2. 正式输入框
3. 动作区

输入区的语义应从“聊天”改为“描述创作任务”。

## 5. 关键组件重构策略

### 5.1 `TopNavBar`

重构目标：
- 安静但有控制力
- 不像按钮排排坐
- 更像成熟产品的导航层

### 5.2 `KnowledgeBar`

重构目标：
- 成为产品差异化的第一视觉入口
- 不做后台面板感
- 像可信的工作状态条

### 5.3 `EmptyState`

重构目标：
- 表达“准备开始工作”
- 不像 demo 占位
- 不像营销落地页

### 5.4 `MessageList`

重构目标：
- 更像工作过程记录
- 弱化社交聊天感

### 5.5 `AssistantMessage`

重构目标：
- 建议正文 + 证据层的双层结构
- 来源是回复的一部分，不是附件

### 5.6 `SourceCardList`

重构目标：
- 更像证据卡片
- 标题、页码、片段层次清晰
- 可信但不喧宾夺主

### 5.7 `BottomActionBar`

重构目标：
- 正式工作输入台
- 发送为主入口
- 获取灵感为次入口
- 输入区是这轮最该投入打磨的组件

### 5.8 `ContextHintBar`

新增组件，用于表达：
- 当前文档状态
- 当前选区状态
- 本次请求上下文说明

它应当静默但聪明，不应喧宾夺主。

## 6. 正式版输入框主链路（已并入 UI 设计）

正式版输入框主链路采用：

- `written_content`：正文轮询缓存
- `selected_text`：发送前即时读取选区
- `prompt`：用户明确输入

### 请求契约

前端请求体按当前后端 ask 契约发送：

```json
{
  "session_id": "<当前前端单会话 id>",
  "query": "<用户输入>",
  "context": {
    "written_content": "<当前轮询正文>",
    "selected_text": "<发送前读取的选区>",
    "prompt": "<用户输入>"
  }
}
```

### 交互规则

- 宿主 `ready` 才允许发送
- prompt 不能为空
- 发送前即时读取一次选区
- 有选区：正式场景 3
- 无选区：允许降级发送，但提示“仅基于当前文档上下文生成”
- 选区读取失败：允许降级发送，但明确提示

### 入口关系

- `发送`：正式主入口
- `获取灵感`：快捷次入口

两者进入同一条消息流。

## 7. 本轮明确不做的事

为了保证 UI 重构不把工程拖垮，本轮明确不做：

- 历史会话真实切换
- 语音输入
- 附件解析
- 自动插入到文档
- 多模型切换器
- 表格 / 演示专属能力
- 右键菜单增强
- 知识库完整管理页

本轮只做：

> 把当前已经跑通的正式版，做得像一个可以卖的产品。

## 9. 本轮验收标准

至少满足：

1. 一眼看上去不再像工程 demo，而像正式办公产品
2. 知识库状态层成为一级视觉入口
3. 底部输入区成为正式工作入口，不再像普通表单
4. 助手回复区明显呈现“建议正文 + 来源证据层”结构
5. 空态、工作态、错误态、加载态四种状态视觉差异清楚
6. 整体仍然克制，不出现花里胡哨的 AI 套壳感

## 10. 最终结论

当前正式版已经具备功能闭环，但视觉层仍偏工程实现感。

本轮 UI/UX Pro Max 全局重构的最终目标，是把它提升为：

> 一个嵌在 WPS 里的、可信的、专业的、可溯源的论文创作辅助工具。

重点不是加功能，而是：
- 统一视觉语言
- 重构关键骨架
- 提升产品质感
- 让投资展示与正式产品感统一

这份文档是当前正式版 UI 全局重构的唯一有效设计依据，后续实施以此为准。
